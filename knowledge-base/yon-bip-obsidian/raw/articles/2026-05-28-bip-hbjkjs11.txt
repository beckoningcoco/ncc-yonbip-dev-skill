哈勃监控技术
最后更新时间：2024-03-27
本文摘要

哈勃监控技术利用多种数据采集手段将主机和服务涉及的CPU、内存、磁盘、网路、日志、服务调用情况等关键数据统一收集管理，提供了资源、日志、服务调用等多种可视化分析视角。同时，通过链路录制、CPU和内存火焰图、慢SQL日志、异常报警等诊断及预警功能可实现精准、实时、高效的故障排查。最终形成“哈勃监控中心”产品，帮助开发人员、运维人员、IT管理者等角色分析识别系统故障及潜在风险，并做出快速应对，保障服务稳定高效运转。

本文主要介绍了以下内容：

哈勃监控技术的范畴及亮点；
哈勃监控技术所采用的主要技术及基本工作原理；
哈勃监控中心的主要功能；
哈勃监控技术使用中需要特别注意的一些可能发生的故障点；
哈勃监控中心运行过程中可能涉及的一些运维事项；
哈勃监控中心典型使用场景；
哈勃监控中心的开放能力，开发者可以利用这些能力做必要的个性化开发。
产品概述
概述

哈勃监控从资源池（主机）、中间件及数据存储组件、微服务和服务管理平台（K8S等）入手。利用多种数据采集手段将主机和服务涉及的CPU、内存、磁盘、网路、日志、服务调用情况等关键数据统一收集管理。将数据可视化呈现，提供了资源、日志、服务调用多种分析视角。以及链路录制、CPU和内存火焰图、慢SQL统计、异常报警等诊断及预警等功能辅助故障排查。最终形成哈勃监控中心产品，帮助开发人员、运维人员、IT管理者等角色分析识别系统故障及潜在风险，并做出快速应对，保障服务稳定高效运转。

名词解释

哈勃链路Agent：链路跟踪Agent是负责从Java应用程序里面采集链路数据的javaagent。

链路录制：链路录制是场景级链路跟踪产品，擅长特定场景问题的诊断分析优化。

实时链路：实时链路是全量的链路采集产品，擅长还原系统运行的第一现场。

Prometheus：开源的系统监控及报警工具箱。

Grafana：开源的支持多种数据源的强大的可视化面板定义展示工具。

MinIO：高性能的，AWS S3 API兼容的开源文件存储服务工具。

Fluentd：开源的统一灵活的日志采集工具。

Async Profiler：开源的程序调用栈及内存分配跟踪工具，支持生成火焰图。

OpenTelemetry：开源的可观测数据采集的工具、API和SDK，目前主要支持链路数据采集。

Jaeger：开源的分布式链路跟踪系统。

产品亮点

哈勃监控中心从不同视角提供了功能完备且易用的全方位监控解决方案。下面介绍一些哈勃监控中心的关键特性。

IT资源的全面监控及报警通知

首先，对资源池（主机）、中间件及数据存储组件、微服务和服务管理平台（K8S等）等资源的度量指标（Metric）实现统一采集及管理。

其次，提供了灵活完备的数据可视化分析展示功能。

再次，根据指标特点制定了精准的阈值报警策略及灵活的报警通知策略实现系统故障或瓶颈的及时报警。

最终，实现了将整个IT系统可能存在的资源瓶颈、资源浪费和资源运行故障等关键问题挖掘出来并做到灵活可视化展示和及时预警。

服务运行时动态链路跟踪分析

借助链路跟踪技术，我们能够精准地抓住系统或服务处理每一次用户请求所涉及的关键点。主要内容有，微服务间的Restful / RPC调用及上下游关系，微服务内部的关键调用细节（Method调用、数据库（SQL/NoSQL）调用、缓存调用和MQ生产消费等）。以及这些关键点的耗时或异常信息等。

哈勃监控中心在链路跟踪方面提供了两大使用场景，一种是我们自创的场景级链路录制，另一种是一般意义上的分布式链路跟踪即实时链路。

借助于日志系统健康状态分析

通过对访问日志（Nginx/K8S Ingress）、业务日志（微服务日志）、K8S集群事件，火焰图（CPU/内存），慢SQL等日志的统一采集管理，我们能够与前述指标监控及报警、链路数据分析集合起来帮助我们更准确地明确故障原因并尽快排除故障。

技术概述
技术总览

技术总览

哈勃监控中心采用了多种数据采集器负责数据采集，同时由分别负责监控、链路、日志的后端服务接收和预处理数据，最终将数据存储到多种中间件和存储组件，并提供丰富的数据检索分析和动态数据采集分析功能。

采集器
Prometheus Exporters

Prometheus Exporters负责采集需要监控的IT资产的性能指标。主要的Exporter有：

cAdvisor：容器性能指标采集
MySQL Exporter：MySQL数据库性能指标采集
Postgres Exporter：PostgreSQL数据库性能指标采集
Redis Exporter：Redis数据库性能指标采集
ZooKeeper Exporter：ZooKeeper性能指标采集
RabbitMQ Exporter：RabbitMQ性能指标采集
Nginx VTS Exporter：Nginx性能指标采集
Elasticsearch Exporter：Elasticsearch性能指标采集
Async Profiler

Async Profiler 主要用来生成CPU、内存火焰图。

哈勃链路Agent

哈勃链路Agent分为链路录制和实时链路两个版本，未来将统一为一个Agent。

Fluentd

Fluentd用做所有日志的采集器，目前主要采集了业务日志、访问日志两大类。

中间件/存储组件
Prometheus

主要用来收集和存储主机、服务、中间件等资源的性能指标数据，并提供数据检索聚合API供数据分析之用。

Grafana

Grafana 主要用来可视化呈现来自Prometheus，Elasticsearch以及其他自定义数据源性能数据。

Elasticsearch

Elasticsearch主要存储各类日志数据和链路数据，并提供并提供数据检索聚合API供数据分析之用。

Kibana

Kibana配合Elasticsearch提供数据检索和可视化分析功能。

MinIO

MinIO目前主要用来归档存储业务日志，并依赖其提供业务日志的下载服务。

Kafka

Kafka 主要用来接收链路数据，以实现实时链路数据分析。

InfluxDB

InfluxDB 主要负责存储微服务链路数据，由于微服务链路已经采用新的技术实现，该功能目前已经进入仅维护状态，未来将有新的链路分析功能替代。

MySQL

MySQL最为基础组件，哈勃监控中心的大部分功能都依赖MySQL，除了用做哈勃监控中心基础数据的存储检索，还负责了链路录制数据、性能数据聚合降级存储等。

Redis

Redis主要用做部分静态数据的缓存，提效之用。

公有云日志服务

公有云日志服务主要指阿里云日志服务SLS、华为云日志分析服务LOG等，目前主要用做特定客户的日志存储检索。

公有云文件存储服务

公有云文件存储服务主要指阿里云对象存储OSS、华为云对象存储OBS等，目前主要用做特定客户的日志归档存储下载。

服务
全链路监控服务

全链路监控服务作为哈勃监控中心的旗舰服务，承载了监控中心的大部分功能。

监控数据收集服务主要负责收集处理哈勃链路录制和微服务链路监控两大类数据，其中微服务链路监控未来将由新的链路监控服务替代，目前已进入维护模式。

监控数据收集服务部署依赖（虚线表示可降级）

日志管理服务

日志管理服务主要基于已经采集上来的日志数据提供日志检索相关的服务，主要有业务日志分析，日志下载，SQL慢日志分析等。在专属化环境，日志收集主要依赖Fluentd实现。Fluentd将日志以消息形式推送到Kafka，智能搜索服务消费Kafka将日志存入Elasticsearch。Fluentd同时将日志以文件形式按时间分段存储在Minio提供日志下载服务。在公有云环境，直接采用了公有云服务商提供的日志收集组件将日志采集并存储在公有云日志服务，例如利用阿里云logtail将日志收集并存储到阿里云日志服务SLS中，并借助公有云日志服务提供日志检索下载功能。

功能概述

哈勃监控中心从模块上主要划分为指标监控、日志中心、报警中心和链路跟踪四大模块。

指标监控

指标监控只要包括资源池（主机）监控、微服务监控和中间件监控。

资源池（主机）监控

资源池（主机）监控实际上主要从主机粒度进行监控。

当我们将主机分组管理，也就是引入资源池概念，以满足大型系统各个领域模块主机资源的独立使用，这种场景下资源管理者需要关注单个资源池的主机使用情况，我们支持按照资源池来分析资源是否冗余或者存在资源紧张的情况。

同时支持数据中心和环境维度的资源使用情况分析。

场景化功能介绍

根据关注点的不同，哈勃监控中心分为IT管理工作台、运维工作台、开发工作台、性能工作台和报警中心五大场景。方便不同角色专注分析问题。

IT管理工作台

IT管理工作台功能概览

IT管理工作台主要从性能指标入手，主要呈现主机和服务的CPU使用率、内存使用率、磁盘使用率、磁盘I/O、网路I/O、服务调用数、服务错误调用数、服务慢调用数等关键性能指标。能够直观的展示系统运行过程中各个指标的趋势。

运维工作台

运维工作台功能概览

运维工作台重点从系统访问、中间件性能监控、服务管理平台（K8S等）运行情况切入，重点聚焦公共资源的监控及故障排查。

开发工作台

开发工作台功能概览

开发工作台主要服务于应用开发者，开发者可以利用链路录制、SQL慢日志、业务日志等功能帮助分析评估代码层面的SQL性能、事务大小合理性、链路性能卡点等问题。

性能工作台

性能工作台功能概览

性能工作台提供了CPU、内存火焰图生成工具，支持自动和手动两种生成模式，帮助分析代码性能问题。同时，针对Java应用提供了JVM监控以及内存Dump工具。

报警中心

报警中心功能概览

报警中心提供了报警策略管理，通知规则配置，告警查询认领等功能。支持短信、邮件和友空间等通知模式。

开启哈勃链路录制

如果开启哈勃录制之后，服务启动异常或者哈勃录制功能异常，需要检查以下几点：

如果环境变量配置了hubble_agent_url，请确认Agent版本号是否正确，正确的版本号一般可以联系系统运维人员获的；
检查应用依赖的apm-common是否与Agent版本一致，如果不一致请更新apm-common版本；
服务使用的基础镜像是否正确，基础镜像在发版时可能会更新版本以支持新的基础组件，因此需要检查服务使用的基础镜像会否是更新之后的，正确的基础镜像版本号可以联系系统运维人员获的；
如果故障依然无法排除，可通过反馈渠道寻求产品开发团队支持。
实时链路Agent配置

实时链路可通过增加developer_otel_apm=true环境变量来开启，如果无法正常开启或者数据采集异常，请确认以下几点：

服务是否已经开启哈勃监控，目前哈勃实时链路与哈勃链路录制同时开启可能会有不可预知的冲突，建议不要同时使用；
服务是否启用性能监控，性能监控是基于Pinpoint实现，Pinpoint采集数据使用的Agent原理与实时链路相似，在部分采集点上存在冲突（如，SQL采集等），建议只开启其中之一。

典型场景
故障排查根因分析

如何利用哈勃监控中心进行故障排查及根因分析，下面整理了一个典型的综合性示例，供参考。

问题描述

收到报警反应微服务出现访问超时故障，需要排查故障根因。

检查微服务运行情况

入口：哈勃监控中心 / 开发工作台 / 微服务监控

如图所示，微服务CPU、内存都出现了多次峰值，尤其是6个CPU核心全部用尽。

检查SQL慢日志

入口：哈勃监控中心 / 开发工作台 / SQL慢日志

按照故障发生时间范围查询得出该微服务有一条SQL执行时间25秒，返回结果集7万多行，最大扫描行数1000多万行，共执行2次。

检查K8S集群事件

入口：哈勃监控中心 / 运维工作台 / 集群事件中心

K8S集群事件显示，该微服务在故障发生期间有多次健康检查失败，K8S多次尝试重启Pod试图恢复服务。

检查访问日志

入口：哈勃监控中心 / 运维工作台 / 访问日志

访问日志中出现了大量499状态请求，占比59%左右，进一步证明服务已经严重阻塞。

检查业务日志

入口：哈勃监控中心 / 开发工作台 / 业务日志

业务日志中出现了大量数据库超时异常，大量请求长时间占用数据库连接，已经将数据库连接数耗尽。

检查火焰图

入口：哈勃监控中心 / 性能工作台 / 火焰图

![图形用户界面

中度可信度描述已自动生成](https://ncc-pub-ublim.oss-cn-beijing.aliyuncs.com/help/photo/5d6979f0-b7d6-4f91-ae96-d00ada3d0ec2.png)

哈勃监控中心提供了由报警自动触发CPU火焰图采集的功能。如图所示，在故障发生期间自动触发了三次CPU火焰图采集。

CPU火焰图反映出在故障期间主要是一个方法的调用消耗了几乎所有的CPU，从开发层面很快将问题定位到该方法。

通过对该方法调用相关源代码分析发现逻辑中有两层循环，每层的循环次数大概在7万多次，加上循环内部的业务逻辑处理，这几乎是一个不可能在短期完成的任务。

更有业务平台大神经一步针对该火焰图给出了非常深层次的诊断细节，整理出来供参考学习：

CPU大部分消耗在了java/lang/Long.toString，java/util/HashMap.get两个本该是低CPU消耗的方法调用上，由此判断这两个方法的调用频次极高，进而推断出此处必有死循环。
火焰图中方法ProductSetDetailQueryForNewModelRule.execute提示显示采样次数是32000多次，录制时间长60s，Async Profiler默认采样频率100Hz。综合这些数据，由此（32000 / 60 / 100 = 5.3）计算说明当时差不多有6个线程被困在这个死循环里了，刚好这个实例分配了6核CPU，当时抓到CPU占用率是600%。
六个线程把CPU死死抱住，其他线程只能得到一点点时间片，导致无法处理完数据库返回的数据，都变成慢请求，500个HTTP线程全被占满。近300个HTTP线程在等待，占用了全部200个数据库连接，一分钟后纷纷触发从连接池获取数据库连接超时。
至此，所有的日志和监控数据都得到了解释，结案
总结

当故障发生时，可以利用监控、K8S集群事件、访问日志、业务日志、SQL慢日志和火焰图等数据和工具结合起来共同辅助分析问题故障，尤其是要关注报警触发采集的火焰图来帮助分析故障根因，最后探索出解决故障的最终方案。
