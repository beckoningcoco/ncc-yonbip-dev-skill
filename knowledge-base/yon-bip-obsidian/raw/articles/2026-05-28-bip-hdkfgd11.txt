iuap-开发规范红皮书
最后更新时间：2024-10-23
第一章 概述

本规范主要针对开发做出规范，使公司各领域产品的代码具有规范性、统一性、可读性，减少编码中不必要的错误，提高软件质量与开发效率。帮助开发养成良好的编程习惯，写出清晰易懂、易维护的代码。

第二章 适用范围

本规范适用于用友云产品开发，软件服务序列由于已有技术框架不同，可以作为技术参考，不作为强制规范。

统一三方包规范

通过 yms 统一管理可引入的第三方包及三方包的版本号。统一三方包是对各领域开发中使用的开源框架、及对应的版本的统一管理，其主要目标包括：

开源协议问题，避免开发中使用协议有问题的三方包；
引用其他业务组开发的组件时，避免产生包冲突；
安装盘合包问题，避免安装盘合包出现版本冲突；
安全问题，避免使用的三方包版本存在严重漏洞；

Java 开发规范
命名风格
 *           *                1. 【强制*】代码中的命名均不能以下划线或美元符号开始，也不能以下划线或美元符号结束。


反例：name / name / $Object / name / name$ / Object$

 *           *                1. 【推荐】代码中的命名严禁使用拼音与英文混合的方式


说明：正确的英文拼写和语法可以让阅读者易于理解，避免歧义。注意，即使纯拼音命名方式也要避免采用。

正例：alibaba / taobao / youku / hangzhou 等国际通用的名称，可视同英文。

反例：DaZhePromotion [打折] / getPingfenByName() [评分]

 *           *                1. 【强制】不允许直接使用中文的方式。


说明：正确的英文拼写和语法可以让阅读者易于理解，避免歧义。注意，即使纯拼音命名方式也要避免采用。

反例：int 某变量 = 3

 *           *                1. 【强制】类名使用 UpperCamelCase 风格，必须遵从驼峰形式，但以下情形例外： DO / BO / DTO / VO / AO


正例：MarcoPolo / UserDO / XmlService / TcpUdpDeal / TaPromotion

反例：macroPolo / UserDo / XMLService / TCPUDPDeal / TAPromotion

 *           *                1. 【强制】方法名、参数名、成员变量、局部变量都统一使用 lowerCamelCase 风格， 必须遵从驼峰形式。


正例： localValue / getHttpMessage() / inputUserId

 *           *                1. 【强制】常量命名全部大写，单词间用下划线隔开，力求语义表达完整清楚，不要嫌名字长。


正例：MAX_STOCK_COUNT 反例：MAX_COUNT

 *           *                1. 【推荐】抽象类命名使用 Abstract 或 Base 开头；异常类命名使用 Exception 结尾；测试类命名以它要测试的类的名称开始，以 Test结尾。
   2. 【强制】中括号是数组类型的一部分，数组定义如下：String[] args; 反例：使用 String args[]的方式来定义。
   3. 【强制】POJO类化错误。


中布尔类型的变量，都不要加 is，否则部分框架解析会引起序列

反例：定义为基本数据类型 Boolean isDeleted；的属性，它的方法也是isDeleted()，RPC框架在反向解析的时候，“以为”对应的属性名称是deleted，导致属性获取不到，进而抛出异常。

 *           *                1. 【强制】包名统一使用小写，点分隔符之间有且仅有一个自然语义的英语单词。包名统一使用单数形式，但是类名如果有复数含义，类名可以使用复数形式。


正例： 应用工具类包名为 com.common.util、类名为 MessageUtils（此规则参考 spring的框架结构）

 *           *                1. 【推荐】为了达到代码自解释的目标，任何自定义编程元素在命名时，使用尽量完整的单词组合来表达其意。


正例：从远程仓库拉取代码的类命名为

PullCodeFromRemoteRepository 。 反例：变量 int a; 的随意命名方式。

 *           *                1. 【推荐】如果模块、接口、类、方法使用了设计模式，在命名时体现出具体模式。说明：将设计模式体现在名字中，有利于阅读者快速理解架构设计理念。


正例：

public class OrderFactory; public class LoginProxy;

public class ResourceObserver;public class ResourceObserver;

 *           *                1. 【推荐】接口类中的方法和属性不要加任何修饰符号（public 也不要加），保持代码的简洁性，并加上有效的 Javadoc注释。尽量不要在接口里定义变量，如果一定要定义变量，肯定是与接口方法相关，并且是整个应用的基础常量。


正例：接口方法签名：void f();

接口基础常量表示：String COMPANY = "alibaba"; 反例：接口方法定义：public abstract void f();

说明：JDK8 中接口允许有默认实现，那么这个 default 方法，是对所有实现类都有价值的默认实现。

 *           *                1. 接口和实现类的命名有两套规则：


1）【推荐】对于 Service和 DAO类，基于 SOA的理念，暴露出来的服务一定是

接口，内部的实现类用 Impl的 后缀与接口区别。

正例：CacheServiceImpl实 现 CacheService接口。

2）【推荐】如果是形容能力的接口名称，取对应的形容词做接口名（通常是– able的形式）。

正例：AbstractTranslator实现 Translatable。

 *           *                1. 【推荐】枚举类名建议带上 Enum后缀，枚举成员名称需要全大写，单词间用下划


线隔开。

说明：枚举其实就是特殊的常量类，且构造方法被默认强制是私有。

正例： 枚举名字为 ProcessStatusEnum 的 成员名称： SUCCESS / UNKOWN_REASON。

 *           *                1. 【推荐*】谨慎注释掉代码。在上方详细说明，而不是简单的注释掉。如果无用， 则删除。


说明：代码被注释掉有两种可能性：

1）后续会恢复此段代码逻辑。

2）永久不用。

前者如果没有备注信息，难以知晓注释动机。

后者建议直接删掉（代码仓库保存了历史代码）。

 *           *                1. 【推荐】对于注释的要求：第一、能够准确反应设计思想和代码逻辑；第二、能够描述业务含义，使别的程序员能够迅速了解到代码背后的信息。完全没有注释的大段代码对于阅读者形同天书，注释是给自己看的，即使隔很长时间，也能清晰理解当时的思路；注释也是给继任者看的，使其能够快速接替自己的工作。
   2. 【推荐】好的命名、代码结构是自解释的，注释力求精简准确、表达到位。避免出现注释的一个极端：过多过滥的注释，代码的逻辑一旦修改，修改注释是相当大的负担。

异常规范
异常开发

平台提供基础异常 BusinessException 继承自 RuntimeException，属于非受查异常，领域可以只针对要处理的异常进行捕获，不需要全部处理。

 *           *                1. 目前绝大部分领域原厂开发时使用的都是 BusinessException，这类异常不强迫各层严格捕获并上跑，只在框架最外层进行处理。建议领域开发时使用此类异常，仅在需要处理时对异常进行捕获和处理；
   2. 对第三方提供的 Open API，或者期望调用方特定异常并处理时，请明确方法抛出指定异常。


所有领域的业务异常均需要继承 BusinessException，完整的异常信息包括异常编码、错误提示、异常堆栈、以及异常类型。

其中错误提示需要支持多语展示，平台已经支持对代码中的词条进行扫描和多语翻译，不开启多语的话走默认显示简体中文。领域在增加新的异常信息后需要使用平台工具完成多语词条抽取；
异常类型在异常码中进行体现，用于在错误中心内对异常信息进行分析和归类通知； 
异常堆栈需要记录异常发生时完整的初始异常原因；
异常编码规范见 2.2。

错误信息返回前端时，按实际的异常编码返回，放到错误详情中，不要统一包装成999。领域内部的异常码和 OpenAPI 的异常码也保持一致，由开放平台/帮助中心维护各领域 API 的异常码信息列表（同时含错误原因和修复动作）。

异常编码

异常编码由领域前缀和异常序号来组成，格式为“领域前缀-异常序号”，通过“-”连接，具体规则如下：

 *           *                1. 领域前缀由领域编码和子领域/服务编码组成，格式为“领域编码-子领域/


服务编码”，具体的编码值和 yms 微服务编码中的后两段保持一致；

 *           *                *                     1. 原厂开发已有的微服务在抛异常时可以通过 yms 框架来获取，新的微服务编码在开发前统一向总体架构部提出申请；
      2. 行业以及 ISV/伙伴的领域编码和子领域/服务编码需要向产品管理部申请分配；
        
   1. 异常序号：六位数字，第一位代表具体的异常类别，0xxxxx-4xxxxx 为公共


类异常，5xxxxx-9xxxxx 为业务类异常。

 *           *                *                     1. 发生异常的时候可以根据不同的异常类别向设置的人员发送邮件或信息，同一个 traceid 的异常只发送一次。


 *           *                1. 在表示业务异常时，5、6、7 开头的异常表示遇到该异常不需要再进行重试，8 开头的异常则表示遇到该异常可以发起重试，见下表。

其他要求
 *           *                1. 产品模块的所有子异常都需要来自基本异常 BusinessException，业务在抛出异常时须定义明确的异常编码。
   2. 所有的异常名称必须以 Exception 结尾，所有的异常定义都必须包含异常链， 提供对原始异常的跟踪，在日志里输出异常信息时遵循日志规范。
   3. 在多语资源文件中增加异常编码对应的多语消息。
   4. 异常定义时区分 unchecked/checked 异常，避免直接抛出 new RuntimeException()，更不允许抛出 Exception 或者 Throwable，应使用有业务含义的自定义异常，如 DAOException、ServiceException 等。
   5. 抛出的异常必须具备一定的描述该异常发生的原因的信息，以便定位错误业务代码不能直接抛出没有任何业务意义的异常如 throw new Exception(“msg”)。
   6. 异常捕捉时，最一般的异常永远放在最后的 catch 部分，捕捉异常应该尽量的精确，这样对各个异常的处理才能按照不同的异常进行更好的业务处理。
   7. 控制 try/catch 的规模，通常代码中由多个位置抛出多种异常，针对每一类异常单独的进行捕捉。如果代码中涉及到敏感的资源如 Connection、Socket,文件等，一定要加上 finally 处理块，释放资源。若资源类实现了AutoCloseable 接口，则可以在 JDK 8 里使用 try-with-resources 方式。
   8. 不要吃掉可能导致业务问题的异常，NullPointException、ArrayIndexOutofBOundException、DbException、IllegalArgumentException 等异常通常是业务相关的，需要按照业务逻辑进行捕捉处理。
   9. 如果异常需要抛出给前端应用，出于安全的考虑可能不希望前端程序保留异常的链，这时候需要首先日志改异常，然后把该异常转化为业务异常，去掉异常链抛出。
   10. 如果捕捉到的异常处理抛出后去掉了异常链，对原始异常应该首先进行日志。如果异常抛出后保持了异常链，在抛出之前对原始的异常不需要进行日志工 作。
   11. 应用程序一般不需要关心运行时刻异常(RuntimeException 及子类)或者错误


（Error 或者子类）。

 *           *                1. DBException 和 SQLException 都是受查异常，业务系统在捕捉异常后，如果不能按照逻辑正常地进行处理，需要直接转化为产品模块的业务异常再进行抛 出，不需要保持异常链，但注意在抛出异常之前需要进行日志工作。
   2. 如果异常已经在 throws 语句中抛出，一般不需要进行捕捉和处理。
   3. 如果捕捉到异常后不把异常再次抛出，则需要对异常进行日志工作，日志的级别根据业务逻辑可以选用 debug 或者 warn。对于对系统的表现有一定影响的采用 warn，而几乎没有影响的采用 debug。
   4. 如果捕捉了 InvokeTragetException，不能把该异常直接抛出，而是需要对其中的目标异常进行处理(getTargetException())。
   5. Java 类库中定义的一类 RuntimeException 可以通过预先检查进行规避，而不 应 该 通 过 catch 来 处 理 ， 比 如 IndexOutOfBoundsException， NullPointerException 等。


正例：if (obj != null) {...} 反例：try { obj.method() }

catch (NullPointerException e) {...}

无法通过预检查的异常除外，如在解析一个外部传来的字符串形式数字时，通过 catch NumberFormatException 来实现。

 *           *                1. 异常不要用来做流程控制，条件控制，因为异常的处理效率比条件分支低。
   2. try 块放到了事务代码中，catch 异常后，如果需要回滚事务，一定要注意手动回滚事务。
   3. 不能在 finally 块中使用 return，finally 块中的 return 返回后方法结束执行，不会再执行 try 块中的 return 语句。
   4. 捕获异常与抛异常，必须是完全匹配，或者捕获异常是抛异常的父类。
   5. 方法的返回值可以为 null，不强制返回空集合，或者空对象等，必须添加注释充分说明什么情况下会返回 null 值，调用方也同时需要进行非空判断防止NPE 问题。
   6. 注意 NPE 产生的场景： 
 
      1. 返回类型为基本数据类型，return 包装数据类型的对象时，自动拆箱有可能产生 NPE。反例：public int f() { return Integer obj}， 如果为null，自动解箱时会抛 NPE；
      2. 数据库的查询结果可能为 null；
      3. 集合里的元素即使 isNotEmpty，取出的数据元素也可能为 null；
      4. 远程调用返回对象时，一律要求进行空指针判断，防止 NPE；
      5. 对于 Session 中获取的数据，建议 NPE 检查，避免空指针；
      6. 级联调用 obj.getA().getB().getC() 一连串调用，易产生 NPE。可以使用 JDK8 的 Optional 类来防止 NPE 问题。

日志规范
日志级别

目前规定日志支持四种日志级别 DEBUG、INFO、WARN、ERROR，顺序为 DEBUG<INFO

<WARN<ERROR，如果日志级别调的较高，低级别的日志就不能输出。例如设置为 WARN， 那么 DEBUG 与 INFO 的信息就不能输出。对这四个级别的选择如下：

 *           *                1. DEBUG: 对调试（debug）有帮助的信息、事件，默认情况下不打开日志记录，主要用于开发环境的信息输出；
   2. INFO: 在正常运行状态中，粗粒度的、关键流程/事件、业务软件重要的配置和参数信息，以及可用来表示业务软件健康度的信息，如程序运行所花费的时间等；
   3. WARN:有潜在风险的、不会造成大危害的错误，需要开发人员给予足够关注，往往表示有校验问题或者程序逻辑缺陷，如系统对用户输入参数校验时发生错误， 输出警告信息；
   4. ERROR: 业务软件发生错误，后续流程还能继续进行，但也已经影响了用户的正常访问。该级别的错误需要马上被处理，如服务端抛出了一个NullPointException,导致系统不能正常运行，进行错误信息记录。


在使用日志等级时要有严格的等级评估意识，科学使用 WARN、INFO 等级别，后续可以针对日志级别出现次数进行统计监控预警以及优化代码，提升应用健壮性。

 *           *                1. 系统运行时默认日志输出级别为 ERROR，此级别只记录系统逻辑出错、异常或者重要的错误信息。异常日志输出时需要包含明确的异常编码，这样就可以在日志中按租户 ID、用户 ID、异常编码进行检索。平台在调研运行中时遇到异常后再对调用链中的入参进行日志记录，这样既可不用输出过多日志，又可快速定位解决用户因输入参数校验错误投诉的场景，而不需要用户打开 Hubble 重试；

分布式缓存规范

适用范围**:**本规范主要面向用友云的 Redis 开发规范，从键值设计、命令使用、客户端使用、相关工具等方面进行说明，遵循此规范可以减少使用 Redis 过程带来的问题。

缓存设计
【推荐】避免缓存穿透

数据库中未查询到的数据，可在 Redis 中设置特殊标识，以避免因缓存中无数据而导致每次请求均达到数据库。

【推荐】避免缓存雪崩

当大量缓存集中在某一个时间段失效，这样在失效的时候也会给数据库带来很大压力。

【推荐】避免缓存击穿

某个 key 的缓存过期后，同一时间内有大量的请求均访问该 key，由于缓存过期， 大量的请求均会访问数据库，并重建缓存；重建缓存的过程加锁，保证只有一个人执行，其他人等待。

【推荐】可以进行适当的缓存预热

对于上线后可能会有大量读请求的应用，在上线之前可预先将数据写入缓存中

【推荐】数据一致性问题

数据源发生变更时可能导致缓存中数据与数据源中数据不一致，应根据实际业务需求来选择适当的缓存更新策略：

 1. 主动更新：在数据源发生变更时同步更新缓存数据或将缓存数据过期。一致性高，维护成本较高。

被动删除：根据缓存设置的过期时间有 Redis 负责数据的过期删除。一致性较低，维护成本较低。
推荐策略：主动更新，数据源发生变更时将缓存数据过期。缓存和 DB 的更新不在同一个事务问题：

利用 Spring 的事务管理机制，在事务管理器上注册一个事务提交后回调，在回调方法中进行缓存清理；

使用异步消息队列，并且支持重试补偿机制。（如果缓存里数据时间与数据库时间不能匹配，意味着另外一个服务更新了该数据，那么就先从 DB 里读取最新数据版本， 然后在新版本上提交数据）

键值设计
【强制】key 值不要包含特殊字符

Redis 以“\n”作为命令结束符，以空格作为命令和参数分隔符。反例：包含空格、换行、单双引号以及其他转义字符

【强制】拒绝 bigkey(防止网卡流量、慢查询)

反例：一个包含 200 万个元素的 list。非字符串的 bigkey，不要使用 del 删除， 使用 hscan、sscan、zscan 方式渐进式删除，同时要注意防止 bigkey 过期时间自动删除问题(例如一个 200 万的 zset 设置 1 小时过期，会触发 del 操作，造成阻塞，而且该操作不会出现在慢查询中(latency 可查))

【推荐】key 值可读性和可管理性

以业务名(或数据库名)为前缀(防止 key 冲突)，用冒号分隔，比如[模块名]:[业务描述]:[ID]

【推荐】key 值简洁性

保证语义的前提下，控制 key 的长度，当 key 较多时，内存占用也不容忽视。

【推荐】根据业务场景合理使用不同数据结构类型

目前 Redis 支持的数据库结构类型较多：字符串（String），哈希（Hash），列表

（List），集合（Set），有序集合（Sorted Set）, Bitmap, HyperLogLog 和地理空间索引（geospatial）等,需要根据业务场景选择合适的类型，常见的如： String 可以用作普通的 K-V、计数类；Hash 可以用作对象如商品、经纪人等，包含较多属性的信息；List 可以用作消息队列等；Set 可以用于推荐；Sorted Set 可以用于排行榜等。

【推荐】控制 key 的生命周期

redis 不是垃圾桶，建议使用 expire 设置过期时间(条件允许可以打散过期时间， 防止集中过期)，不过期的数据重点关注 idletime。

【推荐】冷热数据分离，不要将所有数据全部都放到 Redis 中

虽然 Redis 支持持久化，但是 Redis 的数据存储全部都是在内存中的，成本昂贵， 建议根据业务只将高频热数据存储到 Redis 中。

命令使用
【强制】线上禁止使用 keys 命令

Redis 是单线程处理，在线上 KEY 数量较多时，操作效率极低【时间复杂度为O(N)】，该命令一旦执行会严重阻塞线上其它命令的正常请求，而且在高 QPS 情况下会直接造成 Redis 服务崩溃。如果有类似需求，请使用 scan 命令代替。

【强制】禁用命令

禁止线上使用 flushall、flushdb 等，flushall、flushdb 会清空 redis 数据。通过 redis 的 rename 机制禁掉命令，或者使用 scan 的方式渐进式处理。

【推荐】O(N)命令关注 N 的数量

例如 hgetall、lrange、smembers、zrange、sinter 等并非不能使用，但是需要明确 N 的值。有遍历的需求可以使用 hscan、sscan、zscan 代替。

【推荐】合理使用 select

redis 的多数据库较弱，使用数字进行区分，很多客户端支持较差，同时多业务用多数据库实际还是单线程处理，会有干扰。

【推荐】使用批量操作提高效率

原生命令 mset、mget，非原生命令 pipline；但要注意控制一次批量操作的元素个数(例如 500 以内，实际也和元素字节数有关)

【推荐】Redis 事务功能较弱，不建议过多使用

Redis 的事务功能较弱(不支持回滚)，而且集群版本(自研和官方)要求一次事务操作的 key 必须在一个 slot 上(可以使用 hashtag 功能解决)

【推荐】Redis 集群版本在使用 Lua 上有特殊要求：
 *  Redis 从 2.6 版本开始引入对 Lua 脚本的支持

所有 key 都应该由 KEYS 数组来传递，redis.call/pcall 里面调用的 redis命令，key 的位置，必须是 KEYS array, 否则直接返回 error
Redis 集群对多 key 操作有限制，要求命令中所有的 key 都属于一个 slot， 才可以被执行。
【推荐】必要情况下使用 monitor 命令时，要注意不要长时间使用

monitor 命令在开启的情况下会降低 redis 的吞吐量，根据压测结果大概会降低redis50%的吞吐量，越多客户端开启该命令，吞吐量下降会越多。

客户端使用
【推荐】避免多个应用使用一个实例

避免多个应用使用一个 Redis 实例。不要将不相关的业务数据都放到一个实例中， 建议新业务申请新的单独实例。因为 Redis 为单线程处理，独立存储会减少不同业务相互操作的影响，提高请求响应速度；同时也避免单个实例内存数据量膨胀过 大，在出现异常情况时可以更快恢复服务，公共数据做服务化。

【推荐】使用连接池

使用带有连接池的数据库，可以有效控制连接，同时提高效率

【推荐】添加熔断功能

高并发下建议客户端添加熔断功能(例如 netflix hystrix)

【强制】设置合理的密码

设置合理的密码，如有必要可以使用 SSL 加密访问（阿里云 Redis 支持）

【推荐】根据业务类型选择淘汰策略
 *  根据自身业务类型，选好 maxmemory-policy(最大内存淘汰策略)，设置好过期时间。默认策略是 volatile-lru，即超过最大内存后，在过期键中使用 lru 算法进行 key 的剔除，保证不过期数据不被删除，但是可能会出现 OOM 问题。

其他策略如下：
allkeys-lru：根据 LRU 算法删除键，不管数据有没有设置超时属性，直到腾出足够空间为止。
allkeys-random：随机删除所有键，直到腾出足够空间为止。
volatile-random:随机删除过期键，直到腾出足够空间为止。
volatile-ttl：根据键值对象的 ttl 属性，删除最近将要过期数据。如果没有，回退到 noeviction 策略。
noeviction：不会剔除任何数据，拒绝所有写入操作并返回客户端错误信息"(error) OOM command not allowed when used memory"，此时 Redis 只响应读操作。
【强制】设置连接池时，如果非默认 database，直接在 URI 中指定，不能每次查询 selectDb

如 ：new JedisPool(config,new URI("http://redis:password@host:port/database)

分布式锁规范

适用范围：本规范主要面使用业务上用于控制并发的场景下，使用分布式锁组件组件时的注意事项，避免发生问题。

分布式锁使用场景

分布式锁的主要作用是用于控制业务并发，原理是生成一个字符串，向分布式锁服务申请，如果其他的服务或线程申请相同字符串时，返回锁已存在，提示加锁失 败。

分布式锁的使用场景有两个：
 *  **效率：**使用分布式锁可以避免不同节点重复相同的工作，这些工作会浪费资源。例如缓存初始化。

**正确性：**加分布式锁可以避免破坏数据正确性的发生。微服务场景下，业务扩多个应用时，其他人操作数据会导致数据发生问题。
不推荐使用分布式锁的场景：
 *  单一服务内的可以使用数据库乐观锁保证数据并发修改的场景。

redis 和 zookeeper 等框架存在独立的事务处理方案，建议使用本事的事务来保证数据的并非修改问题，不建议使用分布式锁。
业务执行时间过长，导致锁超时失效的场景（建议通过数据级别的锁机制保障，例如建立单独的锁表，增加记录，数据库锁控制）** **
分布式锁使用规范
【推荐】指定合理的超时时间， 不建议设置超时时间为空（即不超时）。
 *  对于现有的微服务的运行环境，应用发生宕机、或者运维人员或系统对应用进行缩容时，可能都会导致当前服务的中断。在出现此类异常时，锁不会被释放。如果设置锁的超时时间为空，会导致下次服务重启后，业务重新运行


时，加锁失败，导致业务无法执行。为避免该问题，需要为锁指定超时的时间。

 *  超时时间，建议根据业务系统的加锁场景确定，要超过所有加锁场景下的最长业务执行时间。建议可以把超时时间尽量设置长一些，例如 30 分钟以上， 但不建议设置过长，例如超过 1 天。

【推荐】对于 web 服务建议配置过滤器，防止忘记解锁。
 *  通过过滤器，确保业务开发忘记解锁时，在调用完成后，通过拦截器对线程中加锁的锁键值进行解锁。在响应式编程时，此方案不推荐，建议有发起应用自己控制解锁。

【强制】有数据库事物的情况下，使用 spring 事务拦截器进行解锁。
 *  在部分场景下，非跨服务的数据库保存操作时，可以考虑使用 spring 的事物拦截器进行解锁。示例代码如下：

统一集成规范

适用范围：随着业务的发展，领域间融合程度越来越高；本规范定义了如何快速实现各产品进行集成。

目的
 *           *                *  在开发、测试、预发、生产等多环境下把相关的服务无缝的集成起来。
   *  协调一致、减少沟通
   *  内建质量、自动化
   *  统一资源申请、管理
   *  统一中间件安装、维护
   *  统一发布流程，节奏控制

环境定义
 *           *                *  【建议】开发环境——定义：用于开发人员进行测试。管理：由开发人员管理，开发人员进行功能调试
   *  【强制】测试环境——定义：用于开发完成后的联调自测。管理：由开发人员启动代码提交-构建-部署。要求：自测完成有需求和测试确认后，允许开发提交下一环境（测试环境）
   *  【强制】日常环境——定义：用于提交集成测试。管理：由测试人员管理，经测试批准，开发负责人合并代码完成构建部署，接受测试。要求：测试发现问题，要求开发在特性分支重新修改后，再次请求合并代码至测试环境；通过全面测试后，由测试发起上日常请求
   *  【强制】预发环境——定义：用于上线前的预发布测试，与生产环境连接同一套生产数据。管理：由发布负责人管理，响应测试人员请求，合并代码完成构建部署。要求：测试发现问题，要求开发重新修改并再次经过测试环境的测试后，再由测试通知运维重新合并代码上预发；完成主流程回归和新特性测试 后，由测试及研发负责人审核上生产
   *  【强制】生产环境——定义：线上正式发布环境。管理：由发布负责人负责部署上线要求：由预发布环境测试通过的部署包推向生产环境部署。特别：灰度环境对应灰度发布，指部分产品特性为部分用户开放，没有用户问题后再逐步平滑过渡到全部用户开放

代码版本管理
 *           *                *  【强制】分支分类—— 特性分支、develop 分支、release 分支、master 分支
   *  【强制】开发环境：develop，新特征开发建议创建 Feature 分支，待特性分支开发完成后合并到 develop 分支进行测试

SQL 脚本规范
初始化脚本

专属化需要预置初始化脚本，包括 DDL 建库脚本和 DML 预置数据脚本。

准备脚本目录
脚本的分类

专属化脚本指所有数据库 sql 脚本。

从脚本类型上区分包括不同数据库类型的 sql 文件，mongodb 的脚本文件（zip 格式），groovy 脚本（通过程序处理 sql），目前存在以上这几种脚本。

从执行库不同区分的话，存在三种脚本：mdd 框架的公共脚本（基于 mdd 开发的服务会涉及到）、插入到公共服务的脚本、插入到业务服务自身库的脚本。

原则

脚本目录规范就是设计好目录结构，能规范好以上这些不同的脚本，使安装器能够正确识别脚本，执行到正确的库里。

脚本的原则：

 *  每个脚本都需要有所有者，所有者管理自己的所有脚本。预置在公共服务里的数据，需要抽取出来 dml 脚本放在业务服务里。

脚本根目录 scripts 放在微服务打包工程里，与 src 同级，确保脚本与代码一样打包到 war 包。
所有脚本应该是支持幂等。 
脚本目录结构

初始化脚本放在 init 目录下，init 目录下区分不同的数据库类型：mongodb、rdb

（代表关系型数据库），在不同的数据库类型目录下要放对应数据库语法的脚本。以下以 rdb 为例，mongodb 的目录结构同 rdb。

升级脚本

准备脚本目录

upgrade 放的是金盘发布的本版本对某一历史版本的数据库升级脚本，如针对YonBIP202105 的升级，或者针对 YonBIP202111 的升级等。

脚本要求幂等，升级脚本都是每次大版本发布时从 patch 脚本中迁移过来的。通过 DBDescribe.xml 来描述升级目录。

脚本导出工具

upgrade 脚本没有专门的导出脚本工具。

每个大版本整理升级脚本，是通过把从上一版积累下来的 patch 中脚本迁移到升级目录中而来的。

脚本描述文件

描述文件中关于升级的描述如下：  fromVersion：指的是从哪个版本升级

path：是升级要执行哪几个目录，配置的是 upgrade 直接下层目录名，多个用分号隔开

安装器的逻辑：

当发现目标版本是 YonBIP202105 时，那么找 fromversion=YonBIP202105 的配置， 依次执行 path 中配置的目录脚本，最后执行 patch。

针对每一个大版本，都需要配置一个升级描述（同时有升级目录）。

小版本不需要，比如 202203 是小版本，没有整理升级目录，同时没有配置描述文件。 那么：

 *           *                *                     1. 针对目标是 202105 版，执行 YonBIP202105-YonBIP202111 目录，同时执行patch
      2. 针对目标是 202111 版，只执行 patch

公有云脚本流水线

公有云执行脚本必须通过配置脚本流水线的方式，不允许手工执行脚本到数据库中。公有云脚本流水线只执行 patch 目录下的脚本。

公有云脚本流水线由运维创建。

**【强制】**sql 文件编码都是 UTF8，大部分数据库都是 utf8
