函数组件
最后更新时间：2024-03-05
概述

在低代码的开发过程中，需要对业务的服务器端逻辑进行扩展开发，YonScript提供了云环境下服务器端编程的运行引擎，并且提供了代码编写，调试，部署工具的配套工具。相比传统的JVM脚本引擎，提升了安全性和性能。和传统的FaaS服务类似，使用YonScript只需要关注业务代码逻辑，无需关注服务器资源。YonScript提供了SPI层，可以提供函数扩展能力，并且扩展了大量的预制函数，可以无缝地和YonBIP的服务进行集成，如操作元数据对象进行持久化，操作YonQL进行复杂的查询，发送协作消息等。YonScript提供了多种触发机制，可以基于事件，API网关， MDD 规则，流程来触发，也可以通过JAVA SDK来直接调用。

基本概念
YonScript功能架构

YonScript由YonScript-V8 、YonScript-Engine、 Yonscript-MDD、开发工具组成。

YonScript V8 VM

YonScript内核部分基于Eclipse 基金会的J2V8项目，这个项目为Android系统下提供了一个轻量级高性能的JavaScript脚本引擎，大量Android　APP使用这个引擎来实现业务逻辑的热更新。目前官方只提供Android系统下的AAR包，为了能在云端和PC端使用，我们将J2V8从Android系统移植到Windows, MacOS,Linux(amd64,aarch64)系统,主要工作是适配不同的操作系统和指令集下的头文件、编译脚本。

J2V8

J2V8 是对 Google 的 JavaScript 引擎 V8 的Java 封装，在执行效率与内存占用上都有不俗的表现。

尽可能的返回基础类型

为了更好的性能表现，J2V8在返回JS脚本的执行结果时会尽可能的返回对应的Java基本类型，而不是创建一个新的JAVA类实例。同时J2V8还提供了一种懒加载机制，只有当JS的对象被访问时才会通过JNI机制向JAVA环境进行复制。

辅助内存管理

J2V8是一系列的V8接口的Java封装，V8引擎是C++编写的。为了访问V8对象，J2V8通过JNI机制调用V8Api，因为两种语言间的区别产生了内存管理上的问题。J2v8提供了一套内存管理机制。帮助用户释放不再需要的V8对象防止内存泄漏。

线程模型

Js是单线程执行的，J2V8在运行Js脚本时保证了执行线程是单线程的。访问Js对象的的线程必须是相同的,这种机制防止了死锁的发生。而当多线程的java程序执行V8脚本时，J2V8提供的官方解决方法是采用多个独立的运行时。

J2V8对象池

上面提到J2V8不允许不同线程对V8运行时对象进行访问，而是采用新建独立运行时执行，这种方案不太适合在高并发的web应用场景。因此我们采用类似连接池的J2V8对象池执行机制。

适配不同环境

早期的J2v8支持Android 、Windows、MacOS、Linux操作系统，但是随着项目的发展目前支持Android操作系统了。同时因为J2v8项目涉及了V8、JNI、JAVA三个部分的编译工作、编译脚本较为复杂，为了支援开发与生产环境需要我们对编译脚本进行了修改，实现了对以下环境的支持：

Windows-x64
MacOS -x64
Linux-x64
Linux- aarch_64

V8vm因为针对不同环境有不同格式的动态链接库，所以需要根据操作系统选择对应的maven依赖包。为了简化操作，我们再后端脚手架的maven工程中加入了自动选择匹配当前系统依赖包的配置。

SPI扩展机制
简介

Yonscript的J2V8引擎提供了函数的运行环境，而实际业务开发中需要各种函数支持。为了提供扩展性。我们基于java SPI机制，设计了Yonscript的函数扩展功能。

扩展逻辑

整个函数扩展逻辑以java spi为基础。通过spi读入实现了YonScriptNativeFunction接口的java函数类。该类提供了函数名、函数定义、函数逻辑等信息。运行引擎通过读入的信息、在J2V8运行时中设置对应的函数入口、最终交给JS运行上下文生成函数定义，供开发者调用。

模块分割

因为Yonscript的扩展函数种类繁多，为了能够更清晰的管理函数扩展。我们以函数的作用为标签，将其分成了三类，封装在了两个模块内（Yonscript-engine和Yonscript-mdf）。

Yonscript-engine封装了运行引擎所必须的基础函数，方便engine模块的复用。

Yonscript-mdf封装了Yonbuilder平台所使用的mdd框架的相关函数、并且集成了一些业务方的函数。这样将mdd和业务方的依赖控制在mdf包内，避免引擎底层包引用无关依赖。

缓存管理

为了提升YonScript-engine的文件加载性能，避免给服务器带来压力。我们设计了一套本地缓存机制，来加速脚本文件的加载。前端后端的所有扩展脚本都以调用扩展脚本服务的方式跟运行时进行集成，为强化性能和同步时效，设计了缓存加inotify同步的策略。以设计器为前端绑定事件函数为例，在设计态下总体流程为下图。

功能介绍
模块化
概述

YonScript为用户设置了多种触发类型的函数扩展点，用户通过开发自己的业务扩展函数并挂载到对应的扩展点实现后端函数扩展功能。

不同触发类型的上下文环境以及参数差异较大，需要解析类解析参数传给对应触发模块，同时JS引擎方面也需要释放执行完毕以后的函数相关对象。为了解决上述问题，应用引擎采用了模块化的函数引入机制，读入外部扩展脚本、解析上下文参数、管理JS运行时对象。

扩展函数模块化读取机制

函数引擎通过ModuleLoader工具读取扩展函数，并根据扩展函数实现的扩展类加载匹配的业务处理逻辑，以便进行后续的参数、上下文解析等工作。

扩展函数类型

扩展函数目前预制了4种类型：

APi扩展（AbstractAPIHandler）
事件扩展（AbstractEventHandler）
触发器扩展（AbstractTrigger）
工作流扩展（AbstractWorkflowAPIHandler）

每一项扩展函数类型都对应了一种触发规则和相匹配的Js处理脚本。当用户请求符合相应的触发条件时，会将请求的上下文和参数交给函数引擎，引擎提前加载的FunctionWarrper脚本会通过ModuleLoader读取处理脚本解析相关参数，并执行用户扩展逻辑最终返回执行结果。

扩展函数类型判断

ModuleLoader实现了可扩展的函数类型判断功能，当用户函数读入后，可以根据函数脚本内容加载对应的处理器，同时处理器可以根据需要进行扩展。

函数脚本分类

函数根据使用场景主要分为前端函数和后端函数，以下介绍两种函数的触发机制和使用示例。

后端函数
功能介绍

后端函数执行依赖于MDD规则扩展来实现，函数SDK实现了extBackEndRule 规则，extBackEndRule 规则通过租户下的函数ID获取函数内容并执行，租户下单据动作规则链可以新增函数规则来实现函数扩展，例如可以在单据save动作最后添加一个函数规则来实现保存之后的扩展逻辑。

function _workflowApiExec_(contextJson) {
    let workflowClass = require('entryPoint');
    var workflow = new workflowClass();
    var context = JSON.parse(contextJson);
    var methodname = context.methodname;
    var params = context.params;
    var res = {}
    try {
        var result = workflow[methodname](params);
        res.info=result;
        res.message = 'success';
    }catch(e) {
        res.message="fail";
        if (e.message) {
            res.exception = e.message;
            // 日志里打印异常信息
            console.error("函数执行失败，异常信息：{}", e.message);
        } else {
            res.exception = JSON.stringify(e)
            // 日志里打印异常信息
            console.error("函数执行失败，异常信息：{}",JSON.stringify(e));
        }
    }

    return JSON.stringify(res);
}

函数调试器
概述

为了方便用户在为执行引擎开发扩展函数，我们提供了一套执行引擎调试方案。因为扩展函数运行环境的特殊性，调试方案实现了以下功能特征，多人调试工具、调试请求穿透到内网、对调试请求的管理、生产与调试的隔离等。

调试机制

J2V8框架可以暴露了单步调试的API，但是并没有一体化的调试能力，目前有第三方插件j2v8-debugger 提供了通过 Stetho连接到Android系统上进行真机调试的能力，这套调试模式适合Android移动开发，并不适合公有云。

YonScript设计了一套基于会话的远程调试机制，可以进行在线的调试而不影响其他人使用系统。这套系统主要由YonScript V8DebugServer, 调试句柄管理器和调试器（基于开源的node Inspector改造）构成。

YonScript V8DebugServer与开源的J2V8-Debugger起的作用相同 —— 连接J2V8引擎，在开启调试模式对外暴露调试端口，不同的是J2V8-Debugger使用的是Steho 连接Chrome DevTools ，而V8DebugServer 通过http对外暴露 Inspector调试协议.协议内容（https://github.com/buggerjs/bugger-v8-client/blob/master/PROTOCOL.md ）

调试句柄管理器是调试工具链的核心，负责分发，鉴定调试句柄，建立调试器和V8DebugServer的关联关系。调试器是基于开源node Inspector改造，改造的核心是修改了会话管理机制，使得原来单机版的Inspector可以支持多人并发使用。

调试机制框架

调试器改造

J2V8框架中调试协议并没有使用nodejs中较为流行的node-inspector协议，而是使用了V8引擎原生的debug-agent协议。因此无法使用浏览器自带的工具进行调试,同时因为云环境本身的限制也导致很难复用现有的调试工具。

协议适配

我们选择了node-inspector调试工具，对该调试工具的调试协议进行修改，使得该工具能够连接J2V8调试框架使用的debug-agent协议。协议匹配后我们就能通过node-inspector向打开调试端口的应用引擎发起基础的调试请求。

内网穿透

扩展函数的应用引擎可能部署在未向外网暴露的环境，无法与外网进行直连。为了应对这种情况，我们暴露一个公用调试器，通过调试器进行反向代理，将调试请求转向对应的应用引擎调试端口

单人改多人

这部分改造主要涉及，调试端口的连接机制、监视变量的隔离、调试session的存储方式等，通过这些改造将每组调试请求隔离开。调试器将静态文件和脚本传输至用户浏览器，然后通过浏览器与调试器后端建立websocket连接保持调试session。

调试控制

考虑到引擎的运行环境，需要对调试求加以控制，防止调试无限制的进行、无意的触发调试、调试影响生产环境等问题的发生。为了解决这些问题引擎方面开发了任务监控器、调试句柄管理器。

任务监控机制基于时间轮算法开发，限制一切任务的执行时间，防止资源被长时间占用。

调试句柄管理器

调试交互

开发者在函数编辑器上点击调试按钮，会生成一个调试句柄，通过调试句柄可以进行会话级别的调试，不会影响其他的请求。

一般开启调试有多种方式，如在MDD开发的界面上按住ALT-Shift-D，会弹出一个输入调试句柄的请求。（这个是在MDD框架中植入了一个调试快捷键）， 输入调试句柄后将句柄写入Cookie。请求从浏览器发送到后端时，会携带了这个Cookie； 在API网关的调试界面说有输入调试句柄的输入框，输入后点击测试 ，请求会携带这个句柄。 YonScript引擎会调用句柄的API判断这个调试句柄是否使用过，是否是要调试当前的函数，如果是，则调用DebuggerServer开启调试端口，并通知浏览器打开调试器。

打包上线
概述

我们利用开发平台开发一个工程之后，会有把函数等相关内容导入到另一个租户下的需求，打包上线就是为了满足客户这一需求，将一个租户下的函数数据，包括函数信息，函数内容，函数注册关系，函数在应用下的树状结构通过打包的方式上传，然后在另一个租户下将之前上传的包下载下来，导入到当前租户下，就完成了函数的迁移，迁移之后的函数，就可以在新租户下正常使用。

核心思想

为了满足打包之后的函数在新租户下能正常使用，我们需要保证数据的id相同，因为我们都是通过id作为关联关系的。所以我们需要需要如下操作

打包数据的表的主键设置为id+租户id的联合主键
在打包时查询出当前租户下的所有数据，做成我们设计好的数据结构
在下载时，将本租户下的数据查询出来，做一个id+租户id为建的hashmap，获取下载包中的数据，如果下载包里的数据，他们的id+本环境租户id在我们做好的hashmap中存在，我们视为更新操作，执行更新函数内容操作，否则视为新增操作，插入数据库中，让函数方面的数据能在新租户下正常使用
