基本概念
背景

区块链云服务平台是用友自主研发的高可靠、高性能、强安全、强隐私保护、多框架的企业级区块链PaaS（ Platform-as-a-Service ）平台服务 -- 区块链即服务BaaS（Blockchain-as-a-Service），帮助客户快速构建更稳定、安全的生产级区块链环境，减少在区块链部署、运维、管理、应用开发等方面的挑战，使客户更专注于核心业务创新，并实现业务快速部署。

名词解释

CA：证书授权中心（Certificate Authority） 负责管理和签发数字证书，作用是检查证书持有者身份的合法性，并签发证书，以防证书被伪造或篡改。

**数字证书：**数字证书 由CA经过信息认证后颁发的证明持有者身份的数字文件。

根证书： CA的公钥证书。

通用证书： 包含身份证书和通讯证书，区块链网络中每个参与服务节点都有一对，其中通讯证书用来保证通信链路安全，身份证书用来验证服务节点权限。

**机构 ：**指具有社会信用代码的实体。

联盟 **：**若干独立的法人企业为了相同的业务目标通过正式约定形成的团体，在平台中表现为多个机构组成的集合。

**网络： **根据不同的行业/领域建立的机构集合，是联盟机构集合的子集。

排序服务 **：**网络必备的服务（技术上指若干Orderer的集群），向业务链发送交易，每个网络至少建立一个排序服务。

**业务链 ：**特定业务的区块链（技术上指Channel）。

工作节点： 执行交易计算、区块存储的服务节点。

智能合约 **：**业务链上可执行的逻辑代码，对区块链的操作必须经过智能合约。

产品亮点

区块链云服务可以帮助客户快速构建更稳定、安全的生产级区块链环境，减少在区块链部署、运维、管理、应用开发等方面的挑战，使客户更专注于核心业务创新，并实现业务快速部署。

区块链云服务核心功能包含如下：

联盟和网络管理：主要包含新建联盟、加入联盟、新建网络、加入网络等功能，联盟与网络是一对多的关系，都是在业务关系层面对区块链使用范围进行的约束，其中联盟约束参与的实体机构，网络在联盟的基础上对垂直行业/领域进行进一步的划分。

业务链管理：主要包含部署服务节点，创建和启动业务链等功能，业务链是根据具体业务建立的区块链，比网络更进一步的限定了参与的服务节点。

智能合约管理：主要包含新建合约、查看合约、版本管理、安装合约、升级合约等功能，对智能合约全生命周期进行管理。

监控审计：主要包含区块链网络监控、资源监控、节点监控、业务链监控、网络拓扑和审计日志等功能，从底层资源、服务节点运行状况、链上数据、区块链网络拓扑、业务管理数据自底向上垂直全方位监控。

区块链的特性与价值
区块链的技术特征

区块链技术是分布式的网络数据管理技术，利用密码学技术和分布式共识协议保证网络传输与访问安全，实现数据多方维护、交叉验证、全网一致、不易篡改。作为新一代信息通信技术的重要演进，由于区块链产生的数据具有不可篡改、透明可追溯等特征，使得区块链技术正在成为解决产业链参与方互相信任的基础设施。

区块链的应用价值

为了解决现实世界不信任的问题，区块链技术通过建立电子信息、加密、确认交易、实时广播、添加区块和网络复制记录等步骤完成工作。基于数字方式的交易行为，区块链技术能潜在地降低交易成本、加快交易速度，同时能提高安全性。随之带来的业务特性可能包括以下内容：

可信任性：区块链技术可以提供天然可信的分布式账本平台，不需要额外第三方中介机构参与；
降低成本：与传统技术相比，区块链技术可能通过自动化合约执行带来更快的交易，同时降低维护成本；
增强安全：区块链技术将有利于安全、可靠的审计管理和账目清算，减少犯罪风险。
区块链应用场景

区块链服务BaaS可应用于各大行业的多种业务场景中，例如：商品溯源、数据资产交易、供应链金融、数字内容版权、公益慈善、信用证、资产证券化、资产托管、能源化工交易、房产交易租赁、数字身份等。但是从技术角度看，区块链主要应用在存证防伪和商品溯源等领域。

存证防伪

由于区块链的不可篡改特性，区块链上的数据天生具有存证特性，因此借助区块链技术实现存证成为区块链应用使用较多的场景。但是由于原始数据的价值比较低，而且冗余性高，为了避免区块链上的数据过大，通常情况下数据在上链之前都会对原始数据进行一次处理，这种情况下，区块链是无法防止数据的“原始失真”和“原始恶意”。但是当数据上链后，由于链上数据的不可篡改性，导致数据造假被发现的几率会大大增加，而造假一旦被监管机构或社会机构发现，常常面临巨大的惩罚，这对企业来说得不偿失，这样避免了企业的恶意造假。

在当下，由于区块链存证能够保障上链数据的安全性、客观性、真实性，目前已经被司法、监管、金融机构所认可，企业可以通过区块链应用获得这些机构的信任，从而挖掘出更多的商业机会。比如数字内容版权已经被司法机构认可，基于区块链的公益慈善也被社会大众所认可，这些区块链应用直接带来了经济价值和社会价值。

商品溯源

溯源，指的是物品或信息在生产、流通及传输的过程中，利用各种采集和留存方式，获得物品或者信息的关键数据，并通过某种方式把数据按照特定的格式和方式进行存储。在传统的溯源场景下，由于数据链条特别长，数据的透明度很低，每个参与方都有可能受利益驱动而篡改数据，而且当交易的一方或多方进行数据造假时，由于不能保证追溯数据的真伪，给司法判定带来很大的问题。

区块链溯源依靠智能合约建立独特的供应链，商品在流通的每一个过程都将被完整地记录在“物流链”上。商品从哪里发出，谁发出的，谁接收的，物流的上一站是哪里、下一站又是哪里，都将有据可查。为了保证交易数据的可信性和完整性，链上的数据应有完整的闭环自证能力。

区块链应用开发指的是基于区块链云服务平台创建区块链网络和业务链，开发智能合约并部署到业务链，开发业务应用并基于开发的智能合约实现数据上链和链上数据查询。

应用开发流程

本节内容介绍如何根据用户的需求开发出符合场景的区块链应用。通常来说，开发区块链由以下五部分组成：

业务分析：帮助用户分析他们的业务痛点，理出各角色在业务中的难点痛点分别是什么，从而确定区块链的使用场景，借助区块链技术实现业务创新，解决客户的根本性难题。
区块链设计：通过区块链的使用场景，设计区块链的网络拓扑结构，链上存储的数据结构以及如何使用链上的数据。
创建区块链网络：基于用友的BAAS服务平台快速创建区块链网络，为部署智能合约提供底层支持。
开发智能合约：根据智能合约API和智能合约编写规范，编写智能合约，并部署到业务链。
应用开发：将区块链SDK集成到应用服务中，通过平台的SDK调用智能合约，实现数据上链与链上数据的查询，然后像开发传统的Web应用一样开发区块链应用。

业务分析

业务分析是收集和处理所有业务数据的过程，并应用统计模型和重复方法将这些数据转化为业务见解的过程。业务分析要揭示出以前未知的见解或未预料的问题，从而创造新的业务价值，也就是说业务分析需要直击用户的痛点，避免避重就轻的解决一些微不足道的问题。

区块链技术是有很强的潜力和通用性的，如果有足够好的创意，它几乎可以在任何场景下发挥巨大价值。但是在使用区块链之前，必须要清楚用户想要创建一个什么样的应用，它会应用在哪个领域？哪些用户会用到它？具体的作用是什么？能够解决哪些现存问题？

本文以某公益机构物资捐赠的存证场景说明如何进行区块链应用的业务分析。

谁要存数据

某公益机构作为出资方，想要把公众物资捐赠的详情存储下来，那么存证的主体自然是公益机构，而不是捐赠方。

为什么存到区块链

公益机构为什么要把数据存储到区块链呢？因为公益机构遇到了业务难题，公众对公益机构产生了信任危机，社会公益难以顺利进行，需要借助区块链技术解决信任危机。

通过对利益相关方深入分析后，其实每个参与方都有他们独特的难题:

捐赠方： 我捐的物资谁拿了？ 救助对象是谁？ 捐的东西有没有收到？

公益方： 捐赠物资的质量难以监控，捐赠物资的计价难以把握，捐赠物资与疫情灾区实际需求难以一致, 捐赠物资的时效难以实现

监管机构：实际捐赠的物资和实际接收的物资是否一致，是否存在物资克扣情形

什么时候把数据存到区块链

作为区块链的设计者需要考虑什么时候把数据存储到区块链，是在每一笔交易发生后，立即将数据上传到区块链，还是定时分批次的将数据上传到区块链？用户（这里指的是所有的参与方）对数据的实时性要求是什么？

将数据实时上传到区块链时，如果捐赠集中在某一时间段，应用系统是否能抗住压力而不崩溃？通过定时任务的方式进行数据上链，能否满足用户的实时性需求?

通过对整个场景的深入分析后，我们发现各方用户最大的需求是数据的真实可靠性，而对数据的实时性要求并不高，而捐赠往往发生在灾情发生后一小段时间内，也就是说捐赠具有短时高并发性，综合考虑，采用定时上链的方式更加经济安全。

怎么存储数据

公益机构为了自证清白，必须能够接受监管机构的实时抽检，最方便的是将数据在监管机构处存储一份数据，但是监管机构是不可能允许公益机构将数据存储在自己的机房内的。为了解决这个问题，公益机构除了将数据存储在自己的机房内之外，还需要将数据存储在公证处认可的地方，然后做公证认证。

存储什么数据

为了消除公众和监管机构的疑虑，需要对每个流程环节的关键数据进行全部上链，让链上的数据形成闭环，从而实现“自证清白”。

发起捐赠阶段： 捐赠人基本信息（姓名/身份证号），捐赠的物资详情（物资名称/单价/数量/来源等信息）

物资流通阶段： 运输人，运输方式，始发地，目的地

物资接收阶段: 接收人，物资完损程度，接收时间

物资分发阶段： 分发者，受益方

存储的数据怎么用

各参与方对数据的要求不一样，需要根据用户的真实需求开发对应的应用。

公益方：了解物资的实时情况，以便进行下一步操作

捐赠方：捐赠物资是否到达受赠方手中，数量质量是否和捐赠信息一致

监管机构：抽检数据和实际物资进行比对

区块链设计

区块链设计是区块链应用开发中的核心内容，也是解决用户问题的关键所在。由于区块链的不可篡改性，一旦数据上链后就不可更改，因此在设计时，一定要充分论证使用场景，避免或减少不合规的数据上链的机会。

区块链设计由区块链网络设计、链上数据结构设计和智能合约设计三部分组成，每个部分的核心内容分别如下：

区块链网络设计：区块链网络的交互模式由联盟链和私链组成，其中联盟链场景是使用最多的模式。在联盟链场景下，需要着重关注每个业务链下的所有参与方的权益，既要保证参与方能够安全访问数据，还要避免非参与方访问数据。

链上数据结构设计：虽然区块链本身并没有对上链的数据进行限制，但是考虑到链上数据的不可篡改性，链上数据会随着时间增长越来越大，为减少存储空间的占用，应该只允许最核心的业务数据进行上链。

智能合约设计：智能合约是线下数据和链上数据的连接通道，在设计时需要考虑安全性、可维护性、鉴权等多个方面因素。

区块链网络设计

区块链云服务平台支持单平台集中部署区块链网络和多平台分布式部署区块链网络两种模式，也就是我们平时所说的私链网络和联盟链网络。用户需要根据实际的使用场景选择对应的区块链应用模式。

联盟链在商品溯源、公益慈善、仓储物流、电子政务、保险理赔等领域得到诸多应用。比如在常见的商品溯源案例中，用户可随时查询商品从生产、质检、物流等每个环节信息，基于对第三方数据核查担保公司的信任及链上数据不可篡改的真实性，让商品更加的真实可信。

链上数据结构设计

为了保证数字文档和交易记录的真实性，以及在整个供应链过程中跟踪产品和商品的来源，链上的数据通常具有完整的闭环自证能力，从而保证交易数据的可信性和完整性。

作为一种难以篡改、历史数据可追溯的分布式账本，区块链系统为不可信的多个参与方提供方便的数据共享服务，支持源数据采用原文结构存储、加密结构存储和哈希摘要存储三种方式上链。

原文结构存储

原文结构存储指的是将多个参与方共享的数据按照数据本身的结构存储方式上链，此种方式能够大大提高协作方的工作效率，但是在全复制数据的基础上，会大大增加共享数据的存储开销。在对等（peer to peer，P2P）网络模式下，网络传输代价会随着报文的增加而增大，此种情况也会影响系统的吞吐量。

加密结构存储

加密结构存储指的是存储在区块链上的数据是以密文形式存储的，此种方式能够大大增强系统的安全性，但是会牺牲一部分性能。从系统的安全性和性能方面综合考虑，通常只会对敏感数据进行加密存储。

常用加密的存储方式有两种：一种是在业务应用侧加解密，另一种是在合约侧加解密。应用侧加密适用于在多方协作开放时保护己方数据时使用，而合约侧加密适用于做一些通用的加解密逻辑处理，比如要整体增强数据在协作方流转的安全性。

哈希摘要存储

哈希摘要存储指的是将业务数据通过某种哈希算法生成一段摘要，业务数据存储在链下（本地服务器或云服务器）,而哈希摘要信息存储在区块链上的存储方式。这种“胖链下”和“瘦链上”的设计，可以在降低存储成本的同时提高系统的吞吐量，是目前区块链应用中使用最多的方案。

智能合约设计

智能合约的出现，使得区块链不仅能够处理简单的转账功能，还能实现复杂的业务逻辑处理。为了保证应用的安全性、可维护性等特征，编写的智能合约要注意以下要点：

重视资源限制

由于智能合约运行在容器内，这意味着单个智能合约所能使用的资源不能超过容器限额，因此在智能合约中不建议编写资源消耗型的应用。如果进行复杂计算或者大量读写数据，会造成Peer侧使用资源过多。更严重的是，如果智能合约存在资源泄露，可能导致容器响应缓慢或崩溃，甚至影响外部系统。

设计合理的智能合约结构

上层应用和智能合约在某种程度上共同完成了业务逻辑。不同的是，智能合约封装了对区块链账本的操作，上层应用通过直接调用智能合约接口来使用，但感知不到底层区块链结构的存在。因此，在某些情况下，部分功能或逻辑可以智能合约层实现，也可以在应用侧实现。从应用架构的角度看，上层应用的逻辑和智能合约逻辑是完全有机的结合，并不存在明显的界限。

在实际应用中，有人把大部分逻辑放进智能合约侧实现，只保留简单、少量的应用接口。这样对上层的应用要求很轻，可以很好的支持现有业务，但当业务逻辑发生变更时，需要对智能合约进行升级。

也有人倾向于只把与账本打交道的核心逻辑封装到智能合约中，在这种情况下，智能合约是一个轻量级接入层服务。此种情形虽然上层应用需要实现更多的业务逻辑，但是底层智能合约服务可以保持长时间的稳定。

这两种设计模式各有利弊，更成熟的做法是将与信任相关的逻辑放在智能合约中实现，容易发生变更的业务逻辑放到应用侧实现。

智能合约开发API

在 Hyperledger Fabric 中，智能合约一般分为系统智能合约和用户智能合约。系统智能合约负责 Fabric 节点自身的处理逻辑，包括系统配置、背书、校验等工作。系统智能合约仅支持 Go 语言，在 Peer 节点启动时会自动完成注册和部署，所以安装，实例化和升级不适用于系统智能合约。

用户智能合约是由应用程序开发人员根据业务场景需求使用 Golang语言编写的操作区块链分布式账本的状态的业务处理逻辑代码，用户智能合约运行在智能合约容器中，通过 Fabric 提供的接口与账本状态进行交互。

本书所描述的智能合约API是默认指的用户智能合约。

func (v *EvidenceCC) set(stub shim.ChaincodeStubInterface, args []string) pb.Response {

var evidence Evidence

fmt.Println("写日志")

}

func (v *EvidenceCC) searchEvidence(stub shim.ChaincodeStubInterface, args []string) pb.Response {
if len(args) != 3 {
return shim.Error("Incorrect number of arguments. Expecting 3")
}
evidenceKey := args[0]
token := args[1]
digest := sha256Hash(token)
grantKey := fmt.Sprintf("%s_%s_%s", GRANT, evidenceKey, token)
grantByte, err := stub.GetState(grantKey)
if err != nil {
return shim.Error(fmt.Sprintf("There is no record of that Grant %s!", grantKey))
}
var grant Grant
err = json.Unmarshal(grantByte, &grant)
if err != nil {
return shim.Error(fmt.Sprint("Failed to Unmarshal grantByte!"))
}
cert, err := byteToCert([]byte(grant.AuthorizedCertificate))
if err != nil {
return shim.Error(err.Error())
}
signature := args[2]
rint, sint, err := getECDSASignatureRS(signature)
if err != nil {
return shim.Error(err.Error())
}
var flag bool
switch cert.PublicKey.(type) {
case *ecdsa.PublicKey:
pub := cert.PublicKey.(*ecdsa.PublicKey)
flag = ecdsa.Verify(pub, digest, rint, sint)
case *rsa.PublicKey:
_ = cert.PublicKey.(*rsa.PublicKey)
return shim.Error(fmt.Sprint("Does not support the rsa!"))
default:
return shim.Error(fmt.Sprint("There is no certificate of this type!"))
}
fmt.Println("验签结果：", flag)
if !flag {
return shim.Error(fmt.Sprint("Failed to Verify signature!"))
} else {
if grant.ReadTimes < 1 {
return shim.Error(fmt.Sprint("The number is not enough searchEvidence!"))
}
endTime := time.Unix(grant.EndTime/1000, 0)
fmt.Printf("授权结束时间 = %+v", endTime)
if endTime.Before(time.Now()) {
return shim.Error(fmt.Sprint("SearchEvidence overtime!"))
}

//所有验证通过，获取存证
evidence, err := stub.GetState(evidenceKey)
if err != nil {
return shim.Error(fmt.Sprintf("There is no record of that Evidence %s!", evidenceKey))
}
fmt.Printf("授权次数-1")
grant.ReadTimes -= 1
grantByte, _ = json.Marshal(grant)
err = stub.PutState(grantKey, grantByte)
if err != nil {
return shim.Error(fmt.Sprintf("Failed to set grant: %s", string(grantByte)))
}
fmt.Println("写日志")
creator, _ := stub.GetCreator()
log := &OperateLog{
ObjectType: LOG,
EvidenceCode: grant.EvidenceCode,
OperateType: "searchEvidence",
Operator: string(creator),
Detail: "取证",
}
err = writeLog(stub, log)
if err != nil {
return shim.Error(fmt.Sprint("Log write failure!"))
}
return shim.Success(evidence)
}
}

func UnmarshalECDSASignature(raw []byte) (*big.Int, *big.Int, error) {
// Unmarshal
sig := new(ECDSASignature)
_, err := asn1.Unmarshal(raw, sig)
if err != nil {
return nil, nil, fmt.Errorf("failed unmashalling signature [%s]", err)
}
// Validate sig
if sig.R == nil {
return nil, nil, errors.New("invalid signature, R must be different from nil")
}
if sig.S == nil {
return nil, nil, errors.New("invalid signature, S must be different from nil")
}
if sig.R.Sign() != 1 {
return nil, nil, errors.New("invalid signature, R must be larger than zero")
}
if sig.S.Sign() != 1 {
return nil, nil, errors.New("invalid signature, S must be larger than zero")
}
return sig.R, sig.S, nil
}

func sha256Hash(text string) []byte {
SHA256Inst := sha256.New()
SHA256Inst.Write([]byte(text))
result := SHA256Inst.Sum(nil)
return result
}

func byteToCert(b []byte) (cert *x509.Certificate, err error) {
certStart := bytes.IndexAny(b, "-----BEGIN")
if certStart == -1 {
return nil, errors.New("No certificate found")
}
certText := b[certStart:]
bl, _ := pem.Decode(certText)
if bl == nil {
return nil, errors.New("Could not decode the PEM structure")
}
cert, err = x509.ParseCertificate(bl.Bytes)
if err != nil {
return nil, errors.New("ParseCertificate failed")
}

return
}

主要API

为了简化区块链的应用开发，平台封装了两个接口，分别实现了数据上链操作和链上数据查询操作，该类位于 com.yonyou.example.service.BlockchainOperateService 文件中。

数据上链接口

/**
* 数据上链接口
* @param operate 智能合约方法
* @param args    智能合约的参数列表
* @return
*/
public Object set(String operate, String... args) {
TransactionProposalRequest proposalRequest = hfClient.newTransactionProposalRequest();
proposalRequest.setChaincodeID(ccId);
proposalRequest.setFcn(operate);
proposalRequest.setArgs(args);
BlockEvent.TransactionEvent event = null;
try {
Collection<ProposalResponse> proposalResponse = currentChannel.sendTransactionProposal(proposalRequest);
event = currentChannel.sendTransaction(proposalResponse).get(30, TimeUnit.SECONDS);
} catch (ProposalException | InvalidArgumentException | InterruptedException | ExecutionException | TimeoutException e) {
e.printStackTrace();
}
Map<String, Object> map = new HashMap<>(3);
map.put("txid", event.getTransactionID());
map.put("valid", event.isValid());
map.put("datahash", HexUtils.toHexString(event.getBlockEvent().getBlock().getHeader().getDataHash().toByteArray()));
return map;
}


链上数据查询接口

/**
* 链上数据查询接口
*
* @param operate 智能合约方法
* @param key     智能合约参数
* @return
* @throws Exception
*/
public String get(String operate, String key) throws Exception {
QueryByChaincodeRequest request = hfClient.newQueryProposalRequest();
request.setChaincodeID(ccId);
request.setFcn(operate);
request.setArgs(key);
ProposalResponse[] responseArray = currentChannel.queryByChaincode(request).toArray(new ProposalResponse[0]);
String response = new String(responseArray[0].getChaincodeActionResponsePayload());
System.out.println("查询结果： " + response);
return response;
}
