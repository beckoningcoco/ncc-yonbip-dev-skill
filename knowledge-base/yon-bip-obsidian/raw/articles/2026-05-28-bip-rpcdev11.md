RPC接口开发与调用（iris）
最后更新时间：2025-05-20
概述
业务场景

项目中如果有多个客开引擎，希望跨引擎进行接口调用，可以发布开放api，也可以通过创建rpc接口进行集成。本文主要介绍rpc接口的发布集成步骤及相关知识。

关键词

Yms Rpc、iris、RemoteCall、远程调用、RPC接口

专业名词术语
RPC

RPC（Remote Procedure Call，远程过程调用）是一种计算机通信协议，允许程序调用另一个地址空间（通常是另一台机器）中的函数或方法，就像调用本地函数一样。它的核心目标是简化分布式系统开发，让开发者无需关注底层网络通信细节。

​客户端存根（Client Stub）​

将本地函数调用转换为网络请求，负责参数的序列化（打包）。

​网络传输

通过 HTTP（比如SpringCloud）、TCP（比如Dubbo） 或自定义协议发送请求到服务端。

​服务端存根（Server Stub）​

接收请求，反序列化参数，调用实际的服务函数。

​响应返回

服务端处理结果后，通过相同路径返回给客户端。

RPC异步开发

目前市场主流的异步解决方案有两种，分别是异步调用-同步处理，异步调用-异步处理，本文讨论的实现方案为异步调用-同步处理，因此，仅需要调用方进行简单的修改，被调方无感。

方案1：IrisCallback

使用IrisCallback回调函数，适用批量调用的场景且在主线程中不关心调用结果，仅需在子线程中回调处理调用结果的场景，如事件中心批量下发事件。

说明：

异步RPC请求发送前的处理，是同步处理逻辑，如负载均衡、熔断准备等；
同步处理过程中的报错，将在同步处理时抛出，不会在异步的回调函数IrisCallback的onFailure捕获到，需要在手动catch异常
闭包中的原本的返回值为null，请勿直接在异步中处理原本的返回值，真实的返回值在IrisCallback.onSuccess中获取
public class Main {
@Resource
private List<IEventService>  eventServices;

@Resource
private EventCallback callback;
public void batchSendEvent() {
// 异步多次调用
for (IEventService eventService : eventServices) {
try {
AsyncStub.asyncExecuteWithCallback(() -> eventService.sendEvent(type, error), callback);
} catch (Throwable e) {
LOGGER.error("异步RPC调用前报错 ", e);
}
}
}
}

@Component
public class EventCallback  implements IrisCallback<Event> {

private EventList list;
public EventCallback(EventList list) {
this.list = list;
}
public EventCallback() {
}

@Override
public void onFailure(Throwable t) {
LOGGER.error("事务操作失败失败：[{}]", t.getMessage());
}
@Override
public void onSuccess(Event event) {
LOGGER.error("事务处理...， [{}]", event.toString());
event.handle();
if (list != null) {
list.add(event);
}
}
}

方案2：IrisFuture

使用IrisFuture，并进行多次调用，适用于调用方法不存在依赖关系，仅各返回结果存在依赖关系的场景。

说明：

异步RPC请求发送前的处理，是同步处理逻辑，如负载均衡、熔断准备等；
同步处理过程中的报错，将在同步处理时抛出，需要在手动catch异常；
异步RPC请求时的异常，将会在IrisFuture.get或IrisFuture.join时，阻塞异步任务获取；
闭包中的原本的返回值为null，请勿直接在异步中处理原本的返回值，真实的返回值通过IrisFuture.get来获取
public class Main {
public MergedEvent batchSendEvent() {   // 异步多次调用
IrisFuture<Event> hrFuture = AsyncStub.asyncExecuteWithFuture(() -> hrService.sendEvent(type, error));
IrisFuture<Event> fiFuture = AsyncStub.asyncExecuteWithFuture(() -> fiService.sendEvent(type, error));

// fiFuture依赖hrFuture
CompletableFuture<MergedEvent> future = hrFuture.thenCombine(fiFuture, Event::merge);
return future.get();  }
}

方案3：IrisFuture+ IrisCallback

使用IrisFuture，并进行多次调用，可支持在IrisFuture中注册回调函数IrisCallback，适用于批量调用的场景且在主线程中关心调用结果，同时可在子线程中回调处理调用结果的场景。

说明：

异步RPC请求发送前的处理，是同步处理逻辑，如负载均衡、熔断准备等；
同步处理过程中的报错，将在同步处理时抛出，需要在手动catch异常；
异步RPC请求时的异常，将会在IrisFuture.get或IrisFuture.join时，阻塞异步任务获取，同时，此处也在IrisFuture中注册了IrisCallback，因此，异步RPC请求时的异常也将会在IrisCallback的onFailure获取；
闭包中的原本的返回值为null，请勿直接在异步中处理原本的返回值，真实的返回值通过IrisFuture.get来获取或在IrisCallback的onSuccess处获取。
public class Main {
@Resource
private List  eventServices;
public EventList batchSendEvent() {
EventList result = new EventList();
// 异步多次调用
List<IrisFuture<Event>> futures = new ArrayList<>();
for (IEventService eventService : eventServices) {
IrisFuture<Event> future = AsyncStub.asyncExecuteWithFuture(() -> eventService.sendEvent(type, error));
future.addCallback(new EventCallback(result));
futures.add(future);
}

// 一起阻塞结果
CompletableFuture<Void> sumFuture = CompletableFuture.allOf(futures.toArray(new IrisFuture[0]));
sumFuture.join();
// 主线程返回各子线程的处理结果
return result;     }
}
public class EventCallback  implements IrisCallback<Event> {

private EventList list;
public EventCallback(EventList list) {
this.list = list;
}
public EventCallback() {
}

@Override
public void onFailure(Throwable t) {
LOGGER.error("事务操作失败失败：[{}]", t.getMessage());
}
@Override
public void onSuccess(Event event) {
LOGGER.error("事务处理...， [{}]", event.toString());
event.handle();
if (list != null) {
list.add(event);
}
}
}

RPC扩展

IRIS提供如下四个扩展点：

IBeforeInvoke 客户端rpc调用前（比如监控、上下文传递）
IAfterInvoke 客户端rpc调用后
IBeforeExecute 服务端执行前
IAfterExecute 服务端执行后

四个接口用法一样，效果不同，下面用IBeforeInvoke举例：

客户端扩展实现类
public class XBizBeforeInvokePlugin implements IBeforeInvoke {
@Override
public String getPluginName() {
return this.getClass().getName();
}

@Override
public int order() {
return PluginConstants.PLUGIN_START_INDEX + 6;
}

@Override
public void run(InvokeRequest req, InvokeResponse resp, InvokeChain chain) {
RPCRequest rpcRequest = req.getRequest(RPCRequest.class);
RemoteInvocation remoteInvocation = rpcRequest.getRemoteInvocation();

//这里填写需要添加自定义上下文及对应值
remoteInvocation.addAttribute("ctxCode", "自定义上下文Code");
remoteInvocation.addAttribute("bizCode", "自定义其他业务上下文");
//注意: 责任链模式，一定要执行下面这一行
chain.run(req, resp, chain);
}
}

服务端限流

在yms控制台的yms-rpc模块配置该服务的限流规则

全局限流

全局限流是对所有服务访问该服务的限制。所有服务每秒访问该服务的次数超过该全局限流配置的qps就会被限流。不配置该配置或者全局限流的生效状态是关闭的时候，全局限流不生效。

标签限流

标签限流是根据自定义的标签进行限流

规则名	标签限流的名称，可以自己定义
标签名	目前支持 tenantid(租户限流)、appId（服务限流）以及其他上下文的数据
逻辑关系	包含和不包含
值	标签对应的值，可以是多个，中间用英文逗号分隔。（也可以是*号表示匹配所有的值）
qps	每秒请求数量
并发数	并发量
生效状态	默认是关闭，开启后该配置才会生效

例如：租户限流
