容器云技术
最后更新时间：2024-02-23
功能介绍
概述

在YonBIP云原生体系中，容器云是至关重要的一环，是产品服务化的底层支撑。在容器云的基础之上，才有devops的持续交付体系，才有微服务应用的治理平台，才有智能AI机器学习的深度整合，才有基于云原生的监控报警能力。

当项目上选择容器云部署方式来交付产品的时候，实施人员或者客开员工经常会因为云原生体系多种多样的组件及配置而感到无从入手，而YKS将云原生组件集成在一起，提供一体化的部署模式，极大的减少了用户使用门槛和配置复杂度，但是因为项目环境和客户要求的不同，仍需要在使用的过程中灵活掌握各种配置和组件接入方式。

对于运行在YKS容器云底座之上的应用，不当操作可能会导致业务故障。因此在使用YonBIP的云原生产品之前，需要理解容器云技术的相关能力，才能更好的维护YonBIP产品的持续运行。

本文的核心在梳理容器云部署及使用过程中的技术方案，并结合实际项目场景，将YKS容器云技术剖析展示出来，让用户获得一致的容器基础设施能力，并加速业务创新。

名词解释

YKS: Yonyou Kubernetes Service的缩写，作为YonBIP产品的容器云底座，打造适配业务产品各服务的一体化云支撑，实现产品的容器化部署和提供高可用运维管理方案。

Kubernete：Kubernetes是一个轻便的和可扩展的开源平台，用于管理容器化应用和服务。通过Kubernetes能够进行应用的自动化部署和扩缩容。在Kubernetes中，会将组成应用的容器组合成一个逻辑单元以更易管理和发现。

Ingress: 在Kubernetes中，服务和Pod的IP地址仅可以在集群网络内部使用，对于集群外的应用是不可见的。为了使外部的应用能够访问集群内的服务，可使用Ingress实现,本质是通过http代理服务器将外部的http请求转发到集群内部的后端服务。

云中立：基于YKS多云组件，提供一致的跨云使用体验，以及面向应用的端到端自动化部署交付。

YMS：Yonyou Middleware Service的缩写，为YonBIP产品提供统一的中间件、数据源及配置项管理，实现产品的规范化部署和动态配置。

多数据中心：将租户按照一定规模划分到特定的、独立的软件、硬件计算资源中，提供独立的域名入口；需要容器云、DevOps等底层技术平台做相应的改造支持，也需要领域产品区分出社会级应用及领域应用。

容器云：基于Docker及Kubernetes技术构建的PaaS云平台，为上层DevOps提供底层支撑能力。

**DevOps：**一套自动化微服务管理、发布平台，为研发、运维提供一体化管理平台，提高研发效率。

**社会级应用：**跨租户应用服务，如用户注册、登录、认证系统，IM即时通讯系统，跨租户维度管理系统等。

**领域应用：**对社会级应用依赖极少，如只依赖社会级应用中的ID信息，应用自身的数据可以存储在该领域的数据库及存储中。

**北京核心数据中心：**部署技术平台服务，社会级应用服务，及全套YonBIP产品领域微服务，作为发布应用服务的基地，应用服务先在此数据中心验证，进入到次数据中心的灰度、生产环境后的应用服务以镜像方式推送到其他数据中心，达到全部数据中心发布的目的。

**资源池：**不同部门，不同租户提供在不同数据中心和所属环境的一组专属主机。

网络配置

网络组件采用calico, 默认网络模式采用 ipip模式，kube-proxy 代理模式默认采用 iptables，docker 启动参数 iptables 设置为true。

配置 DNS

YKS产品会要求安装 YKS-DNS，流水线发布的应用 dnsPolicy 默认采用 Default，会读取主机resolv.conf配置，即会优先走 YKS-DNS服务而不是CoreDNS。

所以不建议使用service短名称来访问其他应用，但是可以使用长的service名称来访问，例如nfs-server.yks.svc.cluster.local。

云原生存储方案
云原生存储介绍

存储的管理是一个与计算实例的管理完全不同的问题。PersistentVolume 子系统为用户和管理员提供了一组 API，将存储如何供应的细节从其如何被使用中抽象出来。 为了实现这点，我们引入了两个新的 API 资源：PersistentVolume 和 PersistentVolumeClaim。

持久卷（PersistentVolume，PV）是集群中的一块存储，可以由管理员事先供应，或者使用存储类（Storage Class）来动态供应。 持久卷是集群资源，就像节点也是集群资源一样。PV 持久卷和普通的 Volume 一样，也是使用卷插件来实现的，只是它们拥有独立于任何使用 PV 的 Pod 的生命周期。 此 API 对象中记述了存储的实现细节，无论其背后是 NFS、iSCSI 还是特定于云平台的存储系统。

持久卷声明（PersistentVolumeClaim，PVC）表达的是用户对存储的请求。概念上与 Pod 类似。 Pod 会耗用节点资源，而 PVC 申领会耗用 PV 资源。Pod 可以请求特定数量的资源（CPU 和内存）；同样 PVC 申领也可以请求特定的大小和访问模式 （例如，可以要求 PV 卷能够以 ReadWriteOnce、ReadOnlyMany 或 ReadWriteMany 模式之一来挂载）。

尽管 PersistentVolumeClaim 允许用户消耗抽象的存储资源，常见的情况是针对不同的问题用户需要的是具有不同属性（如，性能）的 PersistentVolume 卷。 集群管理员需要能够提供不同性质的 PersistentVolume，并且这些 PV 卷之间的差别不仅限于卷大小和访问模式，同时又不能将卷是如何实现的这些细节暴露给用户。 为了满足这类需求，就有了存储类（StorageClass） 资源。

技术中台 容器服务 -> 存储管理 节点，提供云原生存储功能。用户可以在这里创建 存储卷PV 和 存储卷声明PVC，供流水线使用。

资源节点接入
应用场景

因k8s应用Pod调度是随机的，会出现不同部门，不同环境，不同租户共同使用所有主机，不方便租户，用户，部门管理自有应用。这样就衍生出资源池概念，资源池为一组k8s资源主机，有该资源池权限的用户才能在该资源池部署应用。使不同部门和不通环境主机进行隔离，便于用户进行管理（添加，删除资源主机）。

资源池提供三种资源类型：YKS资源池，监控资源池，外部集群资源池。

YKS资源池： 默认k8s集群资源池，支持部署应用。

监控资源池： 只提供主机的监控信息。

外部集群资源池： 除YKS集群，添加的阿里，腾讯等或者自建k8s集群节点的资源池。

资源池不仅提供主机接入，还提供资源池的资源量，应用个数，资源使用率等信息展示。

YMS 通用接入方案
功能介绍

YMS通用接入为非java开发程序（golang，python，PHP等）提供从YMS获取配置新方式。

功能二进制：yms_config_downlaod。

非Java程序容器化在适配YMS时，没有统一SDK提供获取配置接口。yms_config_downlaod则以设置环境变量的形式提供了和java程序SDK动态获取配置一致效果，支持了用户配置快速修改，快速应用。

设计逻辑

通过yms_config_downlaod 程序启动命令 ， 这种方式通过yms_config_downlaod 先根据参数获取应用对应在YMS中的配置，并将获取到的配置写到yms_config_downlaod进程的环境变量中（因为linux环境无法设置key带 . 的环境变量，golang进程能够设置这样的环境变量）。yms_config_downlaod再调用应用的启动命令，这时该应用启动就能获取golang设置在进程环境中的配置。

如果直接执行启动命令，则需要在流水线的环境变量中，将所有依赖的环境变量预置。这样就无法快速修改应用配置。

参数解读
参数名称	描述	是否必须
access.key	应用使用的AK	是
access.secret	应用使用的AS	是
disconf.conf_server_host	YMS控制台地址	是
mw_profiles_active	环境env	是
app_code	应用在YMS中的产品code	是
key	配置名称（如果为json，则获取应用所有配置（json格式））	否（只有单独以二进制形式执行获取配置才启用）

单独以二进制执行，设置key对应的配置名称，则能获取对应key的值。

典型现场问题分析
离线源缺包
现场现象

nc命令不可用，导致安装过程停止。

问题分析

产品开发环境是基于centos7.5 1804. 现场的系统环境并不由我们提供，有差异。

处理方法

有网环境：

切换yum源，采用阿里源或网易源，手动安装nc命令之后继续安装。

无网环境：

先去有网环境下载缺失的nc rpm包， 拷贝至服务器上，手动安装后继续安装。

网络模式要求
现场现象

YKS安装完后，nfs-client-provisioner服务未正常启动。

问题分析

nfs-client-provisioner服务依赖 nfs-server服务，当连不上nfs-server服务时会导致启动失败。

分析步骤：

确认nfs-server是否运行正常
确认nfs-server域名 nfs-server.yks.svc.cluster.local 是否有域名解析记录
在 nfs-client-provisioner宿主机 ping nfs-server.yks.svc.cluster.local 看是否通
在 nfs-client-provisioner宿主机任一pod里 ping nfs-server.yks.svc.cluster.local 看是否通
pod跨主机不通的话，基本确认网络出问题了
处理方法

Pod跨主机网络不通的问题解决：

查看主机网络路由是否有确认。
ipv4转发是否开启。
iptables forward规则是否开启。
主机虚拟网卡是否正常，当启用ipip模式时，会有turn0网卡，vxlan模式会有vxlan.calico网卡。
当都正常时网络还不通，可以确认当前系统网络环境不支持此模式，可以尝试切换calico网络试试。
使用yks-console服务的运维工具网络切换工具，尝试切换到其他模式。

组件弱密码问题

项目实施中会遇到甲方进行安全扫描，针对某些组件出厂弱密码进行了强化，出厂即带复杂密码，允许在安装过程中设定复杂密码。

信创多数据库适配

在信创项目的推进过程中，会遇到适配国产数据库的情况。例如：

神通 Oscar 数据库。
达梦数据库。

JAVA 应用使用了 YMS 体系，支持多数据库的适配。其他语言如：Golang，Python，php等，由于不能借鉴 YMS 数据库能力，加上国产数据库的驱动程序支持匮乏，使用场景少等原因，我们主要采用 ShardingSphere-proxy 代理方式实现。

第三方容器云适配场景
适配需求：

甲方提供阿里云专属化环境，用来部署YonBIP产品，已提供容器云平台ACK，云解析服务及镜像仓库服务。

需求分析：

典型的第三方容器云接入场景。低层k8s集群需要使用甲方提供的阿里云容器云，dns服务只能使用云解析服务，镜像仓库也不能使用自建的Harbor。

适配方案：

目前已有成熟适配方案，适配方案如下：

DNS适配

自建DNS时会预制相关域名解析记录，当用外部DNS时需要手动注册域名解析记录。

操作步骤：

获取云解析服务DNS地址。
通过安装器安装YKS-DNS服务，采用外部依赖服务安装方式接入DNS。
通过云解析服务web界面，手动录入 预制的域名解析信息。
具体域名信息请参看 3.1.2章节。
镜像服务适配

安装Harbor时会初始化带有镜像，当用外部镜像仓库时需要手动创建项目及上传镜像。

操作步骤如下：

获取镜像仓库信息、地址、账号密码等。
通过安装器界面安装YKS-Harbor镜像服务，采用外部依赖服务安装方式接入镜像仓库。
通过镜像仓库服务web界面，手动创建 所需的项目（或工程），然后将所需的镜像上传至镜像仓库。
具体需要创建的项目及上传的镜像清单参看章节 3.1.3。
YKS适配

因为全部用的外部的K8S集群、DNS服务及镜像仓库服务，所以对于K8S集群主机不需要做任何修改。

在外接DNS、镜像仓库服务后就可以外接YKS了。

操作步骤如下：

确认安装器是否可以访问 k8s集群，不能访问的话需要开通网络。
下载待接入集群的Kubeconfig文件。
提前规划下是否安装ingress及Monitor，建议均安装。
通过安装器录入要装Monitor组件的主机。
通过安装器界面安装YKS-Kubernetes，采用外部依赖服务安装方式安装。
外接k8s类型选择 “阿里云ACK”。
认证方式选择 Kubeconfig文件方式。
上传Kubeconfig文件，认证脚本不需要上传。
输入 k8s集群 Apiserver地址。
选择安装 ingress组件，录入待安装的节点信息。
选择安装 Monitor组件，勾选主机。
点击右上角安装，然后开始安装输出安装日志。
相关操作界面可以参考 章节 3.1.4。
