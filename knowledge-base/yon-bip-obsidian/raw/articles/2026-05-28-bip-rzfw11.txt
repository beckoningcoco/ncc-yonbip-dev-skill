日志服务
最后更新时间：2024-02-23
基本概念
背景

系统中有多种日志需要记录：用户上机日志、操作日志、业务日志。上机日志就是用户登录的记录包括登录时间，使用设备类型等；操作日志需要记录用户点击的菜单或者按钮等操作；业务日志是要单独记录用户比较重要的业务操作，如订单提交，审批等等涉及到业务对象状态改变或者业务交易等动作。 日志服务的要求是需要对业务系统尽量没有侵入，并且不能影响业务系统的性能，日志需要统一存储，方便对日志进行后续的统计分析。

名词解释

登录日志：就是用户登录的记录包括登录时间，使用设备类型等。 操作日志：需要记录用户点击的菜单或者按钮等操作。 业务日志：是要单独记录用户比较重要的业务操作，如订单提交，审批等等涉及到业务对象状态改变或者业务交易等动作。 操作日志统计报表：统计用户对某个菜单或按钮的操作次数。

产品亮点
技术架构
总体架构设计

日志服务支持记录登录日志，操作日志，业务日志以及对应的日志查询和分析。

支持记录登录日志
支持记录操作日志到按钮级
mdd框架集成业务日志，支持关键属性过滤标签，业务数据变更历史查询、快照对比
支持查看操作日志统计报表
提供各种日志的查询
业务日志线程池参数配置

业务日志接入mdd框架采用异步线程池的方式，需要配置线程池的参数，默认为线程数为cpu核心数，默认最大队列长度为10000，默认拒绝策略为丢弃并记录error日志，为了适应各个领域的不同需要，提供了以下配置项来配置线程池的参数：

A44.async.core.poolSize //线程池线程数 默认为cpu核心数
A44.async.max.poolSize //线程池最大线程数 默认为cpu核心数
A44.async.queueCapacity //线程池队列长度 默认10000
A44.async.useMainThread //线程池拒绝策略是否使用调用线程 默认false

注意：

如果压测时发现业务日志线程占用了过多资源导致压测失败，可暂时关闭业务日志的功能，关闭方法：在mdd框架规则表billrulegeister中找到SaveBusinessLogRule，删除或其他关闭方式。

单据批量修改时业务日志配置

前端框架对于修改操作只会收集修改部分的数据，而业务日志需要记录全部数据（业务变更、数据比对等），所以日志对于修改的批量业务数据会进行循环detail查单据其他信息，在线上环境就出现过单据批量更新数据量大时出现服务慢请求情况。

针对这种问题，可通过配置来控制是否需要detail查询

A44.data.mode=fill //fill:填充detail ；incr:不需要detail，默认fill

同时提供了拓展接口com.yonyou.ucf.mdd.ext.pub.rule.itf.IBusinessLogCustomExt，传递业务日志的拓展接口，当调用框架提供的BillBiz.detail(billDataDto) 与BillController中/detail返回结果不同的情况下使用，需要档案开发实现并注册bean到springIOC容器中

/**
* @return detail全量数据
*/
Map getCustomBusinessLogDataForMdf(BillContext billContext, BizObject bill);

功能介绍
功能概述

登录日志记录：记录用户登入登出时间，ip，设备等信息

操作日志记录：记录用户点击菜单或者按钮的操作相关信息

业务日志记录：记录用户重要的业务操作

操作日志统计报表：记录用户对某个菜单或按钮的操作次数

数据查询：能够对三大日志进行多条件检索

数据导出：支持三大日志进行全量导出和选择导出

业务变更记录：业务日志支持查看变更情况

支持三员：三大日志支持根据角色权限查看对应内容

登录日志

登录日志可以记录用户的登录相关信息并且支持多条件查询和数据导出.

操作日志

操作日志记录了用户点击具体按钮节点的所属应用服务和用户编码名称ip等信息

能够根据应用过滤查询用户对某个节点的操作情况，支持数据导出。

业务日志

业务日志记录了用户重要业务操作的数据信息，支持多条件查询和数据导出。

光标移到具体信息行可以进行历史数据对比，变更内容进行高亮显示。

操作日志统计报表

操作日志统计报表，对用户节点操作次数进行聚合统计，支持条件查询。

操作日志
MDD场景

mdd框架由前端框架统一采集操作日志，每个按钮的点击都会记录操作日志，如果个别按钮没有记录，需要确认bill_toolbaritem表对应的按钮的text_resid存在。

非MDD场景

不是基于MDD框架开发的节点，没有框架支持，需要自己在代码里埋点调用操作日志的接口。

使用工作台API

前端调用工作台公共API接入

引用jdiwork.js

jDiwork.recordLog(
	code, //服务编码
	{
		operCode: operCode, //操作编码
		button: button //操作按钮
	},
	()=>{
		//callback
	}
);


如果不引用jdiwork.js，但是在diwork框架内。则按照如下调用：

window.top.postMessage({
	messType:'JDIWORK',
	callbackId: 'recordLog',
	detail:{
		serviceCode: serviceCode,
		data:{
			operCode,
			button
		}
	}
},'*')

不使用工作台

前端直接调操作日志的保存日志API接入，或者后端埋点记录操作日志。

业务日志
基于MDD场景

基于MDD的规则框架的服务，默认会配置记录业务日志的规则：com.yonyou.ucf.mdd.ext.pub.rule.SaveBusinessLogRule

类名：com.yonyou.ucf.mdd.ext.pub.rule.SaveBusinessLogRule

包名：mdd-framework-support

元数据打标签：

标签名：“支持业务日志”

编码：在元数据xml中的需要记录为编码的字段attribute中添加 isCode="true"（对应流程下的单据号标签）如果不打标签，在mdd框架中默认取code字段

名称：在元数据xml中的需要记录为名称的字段attribute中添加 isName="true" (对应日志下的业务日志名称标签)如果不打标签，在mdd框架中默认取name字段

注意事项
busiObjCode和busiObjName两个字段至少要传一个，否则无法记录，如果获取不到，请参照上述元数据打标签的方法。
如果没有记录ip请检查线程变量ThreadLocal中有没有传递clientIp参数，rpc调用接口传递线程变量需要iris-iuap-support包的支持，mdd304框架已在com.yonyou.ucf.mdd.ext.interceptor.LoginInterceptor中实现。

链路配置

前提：使用MDD后端框架并在规则链配置了SaveBusinessLogRule

业务日志支持使用方自主配置数据的传输链路，有三种模式:

server：日志服务端
oss：阿里云文件系统
kafka：开发中，需要自主引入kafka相关maven依赖

配置方式：

可配置在系统变量，也可配置在yms控制台

示例：business.log.dataTransfer=oss （默认值为server）

非MDD场景

非基于MDD框架的应用，需要在后端代码里埋点调用业务日志的rpc接口，

Maven依赖

<dependency>
	<groupId>com.yonyou.iuap.log</groupId>
	<artifactId>iuap-log-sdk</artifactId>
</dependency>

元数据方式

如果在统一元数据服务中对业务实体定义了元数据，那么请给该业务实体打如下标签：ID，CODE，NAME。打了标签，业务日志会自动根据元数据解析获取主键，编码，名称字段，（编码、名称要求至少标记一个）

自定义方式

如果没有元数据或没有打标签，那么需要在接口参数中指定ID，CODE，NAME对应的字段名称（编码、名称要求至少指定一个），记录日志时会在新实体中获取指定字段的value，另外如果不定义元数据，在显示详情的时候不能翻译成对应语种，展示newObject中的原始数据。
