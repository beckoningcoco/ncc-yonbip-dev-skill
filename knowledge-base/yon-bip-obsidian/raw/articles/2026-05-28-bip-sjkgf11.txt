iuap-数据库规范红皮书
最后更新时间：2025-06-18
本文摘要

YonBIP数据库规范红皮书主要是针对开发过程中关于数据库对象命名规则、数据库通用设计原则、数据库操作规范、SQL书写风格、性能优化及多数据库适配中等各方面的规范进行阐述，以期使各服务能规范化、一致化，使YonBIP各组成部分在技术上，规范一致，更方便进行技术栈的统一管理，减少数据库编码中不必要的错误，提高软件质量与开发效率。帮助开发养成良好的数据库编程习惯。

本文主要介绍了开发过程中几个重要方面的规范。可概括为以下核心要点：

数据库设计规范；
对象命名规则
通用设计原则
数据库操作规范
数据库SQL编程规范；
SQL书写风格
性能优化建议
多数据库适配规范，包括函数、语法、关键字等

开发者通过对本文的学习，可了解数据库开发过程中需要遵守的各种规范，借此提高SQL质量，更容易的融合到YonBIP整体产品中，在统一规范的基础上才能更方便的统一管理和使用公共的机制和技术组件。

概述

本规范主要针对开发做出规范，使公司各领域产品的数据库编程具有规范性、统一性、可读性，减少不必要的错误，提高软件质量与开发效率。帮助开发养成良好的数据库编程习惯。

适用范围

本规范适用于用友云产品开发。

数据库规范说明
对象命名规范

命名的总体原则：

【强制】通过对象名（包括库名、表名、字段名、索引名），能快速了解对象所属模块、业务含义或主要实现功能
【强制】schema命名平台库为 iuap_产品_业务含义库名，领域库为 yonbip_领域_业务含义库名，本地化库为 lc_产品_业务含义库名
【强制】对象名应简洁明了，表名总长度不能超过63个字符，建议业务表长度不超过40个字符，为其他业务扩展预留一定长度(例如特征表会添加后缀_1/_2)
【强制】对象名应保证一定范围内的唯一性，防止对象重名。例如索引：Oracle的是schema内不重复；MySQL为表内不重复。建议对象名全局唯一
【强制】对象名禁止使用数据库的关键字和保留字，详见【3.6.4多数据库适配-关键字】
【强制】业务表名前需要增加领域标识前缀或产品或模块标识前缀
【强制】临时库、表名必须以tmp_为前缀，如果是按照日期生成的，以tmp_日期_为前缀，tmp_{YYYYMMDD}_{table_name}，tmp_20240301_xxx
【强制】备份库、表必须以bak_为前缀，如果是按照日期生成的，以bak_日期_为前缀，bak_{YYYYMMDD}_{table_name}，bak_20240301_xxx
【强制】待删除表必须以del_为前缀，如果是按照日期生成的，以del_日期_为前缀，del_{YYYYMMDD}_{table_name}，del_20240301_xxx，DBA自动删除半年前待删除表
【强制】命名只能使用英文字母，数字和下划线，字母全部小写并且作为对象命名开头
【强制】命名应使用富有意义英文词汇或简写，不要使用n1、n2或拼音简写之类难以阅读的命名规则
【强制】新建的业务表租户id的命名统一为ytenant_id，类型为varchar(36)，属性NOT NULL，建议不设置默认值（如果程序未对ytenant_id赋值，程序直接报错）。对历史业务表增加ytenant_id字段，为了兼容专属化脚本及多数据库适配，需要携带默认值，默认值禁止是空字符串。表中必须带有索引i_ytenant_id或ytenant_id开头的前缀索引。主表带租户id字段时，子表定义也需要包含租户id字段。禁止业务表缺失ytenant_id字段，按BIP规范针对0租户预置数据 ytenant_id 为字符串0，租户数据ytenant_id为友互通租户ID，其他非0租户非租户数据程序统一设置为字符串 'N'。禁止ytenant_id字段值不正确
表
固定表

【建议】固定表表名一般为 "产品或模块标识前缀_业务表英文含义"。

例如 org_factory_define 表，org为模块名，factory_define为工厂组织自定义项表。

主子表命名规则：如果实体是由一对多的两个表共同组成的，父表（表头）一般命名为“前缀标识_业务表英文含义_h”，子表（表体）一般命名为“前缀标识_业务表英文含义_b”，孙表（表体）一般命名为“前缀标识_业务表英文含义_g”。平行表一般命名为“前缀标识_业务表英文含义_ext”。

生产环境如果表需要删除，请先rename原表为del_20240301_原表名，DBA自动删除半年前的del_{YYYYMMDD}_{table_name}表。

临时表

【强制】数据库真正临时表一律使用“tmp_”开头，其余部分参考业务表命名。

例如：tmp_md_xxxx

字段

【强制】短英文单词或者长英文单词的缩写。字段顺序常用字段放在前面，系统字段放后面。

具体命名可以参考如下表格：

主键

【强制】主键统一以“id”命名。

例如： md_column表的主键名为id。

外键

【强制】不允许在数据库中创建外键，引用完整性由程序控制。

索引

【强制】索引以“i_”开头，唯一索引以“i_u_”，后面接表名/列名。

例如：表t1(c1 int，c2 varchar(20)，c3 varchar(20)，…)

t1(c1，c2)的索引命名为i_t1_c1_c2或者i_c1_c2

t1(c1，c3)的唯一索引命名为i_u_t1_c1_c3或者i_u_c1_c3

表名加索引名总长度不超过63个字符。

视图

【强制】在公有云禁止使用视图。

说明：视图中表关联查询的成本很高，容易产生性能问题，因此不推荐使用。

触发器

【强制】在公有云禁止使用触发器。

说明：不同种类数据库迁移时，触发器移植的成本很高，因此禁止使用。

自定义函数

【强制】在公有云禁止使用自定义函数。

说明：不同种类数据库迁移时，自定义函数移植的成本很高，因此禁止使用。

存储过程

【强制】在公有云业务上禁止使用存储过程。

说明：不同种类数据库迁移时，存储过程移植的成本很高，因此禁止使用。

注释

【强制】表与列上必须添加注释，注释必须言简意赅。

注释只能使用中文、英文、数字、:(冒号)、-(中划线) 、=(等号)、_(下划线)、()(括号)。

表注释举例：org:业务单元

列注释举例：消费模式(0:默认1:推送模式2:拉取模式)

平台业务模块在领域库内建表，注释必须为 iuap:XX模块:表用途。以便清晰知道schema中表是谁创建的

通用设计原则
数据类型
MySQL可用数据类型

【强制】基于多数据库适配考虑，MySQL只可使用以下数据类型，JSON类型目前不建议使用。

类型	范围	MySQL
可变字符型	长度不固定（最大1000）	varchar
16位整数	-32768 ~ 32767	smallint
32位整数	-2147483648 ~2 147483647	int
64位整数	-9223372036854775808 ~ 9223372036854775807	bigint
精确数值型	pppppppppppppp.ss	decimal(p,s)
日期型	yyyy-mm-dd	varchar(10)
日期时间型	YYYY-MM-DD HH:MM:SS	datetime
布尔型	数字0/1操作	smallint
ID/PK	建议有序	varchar(22)/bigint
长文本	4G	longtext
二进制	4G	longblob
主键类型

【强制】主键类型规范优先统一为varchar(22)，mdd依然可以使用bigint，具体生成使用平台增强的雪花算法OID。新旧应用的主键均需要统一使用平台的ID算法来生成，平台一套算法生成Long和String。

为了保证跨数据中心主键不重复，不允许领域使用数据库自增主键和其他无法实现多数据中心ID唯一的算法。

禁止业务表主键使用数据库自增。

业务数据类型

领域使用设计器添加以下业务类型时会默认对应到指定的数据库类型上，同时也支持领域在默认的数据库类型上进行修改。

公共字段类型
字段名	字段类型	含义
pubts	datetime	时间戳，默认值使用CURRENT_TIMESTAMP
ytenant_id	varchar(36)	租户ID，NOT NULL ，不设置默认值
dr	smallint	逻辑删除标志，0代表未删除，1代表已删除
creator	varchar(50)	创建人
create_time	datetime	创建时间
modifier	varchar(50)	修改人
modify_time	datetime	修改时间
主要数据类型的设计规则

【强制】领域引用平台档案的数据类型需要按平台档案的数据类型来统一，领域间档案引用也基于这个规则。禁止领域对象引用平台档案数据类型出现不一致。

数据类型的总体设计原则是“够用就好”。设计阶段应该对字段的长度和类型精打细算，以满足需求的最小长度为最终字段长度。

大字段的查询和排序效率非常差，滥用大字段对数据库整体性能会带来非常严重的影响。大字段上也禁止进行distinct、group by、等值判断及其他函数操作。

不要将Null与空串""视为相同。在不同的数据库中对这两者的理解是不相同的。Oracle中is null无法使用索引，所有经常用来做为查询条件的字段建议都不允许使用NULL值。

表
业务表
【强制】禁止在业务表查询租户数据的SQL中过滤条件里不带租户ID
【强制】每个表必须有主键(primary key)，不推荐复合主键，禁止使用联合主键的表在关联查询的条件中只使用部分键
复合主键必须遵守以下规则：越能成为唯一标识的字段(同值重复少)就越放在前面
对于已经存在的表，如果要增加字段，建议携带具体业务含义的default值
【强制】关联字段，例如ID、CODE等数据类型一定要一致，防止查询过程中发生类型转换
【强制】确定为非null的字段要在字段中设置其为not null
【强制】所有表都需要添加注释
临时表

临时表分为数据库临时表及业务临时表（数据库层面是实体表，存储临时业务数据）。

数据库临时表在数据库连接存在期间有效，一旦数据库连接中断后，临时表的定义和数据都将被销毁。

在复杂的数据库存储过程计算中，业务临时表用来存储计算的中间结果。

【强制】临时表必须有定时清理机制，避免数据库存在过多废弃的临时表。

其他规范

1.【强制】MySQL尽可能不使用LONGTEXT、LONGBLOB类型，禁止对数据库大字段（LONGTEXT、LONGBLOB等）进行频繁读写和批量操作，禁止使用大字段存储日志、异常堆栈、文件等大文本

2.【强制】文件图片等大数据使用公共文件服务器存储，数据库中存储文件服务器文件id

3.禁止在数据库中存储明文密码，把密码加密后存储

4.禁止单条SQL语句同时更新多个表

5.引用类型的字段建议增加索引

6.查询字段最好有默认值，null在Oracle中无法使用索引

列默认值

新建表建议所有列都添加默认值。默认值不能为空字符串，也不能是~，最好是具体业务含义的字符串或者数字。

空字符串在Oracle会变成NULL，~ 这个字符被翻译器用于特殊定义，为了防止歧义，业务禁止等值查询~。

索引

1.【强制】表所有索引中，必须存在一个ytenant_id开头的索引。其余组合索引字段按使用频率及区分度排序。区分度的公式是count(distinct col)/count(*)，表示字段不重复的比例，比例越大我们扫描的记录数越少，唯一键的区分度是1，而一些状态、性别字段可能在大数据面前区分度接近0

2.【强制】联合索引以i_开头，最多不超过7个字段。唯一索引以i_u_开头，最多不能超过5个字段。

3.对于操作较频繁的表，原则上索引数目不要超过10个

4.【强制】不要对相同字段重复创建索引

5.【强制】对于update和delete语句中用于过滤的字段要考虑创建合理的索引，否则容易造成大范围记录锁定

6.【强制】对于数据值变化不大的列，比如只有yes和no，考虑具体业务场景，某些场景可能也需要创建的索引

7.不建议动态创建索引

8.【强制】常用查询、join列需要创建索引

9.索引字段建议设置not null，因为个别数据库 is null 是无法使用索引的

10.删除不再使用或者很少使用的索引。MySQL中可以设置索引隐藏，等确定真正不再使用时，才真正删除

11.条件a = 1 and b = 2 。看业务情况优先创建 (a,b)的联合索引，不建议创建两个a，b的单列索引。例如ytenant_id与dr字段，需要创建(ytenant_id,dr)复合索引

12.【强制】索引可以避免排序，前提条件是等值查询的场景下。 where b = 1 order by a，创建（b，a）索引，可以避免排序

13.【强制】MySQL中bit类型不建议创建索引，某些版本会造成数据无法查询问题

14.规范索引的管理，设计表的时候需要将表的主要查询场景整理形成相应文档，然后根据相应场景设计索引，在上线脚本中标注索引用途

-- 修复物料根据code查询问题

call p_aa_updateindex('pub_org_group_detail','i_ytenant_id_code',"ALTER TABLE pub_org_group_detail add index i_ytenant_id_code(`ytenant_id`,`code`);",'add');

事务和锁

业务设计时要尽可能的设计短事务。

1.业务代码中不要使用lock table来锁表，此类方法会造成大面积阻塞

2.长时间大批的DML操作考虑分批提交

3.事务中DML操作的表要按照一定的顺序进行，例如，主子关系表中先操作子表，再操作主表。尽量避免交叉等待带来的死锁

4.长事务中，在不影响业务逻辑的前提下，尽量将DML操作放在事务的最后执行

字符集和排序规则

【强制】字符集和排序规则在实例和数据库层面统一设置。禁止在代码和脚本中指定字符集和排序规则，统一继承数据库级别字符集。

实例和数据库字符集：utf8（MySQL使用utf8mb4）。

实例和数据库排序规则：utf8mb4_general_ci（MySQL全领域统一使用utf8mb4_general_ci，不设大小写敏感。 如果有业务需要大小写敏感，需要提申请进行审核，原则上是存什么取什么）。

MySQL修改字符集正确语句如下：

ALTER TABLE table_name CONVERT TO CHARACTER SET utf8mb4 COLLATE utf8mb4_general_ci;

考虑多数据库适配，有的数据库是敏感的无法修改，因此也需要支持敏感查询。

时区

【强制】设置为东八区 GMT+08:00。

开发态SQL脚本规范

请参考【iUAP-开发规范红皮书】-【3.13 SQL脚本规范】部分。

SQL书写风格

1.SQL语句建议全部使用小写

2.连接符或运算符or、in、and、＝、<=、>=， +，- 等前后加上一个空格

3.SQL语句包含多表连接时，建议对每个表命名别名，对每个字段的使用都要带上表别名

4.在子查询中前后必须加上括号, select col1, col2 from tablea where col3 in ( select col4 from tableb where col4>0)

5.DDL语句禁止指定明显带数据库特色的语句，Engine = innodb ， ALGORITHM=INPLACE, LOCK=NONE等

6.其他禁止写法，请见【3.6.6多数据库适配-特殊语法】，★标识必须遵守的规范

性能优化
常规规则

1.在进行多条记录的增加、修改、删除时，建议使用批处理功能，建议批处理行数不超过2000条

2.建议每条SQL语句中in中的元素个数在500 以下，如果个数超过时，使用临时表或者分批查询

3.谨慎使用xx in('',''….) or yy in('','','')★

4.【强制】大数据量查询要采用分页查询★

6.尽量避免使用order by和group by排序操作，因为大量的排序操作影响系统性能。如必须使用排序操作，尽量建立在有索引的列上

7.对索引列的比较，尽量避免使用NOT 或 !=，可以考虑拆分为几个条件。如col1 是索引列，条件col1 !=0 可以拆分为col1 >0 or col2 <0

8.任何对列的函数操作会导致索引失效，所以应尽量将数据库函数、计算表达式写在逻辑操作符右边★

10.SQL语句尽可能简单，大的SQL可以拆成小的SQL语句(充分利用多核CPU)★

12.业务代码中尽可能批量查询，避免重复查询，降低SQL执行数量

尽量使用批处理，减少访问数据库的次数

利用批处理能减少程序与数据库的交互,对语句的性能有很大帮助。实现批处理的方式有：

1.批插入、批修改；

2.批操作尽量提供批量接口

3.使用in子句和临时表实现批处理

select COUNT(*)

from ic_general_h h, ic_general_b b

where h.cgeneralhid = b.cgeneralhid

and (h.cwarehouseid = '1008A110000000004ZB0' AND h.dr = 0 AND

(h.cbilltypecode = '40' or h.cbilltypecode = '45' or

h.cbilltypecode = '46' or h.cbilltypecode = '47' or

h.cbilltypecode = '4A' or h.cbilltypecode = '4C' or

h.cbilltypecode = '4D' or h.cbilltypecode = '4E' or

h.cbilltypecode = '4F' or h.cbilltypecode = '4I' or

h.cbilltypecode = '4O') AND b.dr = 0 and h.fbillflag = '2' and

b.dbizdate >= '2009-06-01' and b.dbizdate <= '2009-06-30');

该操作中发现类似的语句会执行几十条,每条语句的差别只有cwarehouseid的值不一样。每条语句的执行需要几十秒,因此整个过程需要半小时左右。可以将这些类似的SQL改成批处理 (用in来拼接这个语句):

select COUNT(*)

where h.cgeneralhid = b.cgeneralhid

and (h.cwarehouseid in (..,..,..,..) AND h.dr = 0 AND

(h.cbilltypecode = '40' or h.cbilltypecode = '45' or

h.cbilltypecode = '46' or h.cbilltypecode = '47' or

h.cbilltypecode = '4A' or h.cbilltypecode = '4C' or

h.cbilltypecode = '4D' or h.cbilltypecode = '4E' or

h.cbilltypecode = '4F' or h.cbilltypecode = '4I' or

h.cbilltypecode = '4O') AND b.dr = 0 and h.fbillflag = '2' and

b.dbizdate >= '2009-06-01' and b.dbizdate <= '2009-06-30');

这样一个语句就能解决了,最后只需要不到 1 分钟就查询出来了。

用WHERE子句替换HAVING子句

避免使用HAVING子句作为条件。HAVING只会在检索出所有记录之后才对结果集进行过滤，而这个处理需要排序、总计等操作。如果能通过WHERE子句限制记录的数目,那就能减少这方面的开销。HAVING中的条件一般用于对一些聚合函数的比较，如COUNT()等等，除此而外,一般的条件应该写在WHERE子句中。

select pk_accsubj, sum(localcreditamount)

from gl_detail

group by pk_accsubj

having pk_accsubj = '1003AA1000000000003A'

可以将SQL改写为如下形式：

select pk_accsubj, sum(localcreditamount)

from ufnczz.gl_detail

where pk_accsubj = '1017AA1000000000069S'

group by pk_accsubj

update 的时候注意写 where 条件

update 的时候注意写 where 条件,避免不必要的更新和造成更新错误(oracle)。

update table test set column=(case when code='' then value else column end);这样将更新表里所有的行，可能原来不需要更新的行也被更新了。

根据实际需要更新的行，加相应的限制条件。

如改写为如下形式：

update table set column=.. where code=...。

避免笛卡尔积

逻辑错误：

1. join 与 on 必须严格匹配，不允许出现没有 on 的 join

2. join...on 后面慎用 or，请把 or 的范围用()括起来。

避免隐式转换

select id,sku,weight,image_path,state,brand_name,name,product_area,upc,sale_unit,category,ele_gift,introduction,param

from ec_jd_sku_common where sku=8345903;

数据类型、字符集等都可能引起隐式转换问题，这种情况会造成索引失效，SQL性能变差。上边SQL由于sku 是varvhar类型，传入的值8345903数字类型导致。

所以改写SQL如下：

MySQL的DML语句避免全表扫描

UPDATE

service_digest

SET

dr = 1

WHERE provider_id = '9194af05-6466-4f8c-8c41-40768e0ee4f5'

AND service_name = 'market'

AND service_profile = 'dev'

AND dr = 0;

上面SQL单次执行6秒，但是有大量service_digest 表的更新语句处于锁等待状态。MySQL 中 DML 语句加锁原理，是对索引加锁。该条语句由于没有索引，导致了全表扫描，那么虽然他根据条件只更新一行，也会对表中所有行加锁，该语句执行 6 秒钟，那么这 6 秒钟，对该表的所有 DML 语句，都会造成锁等待。添加索引后解决。

ALTER TABLE service_digest ADD INDEX i_service_name_service_profile_pid_dr(service_name,service_profile,provider_id,dr);

MySQL中DML语句，需要查看执行计划，如果是全表扫描的话，该DML就会对全表加锁。

DML语句中使用关联代替子查询

UPDATE pb_filter_solution_common SET itemTitle = '交易类型' WHERE solutionId in ( SELECT id FROM pb_filter_solution WHERE filtersId in(select cFilterId from bill_base where cBillNo = 'st_morphologyconversionlist') ) and itemName = 'businesstypeId';

上面语句100秒都没有执行完，改写子查询为JOIN方式，秒级执行完成。

UPDATE pb_filter_solution_common pfsc JOIN pb_filter_solution pfs ON pfsc.`solutionId` = pfs.`id`

JOIN bill_base bb ON pfs.`filtersId` = bb.`cFilterId` AND bb.cBillNo = 'st_morphologyconversionlist'

SET itemTitle = '交易类型'

WHERE pfsc.itemName = 'businesstypeId';

MySQL中对子查询支持较差(8.0.21版本后得到)，尽量使用关联代替子查询。

逻辑删除字段尽可能建立联合索引

mysql> select count(*) from productsku T0 where T0.tenant_id = 2683143336350624 and T0.is_standard = 1;

+----------+

| count(*) |

+----------+

| 13646 |

+----------+

1 row in set (0.07 sec)

mysql> select count(*) from productsku T0 where T0.tenant_id = 2683143336350624 and T0.is_standard = 1 AND T0.iDeleted = 0;

+----------+

| count(*) |

+----------+

| 13646 |

+----------+

1 row in set (7.50 sec)

多了一个 T0.iDeleted = 0 条件，执行时间由0.07 sec 变为 7.50 sec，原因为执行计划走了 index_merge ，Using intersect(IX_PC_productsku_tenant_id,idx_iDeleted)，逻辑删除字段一般不建议单独建立索引，一般租户ID都是必查条件，所以要建立 租户ID_逻辑删除 的联合索引。

正确做法是建立租户ID_逻辑删除字段 联合索引，删除单独的 租户ID索引

alter table productsku add index i_tenant_id_iDeleted(tenant_id,iDeleted);

alter table productsku drop index IX_PC_productsku_tenant_id;

ONLY_FULL_GROUP_BY：

对于GROUP BY聚合操作，如果在SELECT中的列，没有在GROUP BY中出现，那么这个SQL是不合法的，因为列不在GROUP BY从句中

STRICT_TRANS_TABLES：

在该模式下，如果一个值不能插入到一个事务表中，则中断当前的操作，对非事务表不做限制

NO_ZERO_IN_DATE：

在严格模式下，不允许日期和月份为零

NO_ZERO_DATE：

设置该值，mysql数据库不允许插入零日期，插入零日期会抛出错误而不是警告。

ERROR_FOR_DIVISION_BY_ZERO：

在INSERT或UPDATE过程中，如果数据被零除，则产生错误而非警告。如 果未给出该模式，那么数据被零除时MySQL返回NULL

NO_AUTO_CREATE_USER：

禁止GRANT创建密码为空的用户

NO_ENGINE_SUBSTITUTION：

如果需要的存储引擎被禁用或未编译，那么抛出错误。不设置此值时，用默认的存储引擎替代，并抛出一个异常

MySQL中禁止在相同列上创建索引，比如index_a(col_a)/idx_a(col_a)，索引列相同，索引名字不相同，MySQL虽然不会报错，但是会影响性能，其他数据库相同列创建索引会报错。创建联合索引时，注意是否和当前索引功能重复，比如已经存在idx_a_b(col_a,col_b)，在添加索引idx_a_b_c(col_a,col_b,col_c),索引idx_a_b_c基本包含了索引idx_a_b的全部功能，添加索引idx_a_b_c后建议删掉索引idx_a_b。
PG数据库数字类型禁止like操作。
MySQL数据库在设置sql_mode=only_full_group_by后，在group by中对列名使用单引号时会报错。

业务尽量不要依赖数据库数据字典表，例如在DM中判断表存在尽量不要查询dba_tables视图，可以使用SELECT 1 FROM tt LIMIT 1; 判断该SQL是否报错来判断tt表是否存在。
针对order by name排序语句，NULL值在不同数据库排序位置是不同的，业务不要利用数据库该特性。也不要添加nulls first/nulls last 语句指定NULL值顺序，该语句可能造成索引失效。
在一个事务中如果有SQL报错，需要立即回滚该事务，不要在进行后续操作。PG数据库与MySQL、DM、Oracle行为均不同，虽然有autosave=always可以兼容MySQL、DM、Oracle等数据库相同的行为，但是存在性能问题。所以在事务中出现错误SQL程序应该控制回滚，禁止提交。

因为空字符串在Oracle是NULL，因此翻译器针对空字符串做了优化，MySQL数据库SQL语句 id='' 在Oracle中会被翻译成 id='~'。所以为了防止歧义，在MySQL业务上禁止使用~作为判断条件，即MySQL上禁止使用id='~'。

MySQL支持不同数据类型之间的DDL转换语句，例如把varchar转换为int或者longtext。但是其他数据库不支持该操作。 因此可以在DDL语句之前添加注释 /*+ change_datatype */ 让翻译器知道这是一个更改数据类型的SQL。然后翻译器会把数据库不支持该语句的SQL翻译为添加新列、根据原字段赋值新字段、删除原字段、新字段重命名4个语句。因为涉及到原字段的删除操作，开发要注意索引的变化。MySQL索引`i_name_name1` (`name`,`name1`) 删除 name后，索引变为`i_name_name1` (`name1`)，Oracle/DM组合索引删掉一个字段，整个索引都会被删除。修改数据类型的每个列写一条语句，禁止也其他DDL混用。

call p_aa_updatecolumn('tt','name',"/*+ change_datatype */ ALTER TABLE tt MODIFY name int;",'alter');

MySQL如果有单独加注释的语句，可以添加注释 /*+ add_comment */ 让翻译器知道这只是一个纯加注释的语句，翻译器会做特殊处理。

call p_aa_updatecolumn('ba_dataauth_scene','name6',"/*+ add_comment */ ALTER TABLE ba_dataauth_scene MODIFY COLUMN `name6` varchar(255) DEFAULT NULL COMMENT '场景多语字段';",'alter');

对表新加字段有not null属性的，需要有默认值。Oracle数据库中如果表中存在数据新增字段没有默认值会报错。
使用YMSDataSource 时，禁止在连接中手动使用命令切换schema，例如MySQL的 user schema_a; Oracle 的 alter session set current_schema = schema_a; PG 的 SET search_path TO schema_a; 等。
执行 alter table rename 语句时，索引名字是不变的，需要考虑对象名称全局一致性问题。尤其把A表rename成del_A，又立刻创建A表的场景。
