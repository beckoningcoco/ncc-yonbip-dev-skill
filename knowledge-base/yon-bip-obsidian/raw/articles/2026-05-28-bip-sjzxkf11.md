事件中心开发介绍
最后更新时间：2024-09-24
概述
编写目的

事件中心在整个BIP中间件使用中占比很大，内核是Kafka,主要目的是实现应用异步解耦、流量削峰、增强系统的吞吐量等。使用的场景比较广发如单据异步导入导出、审批流事件监听、单据保存或者审核后异步逻辑处理等。编写这个文档主要是让开发人员了解掌握事件中心发布订阅（事件的发送接受）能力，更好的利用事件中心实现业务的解耦增加系统的健壮性。

读者对象

开发人员、架构师、实施顾问、运维人员等

章节简述

事件中心主要包括以下几个内容：事件中心基本信息（事件中心地址、EOS对接），事件中心落库存储、事件中心事件类型的添加和监听、事件中心的发送和接受

添加事件类型和事件监听
添加事件类型

在事件类型管理New这里点击新增

在弹窗内填写事件源编码唯一性，微服务编码和领域编码填写引擎编码这里是(zdkypd),应用编码这里选择自己创建时间类别然后保存即可

添加事件监听

第一种在事件类型哪里选择查看事件监听

第二种点击事件监听管理New，点击新增

填写事件类型、监听名称、节点编码、所属领域、微服务编码、领域编码、请求方式、事件接受地址（回调地址），这里说明下，如果是Token请求填写Rest地址，如果是RPC请求填写RPC地址

事件中心异步消息处理

异步消息使用的场景，主要适用于逻辑比较复杂耗时的操作，事件中心默认超时时间为Rest接口4秒，RPC接口10秒，如果超过的话，那么就会超时timeout消费失败，此时的解决办法使用异步消息处理，页面监听采用上面的Rest监听形式，看下下面的代码

@PostMapping("/asyncCallback")
public Map<String,String> asyncCallback(HttpServletRequest request){
BusinessEvent businessEvent=getBusinessEvent(request);
Runnable runnable = new Runnable() {
@Override
public void run() {
EventAsyncResultVO eventAsyncResult = new EventAsyncResultVO();
eventAsyncResult.setEventId(businessEvent.getUuid());
try {

// 业务逻辑处理，业务需要处理幂等
log.error("====异步逻辑处理:eventAsyncResult:{}",JSON.toJSONString(eventAsyncResult));
// 设置回调状态
eventAsyncResult.setSuccess(true);

}catch (Exception e) {
// 业务失败,设置false
eventAsyncResult.setSuccess(false);
//失败后是否需要重试
eventAsyncResult.setNeedRetry(true);
//失败后返回异常信息
eventAsyncResult.setMsg(e.getMessage());
}

//回调事件中心接口
try {
// 任务出来完后执行回调，通知事件中心已完成，这里不可靠，事件中心如果30分钟未收到会重试
eventCallBackService.callBack(eventAsyncResult);
log.error("异步回调完成");
} catch (Exception e) {
// 回调失败
log.error("异步回调事件中心失败，需要重新处理上报，{}", JSON.toJSONString(businessEvent));
}

}
};

// 主线程
Map<String,String> map=new HashMap<>();
map.put("success", "true");
//子线程处理业务逻辑并且回调事件中心接口
new Thread(runnable).start();
return map;
}
