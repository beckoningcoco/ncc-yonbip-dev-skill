消息开发
最后更新时间：2024-10-23
基本概念
背景

消息目前包括消息中心和消息平台两个服务。

消息平台是BIP平台应用平台的支撑组件之一，核心提供消息模板能力及消息终端集成能力，并依托mdd框架提供对业务的便捷接入能力。

消息中心包含待办消息、工作通知和预警通知三部分。入口都一般放在首页明显的位置，主要用于系统消息、审批消息等帮助用户快捷处理工作事项。

名词解释

** 消息通道**：消息触达用户的物理渠道，目前支持通道有：邮件、短信、微信、APP消息、友空间。除了消息平台默认的集成通道，开发者可以提供扩展通道来覆盖默认通道。

模板类型： 模板类型是业务方使用消息平台模板能力的一种对接约定，相当于业务场景，其粒度为领域/应用/单据，并天然为系统级。

消息模板：消息模板为包含变量预定义的格式，消息模板有且仅有一个模板类型，消息模板定义在通道+语种粒度上。

模板变量：消息模板支持的主要变量，业务变量、系统变量、UI元数据、实体元数据、表单数据。

消息发送策略：消息发送策略用于全局控制消息的发送行为，可以过滤通道、设置延时。

待办消息：待办消息是需要用户处理的待办事项。分为待办、已办、删除三种状态，新发的消息是待办状态，业务变动后一般会通过服务端调用消息接口修改消息状态为已办。

工作通知：工作通知是需要用户查阅的系统消息，分为未读和已读俩种状态。用户查看后变为已读状态，工作通知的状态变更不关联业务变动。

预警通知：预警通知顾名思义就是比工作通知优先级更高的系统消息，内部逻辑和工作通知一致。

产品亮点

消息平台提供了多种通道能力，涵盖了目前主流的短信、邮件、微信等通道，同时还支持第三方app的扩展对接。消息中心支持混合云能力，能够应对混合云的消息场景。

消息模板设置

系统支持四种消息类型：业务动作、预警、流程、业务事件。新的消息模板类型需要事先预置到系统中，并为新的消息模板类型新增一个系统模板。

业务动作和业务事件的区别：业务动作和业务事件很类似，在使用场景上有些许不同，业务动作配合业务消息配置使用，主要针对某个表单的动作发生后自动触发规则配置，业务事件则是在某些业务事件发生时由业务方调用消息平台接口进行消息的发送。简言之业务动作是提前配置，自动触发；业务事件是业务方自行触发。

运行态时，消息模板配置的主要任务是将系统模板复制到企业账号下或者直接新增企业账号下的模板，然后将模板分配到组织。

业务消息配置

业务消息配置是为业务事件类消息发送的通知配置消息模板、发送方式，配置接收人等，根据不同的业务单据配置业务消息。

可以看到，业务消息设置的主要信息包括：

消息名称：新增某一个消息时，名称要手工输入，可以为一个类型的相同动作设置不同的消息，例如同样是采购订单审核动作，当金额大于10万时要发送给领导，否则只通知仓库和需求部门，发送的消息模板也不一样。
消息动作：当动作发生时发送通知，动作可以是单据上的一个按钮，也可以扩展注册的动作。
配置类型：不同的单据类型。
消息接收人：单据上的人员、直接选用户、角色、组织部门（组织部门下的人员）、按规则选人，规则是系统预置的，与单据相关的人员，例如采购订单的需求组织负责人、仓库负责人等。
消息发送通道：可同时选择多种通道：友空间-工作台：发送通知类消息（不是预警和待办）、邮件、短信：短信未充值不会发送，未设置短信签名则不会发送。如果选择了多个通道，但是选择的模板未包含此通道，则消息不会从没有设置模板的通道发送；
发送条件：如果设置则达到条件时才发送，不设则动作发生即发送消息。可设置多个条件，可以设置条件为同时满足或满足其一。

消息发送未使用配置的模板？

答：对于泛匹配模板发送，比如按组织、按模板类型等发送接口，模板的优先级一般按组织默认、租户默认、系统级来匹配，所以检查默认有没有启用、有没有设置默认是必要的。对于显式指定模板的发送，不存在此类问题。

消息没发出来？

答：这个问题非常宽泛，目前实践上按以下链路排查比较

业务调用方是否有请求拦截、阻断，特别是一些前置rule。
消息模板是否启用、设置默认
涉及多语发送的时候对应语种模板有没有配置
消息通道是否配置且验证
接收人用户的手机、邮箱是否配置无误
消息策略是否关闭了通道
消息发送失败

答：消息发送接口返回的mgslist里面有详细的失败原因，按图索骥。

短信发不出来

答：公有云先确认短信余量够，其次确认租户已授权。

专属化请确认消息平台里yms的apilink配置apilink.sms.send.apicode(不要和工作台的短信验证码混淆)，其次排查网络问题

邮件通道使用163邮箱配置密码不正确，明明是对的？

答：类似163、qq这类网络邮箱一般不支持明文密码验证，请在邮箱设置里面开通smtp协议，并使用授权码作为邮件通道密码配置。

模板里设了图片，但是收到的邮件里不显示？

答：类似outlook之类的邮件客户端对于多媒体资源，尤其是网络资源是默认不显示的，并不是邮件本身的问题。

短信发送提示签名错误？

答：境内的短信供应商要求通知或订阅类短信必须有头或尾签名，该签名在你购买短信服务时凭合法资质申请，签名与授权是一致的，这个与你是否使用apilink短信无关。

消息明明显示发送成功了，但是收到了邮件退信？

答：邮件递达的模型可以简化为：发件客户端-发件邮件服务器-收件邮件服务器-收件客户端，这三个阶段时间上是分离的，所以发送端无法感知二三阶段产生的拒绝信息，请选择合适的邮件供应商及白名单设置。

发消息是否一定要创建模板类型、配置模板等一些列操作？

答：否。业务方可以自己组装消息，走消息平台的plainMessage类发送接口或标准接口不设templateCode，消息通道共享也是消息平台的重要能力。

模板上UI元数据取不到，是空的？

答：请参考3.3.1排查在mdd应用注册表里是否正常注册queryapi的configkey。

是否允许跨租户使用模板

答：不允许跨租户使用模板，允许跨租户发送消息，系统模板是全租户可用的。

业务消息配置rule不生效

答：可能因为redis未更新导致billruleregister里面的新增rule未生效，重启服务或者刷新redis

常用消息发送接口

接口：IMsgSendService

接口类型：rpc

说明：发送普通消息给友户通用户接口，调用该接口可以发送普通消息，支持的通道有短信、邮件、工作通知、待办、预警通知。

方法名：sendPlainMessageToUsers(String sysId, String tenantId, List receivers, List channels,String messageType, String title, String content, Map<String, Object> extParams)

参数说明

参数名	意义	是否必须
sysId	系统Id，公有云为diwork	是
tenantId	租户Id	是
receivers	接收人友户通ID列表	是
channels	发送通道	是
messageType	消息类型，支持notice(通知)和creatToDo(待办)和prewarn(预警)	否，默认为notice
title	消息标题，如果是短信，这里是短信签名；如果是邮件，这里是邮件主题；如果是友空间，这里是标题	是
content	消息内容	是
extParams	扩展参数，发送消息时传入各个通道的扩展参数	否

参数示例：

发送短信消息

{

"tenantId": "xcy78984",

"sysId": "diwork",

"receiver": ["1b1123f0-5d62-41fe-b619-15922397f917"],

"channels": ["sms"],

"subject": "用友网络",

"content": "您好，测试短信。"

}

发送友空间工作通知，需要在extParams中传入工作通知需要的参数，key为uspaceExt，value是个Map格式，按需传入，包括：
tabId:工作通知展示的页签ID（不传使用默认值）；
url:工作通知跳转URL；
webUrl:web端跳转URL；
miniProgramUrl:小程序跳转URL。

{

"tenantId": "xcy78984",

"sysId": "diwork",

"receiver": ["1b1123f0-5d62-41fe-b619-15922397f917"],

"channels": ["uspace"],

"subject": "通知测试",

"content": "您好，工作通知测试。",

"extParams": {

"uspaceExt": {

"tabId": "5d2ec7a89d12c02c65e0f557",

"url": "www.yonyoucloud.com",

"webUrl":"www.yonyoucloud.com",

"miniProgramUrl":"www.yonyoucloud.com"

}

}

}

发送友空间待办消息，需要在extParams中传入待办需要的参数，createTodoExt，value是个Map格式，按需传入，包括：
appId:应用Id，不传使用默认值
businessKey:待办key，用来标记待办使用，不传使用UUID
typeName:移动端页签名称
mobileUrl:移动端跳转URL；
webUrl:web端跳转URL；
miniProgramUrl:小程序跳转URL

{

"tenantId": "xcy78984",

"sysId": "diwork",

"receiver": ["1838579e-6b2a-40e0-8c36-9d64817ba53c"],

"channels": ["uspace"],

"messageType":"createToDo",

"subject": "待办测试",

"content": "您好，待办测试。",

"extParams": {

"createTodoExt": {

"appId": "5d2ec7a89d12c02c65e0f557",

发送模板消息给友户通用户接口

说明：调用该接口可以发送模板消息，支持的通道有短信、邮件、工作通知、待办、预警通知。

方法名：sendTemplateMessage(String sysId, String tenantId, String templateCode,List receivers, List channels, String messageType, Map<String, Object> bizData,String metaId, Map<String, Object> extParams)

参数说明

参数名	意义	是否必须
sysId	系统Id，公有云为diwork	是
tenantId	租户Id	是
templateCode	模板code	是
receivers	接收人友户通ID列表	是
channels	发送通道	是
messageType	消息类型，支持notice(通知)和creatToDo(待办)和prewarn(预警)	否，默认为notice
bizData	业务变量数据，key-value格式	否，模板中包含业务变量需要传入
metaId	元数据的es中的id	否，模板中包含元数据需要传入
extParams	扩展参数，发送消息时传入各个通道的扩展参数	否

参数示例和接口1.1.1类似，不再举例。

发送普通消息给组织或角色接口

说明：调用该接口可以发送普通消息给指定的角色或指定组织下的用户。

方法名：sendPlainMessageToSelectionUsers(String sysId, String tenantId, List orgIds,List roleIds, List channels, String messageType, String title, String content,Map<String, Object> extParams);

参数说明,其他参数的说明和举例见接口3.2.1

参数名	意义	是否必须
orgIds	组织Id列表	否，发送给组织下用户时传入
roleIds	角色Id列表	否，发送给指定角色时传入

参数示例和接口1.1.1类似，不再举例。

发送模板消息给组织或角色接口

说明：调用该接口可以发送普通消息给指定的角色或指定组织下的用户。

方法名：sendTemplateMessageToSelectionUsers(String sysId, String tenantId, String templateCode,List orgIds, List roleIds, List channels, String messageType,Map<String, Object> bizData, String metaId, Map<String, Object> extParams);

参数说明,其他参数的说明和举例见接口3.2.2

参数名	意义	是否必须
orgIds	组织Id列表	否，发送给组织下用户时传入
roleIds	角色Id列表	否，发送给指定角色时传入

参数示例和接口1.1.2类似，不再举例。
