概述

软件开发的道、法、术、器：

道：指的是开发人员的思想、理念和价值观，包括如何理解和解决问题、如何进行架构设计、如何进行团队协作等。这部分内容通常涉及到软件工程、计算机科学等方面的基本理论和方法论。

法：指的是开发人员在实践中采用的技术、方法和流程，包括如何进行需求分析、设计、编码、测试、部署、维护等。这部分内容通常包括各种编程语言、框架、工具、流程等具体技术。

术：指的是开发人员在具体实践中积累的经验和技巧，包括如何处理各种具体的问题、如何调试、如何优化、如何提高代码质量等。这部分内容通常需要开发人员进行不断的实践和积累。

器：指的是开发人员在实践中使用的各种工具和软件，包括IDE工具、版本控制工具、构建工具、测试工具、部署工具等。这部分内容通常需要开发人员不断学习和掌握最新的技术和工具。

名词术语

开发平台：开发平台是一种快速构建业务应用程序的工具，通过可视化的开发界面和模块化的组件库，实现了无需编写代码或少量编写代码的应用程序开发方式，缩短了开发周期和降低了开发难度，提高了开发效率和应用程序质量。这里我们着重指YonBuilder开发平台。

分布式系统：分布式系统是由多个节点组成的计算机集群，节点之间通过网络进行通信和协调，以实现系统资源的共享和任务的协同处理。分布式系统具有分布性、异构性、并发性、容错性等特点，是大规模计算和数据处理的基础。

中间件：是一种软件基础设施，它为应用程序和操作系统之间提供了一系列通用服务和功能，如消息队列、缓存、RPC、数据库连接池、负载均衡、容器等，用于提高应用程序的可用性、可扩展性和可靠性。

模型驱动开发：模型驱动开发是一种基于模型的软件开发方法，开发人员通过使用可视化的建模工具，设计和定义软件系统的模型，然后通过自动生成代码的方式将模型转化为可运行的软件系统，从而提高了开发效率、降低了开发成本和提高了软件质量。

微服务框架：微服务框架是一种分布式系统架构，将一个大型的单体应用程序拆分成多个小型的服务，每个服务独立部署和运行，并通过轻量级的通信机制进行交互，实现了服务的高可用、高可扩展性和高性能。微服务框架通常包括服务注册与发现、负载均衡、API网关、熔断器等组件，用于提高系统的稳定性和可靠性。

微服务治理：微服务治理是指在微服务架构中，通过一系列的管理和控制手段，确保各个微服务能够协同工作，稳定可靠地提供服务。常见的微服务治理包括服务注册与发现、负载均衡、熔断降级、容错恢复、安全防护、限流控制等。微服务治理的目的是提高微服务架构的稳定性、可靠性、安全性和可扩展性，使得系统更加灵活和适应变化。

开发平台

开发平台是一种基于可视化组件和简化编程语言的软件开发平台，旨在通过简化开发流程和减少手写代码来提高开发效率和降低开发成本。这种平台提供了一系列可视化工具、组件和预定义模板，开发人员可以使用这些组件和工具来快速构建应用程序，而无需编写大量的代码。低代码平台还提供了一些自动化功能，例如代码生成和自动化测试，可以帮助开发人员快速开发出高质量的应用程序。

低代码开发具有以下优点：

1. 提高开发效率：开发平台通过可视化组件和自动化工具，可以大大减少手动编码工作，从而提高开发速度和效率。

2. 降低开发成本：使用开发平台可以减少人工编码，简化开发流程和减少开发成本。

3. 易于学习和使用：相比传统的编程语言，低代码平台提供了一种更易于理解和掌握的方式来开发应用程序，减少了学习成本和培训时间。

4. 更高的可重复性和可扩展性：开发平台的组件和模块具有可重用性和可扩展性，可以在不同的项目中重复使用，从而提高开发效率和质量。

5. 便于维护和升级：由于低代码平台的代码更加规范化和标准化，因此更容易维护和升级，减少了应用程序维护的成本。

低代码开发虽然具有许多优点，但也存在一些缺点：

1. 有限的自定义能力：在低代码平台上进行开发，开发人员可能会受到平台所提供的工具和组件的限制，无法自由地实现特定功能。

2. 学习成本：虽然低代码平台相对于传统编程语言来说更容易上手，但在使用过程中仍然需要学习平台的规则和语法，对初学者来说可能需要花费一些时间和精力。

3. 可能存在性能瓶颈：由于低代码平台是基于组件的开发方式，有些时候可能会出现组件之间的性能瓶颈，导致应用程序的性能不如传统的手写代码开发。

4. 依赖平台：使用低代码平台进行开发，可能需要依赖于平台提供的工具和服务，一旦平台升级或出现问题，可能会影响到应用程序的正常运行。

总之，低代码开发虽然具有许多优点，但也需要考虑到其缺点，开发人员在使用低代码平台之外，还需要巩固自己的基础开发能力。

概述

开发平台 YonBuilder，基于数字化建模，基于结合自身的平台、通用领域和行业 应用，以及多行业不同规模客户的优势，重点聚焦企业级应用所需要要的低代码平台核心能力： 模型、事件、自动化、集成。

以云原生、多租户、模型驱动为基础技术，将分散独立的构建模型通过可视化、业务组件装配方式有机的融为一体，为开发者、实施、业务一线人员提供低代码甚至无代码的应用构 建能力，快速满足业务变化、实现商业创新。

开发平台提供无代码、低代码、全代码三种开发模式。

无代码：通过可视化设计器，不具备代码开发经验的人，也可以通过简单的向导、可视化拖拽，实现常见的页面及流程设计，包括页面展现、页面相关数据的增删改查、审批流配置、业务流流转等，设计后可直接在线使用。
低代码：具备 js 脚本编写能力的开发者，可以在线针对已经可视化设计完成

的页面、流程等，进行编写和调试前后端事件/逻辑处理脚本，无需编译直接生效。

全代码：当应用需要通过进行微服务自治管理时，在可视化构建的基础上，开发者可以通过创建自建应用引擎的方式下载前后端脚手架进行专业开发，获得前端 React 脚手架工程、后端 Spring Boot的 java 脚手架工程进行专业开发人员的本地开发调试；且通过应用引擎的管理，提供与技术中台的运维监控、微服务治理形成开发运维一体化的闭环。

创建业务对象

新架构最大的变化就是一切基于业务对象，无论是我们的实体、页面、流程、UI模板等，都是在业务对象上面进行操作，并实现对应的场景。

因为业务对象是一个租户级元数据概念，所以它存在我们的元数据存储容器内，当前R6版本为RDB，在iuap_metadata_base下的md_biz_obj表里。

业务对象在各个支持服务下的显示，比如特征、流程、编码规则里的使用，是以业务场景的勾选来体现的，如果某个业务对象没有在对应的服务里显示，可以检查一下，是否场景支持这里，忘记了勾选。

创建实体

实体的存储也在存在我们的元数据存储容器内，当前R6版本为RDB，在iuap_metadata_base下的design_meta_class（未发布）和md_meta_class（已发布）里。通过这两个实体表之间的差距，达成实体的未发布，更新，已发布的状态机变化。

页面建模提供应用的功能页面设计器、组合页面设计器、报表设计器，支持不同功能业务页面的设计，同时支持前后端扩展，实现单据页面、画布页面、业务事件的构建和交互驱动过程的个性化扩展。

模型驱动开发

模型驱动开发（Model-Driven Development，MDD）是一种软件开发方法，它以模型为中心，通过自动化的方式生成代码、文档、测试等工件，提高软件开发的效率和质量。MDD的核心是将系统设计和实现的过程从编码中解放出来，而是将设计和实现的过程都放在模型中进行，通过模型自动生成代码的方式来实现软件系统的开发。MDD通过规范化和自动化的方式，提高了软件开发的效率和质量，同时也能减少开发成本和错误率。

模型驱动的优点：

提高开发效率：通过使用模型驱动的方法，可以快速构建出高质量的代码和系统，提高开发工作效率。
提高质量：模型本身就具有比较严谨、规范、精准的特点，故使用模型驱动开发也可以有效提高系统的质量。
减少错误：模型驱动开发中，因为模型本身就是经过验证的，因而可以减少开发过程中产生的错误。
易于维护：模型驱动开发可以通过修改模型来实现系统功能的变化，这样可以更加容易地维护系统。
更好地与领域专家沟通：使用模型驱动开发，可以将业务领域的专业术语和具体的业务流程转化为可视化的模型，这更有利于与业务专家之间的沟通和理解。

模型驱动的缺点：

可能增加复杂性：模型驱动开发可能导致系统复杂性的增加，需要对模型进行维护。
学习成本高：模型驱动开发需要对模型语言的学习，需要一定的时间。
需要更高的技能水平：模型驱动开发需要高技能的软件开发人员才能进行开发。
可能不能完全满足所有需求：模型本身可能不能完全满足所有需求，需要进行一定的二次开发。

分布式系统知识

分布式系统解决了什么问题，怎么样解决的？分布式系统的出现主要解决的是单体系统方面的不足，下面我们就具体来分析一下。

首先，分布式系统解决了单机性能瓶颈导致的成本问题。由于摩尔定律失效，廉价 PC 机的性能瓶颈无法继续突破，虽然小型机和大型机能实现更高的单机性能，但是成本太高，一般的公司很难承受。

然后，解决了用户量和数据量爆炸性地增大导致的成本问题。进入互联网时代，用户量爆炸性地增大，用户产生的数据量也随之增大。

接着，满足了业务高可用的要求。互联网产品基本都要求 7 * 24 小时提供服务，对于停止服务等故障是无法容忍的。如果想要提供高可用的服务，唯一的方式就是通过增加冗余来完成。那么就算是单机系统可以支撑的服务，因为要满足高可用的要求，也会变成一个分布式系统。

最后，分布式系统解决了大规模软件系统的迭代效率和成本的问题。如果一个大规模的软件系统是一个单体系统，那么大量的开发人员就只能将一个大规模软件，整体进行编译、测试和发布，这样一来开发语言和生态都比较单一，系统的迭代效率就会非常低。同时，对于系统的各个部分，我们也不能有针对性地进行成本优化，这会产生非常大的成本问题。

目前分布式系统依然还在快速发展中，它不停地掀起一波又一波的浪潮，快速地席卷着 IT 技术的方方面面。我们可以看到它在不同的业务场景中有着不同的表现形式，比如流量路由策略加多副本部署（微服务是其中的一种架构形式）是无状态服务的分布式架构方案， Redis Cluster 和 Codis 等方案实现了缓存的分布式化，而 Kubernetes 则完成了操作系统的分布式进化。

基础知识

分布式系统是指由多台计算机组成的系统，这些计算机通过网络进行通信和协作，共同完成一个或多个任务。以下是一些分布式系统中常见的基础知识：

1. 一致性模型：在分布式系统中，为了保证数据的一致性，需要使用一致性模型。常见的一致性模型有强一致性、弱一致性、最终一致性等等。

2. 分布式算法：在分布式系统中，需要设计和实现一些分布式算法，例如分布式锁算法、一致性算法、分布式事务算法等等。这些算法可以保证分布式系统中的数据一致性和并发性。

3. 分布式存储：在分布式系统中，需要使用分布式存储来存储数据。常见的分布式存储系统有HDFS、Ceph、GlusterFS等等。

4. 分布式计算：在分布式系统中，需要进行大规模的计算任务。分布式计算可以将任务分解为多个子任务，由多个节点并行处理，最后将结果合并得到最终结果。常见的分布式计算框架有MapReduce、Spark等等。

5. 服务治理：在分布式系统中，需要进行服务治理，包括服务发现、服务注册、负载均衡等等。常见的服务治理工具有Zookeeper、Consul等等。

以上是一些分布式系统中常见的基础知识，深入理解这些概念可以帮助我们更好地设计和实现分布式系统。

分布式算法

常见的分布式算法包括：

1. Paxos算法：Paxos算法是一种保证分布式系统一致性的算法，主要用于解决分布式系统中的数据同步和复制问题。

2. Raft算法：Raft算法也是一种保证分布式系统一致性的算法，与Paxos算法类似，但更易于理解和实现。

3. Gossip算法：Gossip算法是一种基于消息传递的分布式协议，通过随机地选择节点进行信息传播，以实现数据同步和复制。

分布式锁
分布式锁的概念

首先，先认识一下什么是锁。

锁是多线程同时访问同一资源的场景下，为了让线程互不干扰地访问共享资源，从而保证操作的有效性和正确性的一种标记。

synchronized关键字 、juc的ReentrantLock 等，都是解决单体应用单机部署的资源互斥问题。

与普通锁不同的是，分布式锁是一种用于协调分布式系统中共享资源的机制，其目的是保证在分布式环境中，同一时间只有一个节点能够访问共享资源，从而避免数据竞争和冲突，确保数据的一致性。以下是常见的几种分布式锁的实现方式：

1. 基于数据库的分布式锁：可以利用数据库的事务机制来实现分布式锁，例如使用数据库的行级锁或者乐观锁等机制，保证同一时间只有一个节点能够修改共享数据。但是这种方式存在数据库性能瓶颈问题，因为所有节点都需要访问同一个数据库。

2. 基于缓存的分布式锁：可以使用分布式缓存来实现分布式锁，例如使用Redis的SETNX（SET if Not eXists）命令，通过在缓存中设置一个标志位来保证同一时间只有一个节点能够获取锁。但是这种方式需要考虑缓存节点故障和网络延迟等问题。

3. 基于ZooKeeper的分布式锁：可以使用ZooKeeper来实现分布式锁，例如使用ZooKeeper的临时节点和watch机制，通过创建一个临时顺序节点来表示获得锁的节点，其他节点通过watch机制来监听节点的变化，实现分布式锁的协调。但是这种方式需要考虑ZooKeeper集群的高可用性和性能问题。

这些实现方式都有各自的优缺点，需要根据实际情况来选择适合的方式。

推荐使用非阻塞锁：

只尝试拿一次锁，没拿到就放弃
@Resource(name = "ymsDomainYmsLockFactory")
private YmsLockFactory factory;
  
public void doBusinessInLock()  {
  
    YmsLock lock = factory.getLock(lockKey);
    boolean getLock = false;
    try{
        getLock = lock.tryLock();
        if(getLock) {
           // 这里是获取锁后做的业务逻辑
        }
    } finally {
        if(getLock) {
           lock.unLock();
        }
    }
  
}

分布式事务
分布式事务的概念

分布式事务是指涉及到多个节点的事务操作，其中每个节点可能位于不同的系统、进程或计算机中。在分布式环境中，要确保所有参与者都遵循事务的ACID属性（原子性、一致性、隔离性和持久性）。

实际上，分布式事务主要是解决在分布式环境下，组合事务的一致性问题。实现分布式事务有以下 3 种基本方法：

基于 XA 协议的二阶段提交协议方法；

三阶段提交协议方法；

基于消息的最终一致性方法。

其中，基于 XA 协议的二阶段提交协议方法和三阶段提交协议方法，采用了强一致性，遵从 ACID。基于消息的最终一致性方法，采用了最终一致性，遵从 BASE 理论。

分布式事务框架YTS

分布式事务框架YTS依托于微服务治理平台的RPC框架，和iuap平台的技术平台紧密结合，统一对外提供数据最终一致性解决方案。

YTS框架支持多种运行态，包括MDD、纯IRIS、HTTP等；支持多种一致性控制模式，如同步Sagas、异步Sagas、TCC等。不同的运行时和控制模式的组合满足了绝大多数的业务场景需求。

框架依赖很多技术底层技术：数据库本地事务回调、分段的数据库事务日志、定时任务、Spring框架的事务管理器等等。框架支持多种数据库类型，支持MySql、SQLServer、Oracle等主流数据库及达梦等国产数据库。

同时，YTS框架包含可视化监控及问题排查能力，通过对错误数据的上报，错误堆栈的抓取，全局链路数据的跟踪，可以方便的进行错误事务的拓扑图展示，问题诊断。

YTS理论基础模型

基于YTS的MDD Rule示例
TCC方式

TCC模式把业务的操作从逻辑上划分为两个阶段及Try阶段和Confirm/Cancel阶段，首先通过rpc框架调用各参与者的业务方法（Try），如果都成功则事务管理器调用各参与者的Confirm方法，否则只要事务发起者捕获到失败，则调用各参与者的Cancel方法。

Saga方式

saga直接提交事务而没有TCC的Try的阶段，所以减少了一次通信成本，效率更高，但对于复杂的事务，提供补充机制的成本可能很高，同样没有预提交也没有资源释放及异常处理的问题，这点和TCC相比算是各有优势；另外对于长事务，即i值较大时，回退的流程可能很长，性能和效率会有较大影响；最后，没有预提交阶段，存在上节提到的Ci无法保证系统的整体状态回到Ti发生前。
