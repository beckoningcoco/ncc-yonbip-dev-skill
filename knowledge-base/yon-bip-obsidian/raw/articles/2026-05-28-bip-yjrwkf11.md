业务场景

业务系统中经常会有一些需要定时监控及时预警的任务，如安全库存监控，设备监控，能源监控等等。项目现场开发预警任务时，经常不知道如何进行开发对接，或者开发时遇到一些异常问题不知如何解决，开发效率低下。本篇文章提供具体操作以及代码示例，方便开发人员能够快速开发，提升项目的交付效率。

关键词

预警类型、预警任务、预警通知、消息通道、定时任务

专业名词术语

**预警类型：**开发平台的关键特性，为领域开发、ISV开发、客户开发提供配置服务，包括插件、参数、阈值等。

**预警任务：**运行态关键特性，包括预警任务的启用/停用、定时规则、参数设置、接收设置等；以及预警任务的数据分析、任务监控和手动执行。一个任务关联一个具体预警类型。

返回值

这个接口分为同步返回和异步返回。同步接口立马返回数据，异步接口先返回预警服务一个异步标识。后续等接口处理完毕后再调用预警服务完成后续的业务

同步返回适用于调用服务耗时很短，能短时间内返回给预警进行后续处理

异步返回适用于耗时较长，需要较长时间才能返回数据的情况。推荐优先试用异步返回方式接入

注：关于_details

_details与预警类型结果集字段一致，标有表头的字段的放到_details,不勾表头字段的默认为表体字段，放到_details下的data。如上面示例里的head1、head2属于表头字段，data里的a、b、c、d等属于表体字段。

这里的表头与表体指details里的数据结构。与实际单据里的表头表体不是一个概念。表头跟表体体现在消息模版里是形成一个列表还是单独的一个字段。_details设计为一个数组，每一条数据会独立发送一次消息

常见问题
自定义消息接收人配置了之后没有触发对应的接口

原因：调用自定义接口会先判断预警服务Rest接口有没有返回billId字段，没有billId就不会触发自定义消息接收人回调接口

解决方案：预警服务Rest接口增加返回值billId字段

预警任务执行失败

原因：消息接收人配置的规则的接受对象为空。

解决方案：重新设置一下接收人规则

任务执行成功，但是没有收到消息
场景1

原因：预警任务里没有配置消息接收人

场景2

原因：消息接收人只配置了自定义接收人规则时，规则又没有返回接收人

场景3

原因：接口返回参数不满足条件。参考5.4

预警任务发送预警通知规则是什么

1、如果领域返回数据包含_details字段，且为list类型，则发送预警通知

2、如果为报表模式且url不为空，则发送预警通知

3、如果非报表模式，且回调领域接口返回的data字段不为空（list类型），则发送预警通知

@Override
public boolean isSendMsg(Map<String, Object> msgMap, boolean isReport) {
// 1.接口响应结果中包含_details，且是非空list集合
if (msgMap.containsKey("_details")) {
return isNotEmptyList(msgMap.get("_details"));
}
// 2。报表模式，且url不为空
if (isReport && StringUtil.isNotEmpty(MapUtils.getString(msgMap, Constants.MSG_URL))) {
return true;
}
// 3.非报表模式且接口响应结果中data是非空list集合
return !isReport && isNotEmptyList(msgMap.get(Constants.MESSAGE_DATA));
}


说明：

_details字段不为空，调消息平台接口发送消息的时候优先取 _details字段值作为消息体

没有预警类型可以创建预警任务吗？
