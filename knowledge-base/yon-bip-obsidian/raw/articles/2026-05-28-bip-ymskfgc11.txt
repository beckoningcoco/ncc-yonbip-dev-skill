YMS开发过程
最后更新时间：2024-03-05
基本概念
背景

YMS云中间件是云原生微服务架构下的新一代统一技术平台，它解决了长期困扰YonBIP发展的技术多元化、高成本、低研发效能、低产品质量等问题。

YMS云中间件提供了统一的技术栈、基础技术组件、配置管理、商业中间件适配的能力,有效的支撑了公有云、专属云一套代码的交付体系。其可分可合能力，能够实现统一运行环境下的资源共享，降低微服务的服务器资源。

名词解释

**YMS：**yonyou middleware service，用友中间件服务。

**iuap-boot-starter：**YMS基础技术组件。

**yms-module-maven-plugin：**yms的打包插件，用于生成最终产物；

**服务治理平台：**用友云平台下的一块针对微服务治理统一的框架和服务，和开发者中心无缝集成，本文档中简称为治理平台或者平台；

**链路追踪：**服务治理平台集成和优化了Zipkin的调用记录信息，可以对微服务调用的时间、调用应用、状态、耗时等进行记录，方便后续查看分析；

**服务限流：**平台可以对每一个应用提供服务配置限流规则，以实现对服务的流控，确保服务能够稳定运行。限流规则可以从 QPS 和线程两个维度进行配置，帮助您在应对流量高峰时，确保系统能以最大的支撑能力平稳运行；

**服务权限控制：**平台提供的RPC框架在运行时的调用过程中，会根据客户在控制台设置的接口权限进行判断，线上环境设置为私有权限的接口，只有授权后才能够调用；

**服务元数据：**平台的RPC框架提供一系列java注解，帮助开发者声明微服务的远程方法，服务在启动时，框架会扫描工程中的远程接口的基本信息进行上报，上报的关于微服务应用的基本信息和API信息称作元数据，平台根据元数据进行远程接口的管理；

**组件隔离：**平台提供的RPC框架的SDK中，统一的处理了内部依赖的三方maven组件的版本问题，使用类隔离的机制屏蔽了框架本身引用不同版本的三方包对业务开发的影响；

**ASM：**华为云应用服务网格，提供非侵入式的微服务治理方案，支持完整的生命周期管理和流量治理，兼容Kubernetes和Istio生态，功能包括负载均衡、熔断、限流等多种治理能力。并内置金丝雀、蓝绿灰度发布流程，提供一站式自动化的发布管理。

**Istio：**Istio是一个提供连接、保护、控制以及观测功能的开放平台，通过提供完整的非侵入式的微服务治理解决方案，能够很好的解决云原生服务的管理、网络连接以及安全管理等服务网络治理问题。

**服务网格：**服务网格（Service Mesh）通常用于描述构成应用程序的微服务网络以及应用之间的交互。它的需求包括服务发现、负载均衡、故障恢复、指标收集和监控以及通常更加复杂的运维需求，例如蓝绿发布、金丝雀发布、限流、访问控制和端到端认证等。

产品亮点
统一技术栈

YMS框架对平台能力进行抽象和标准化，提供统一的RPC 框架，服务注册发现，链路追踪，监控度量、分布式事务、数据源 、日志、文件存储等组件，及多种适配器，快速实现了YonBIP全领域400+的微服务底层技术栈的替换和统一。

统一中间件和配置管理

YMS支持数据库、Redis、MQ、ZK等开源中间件及商业中间件的接入，通过统一的编程接口屏蔽了中间件底层差异，简化接入的复杂度；通过中间件物理资源和逻辑资源的分层配置,实现了中间件配置与代码逻辑的解耦；通过公共配置能力及多层级配置管理，实现微服务配置标准化和精简化，可以集中管理不同数据中心、不同环境的配置。

提供微服务治理工具

提供服务的注册发现、限流、降级、熔断及标签路由等功能，实现了微服务在线生命周期管理；提供了与YMS配套的链路追踪工具，通过采集的指标数据，实现调用关系、调用深度、调用频率和效率的检测和分析，对微服务架构治理提供了依据。

技术架构
YMS中间件总体架构

架构设计

YMS 在 Spring Boot 的基础上进行了能力的增强并提供了中间件的轻量集成，且可与 Spring Boot、Spring、Spring cloud 工程无缝集成。

YMS控制台管理了所有YMS应用的中间件和配置项的配置、增加监控中心（统一配置与观测,类似与微服务配置中心、观测中心）、中间件资源的共享

Iuap-boot-starter基于spring boot starter机制， 根据YMS配置构建Spring配置、共享中间件池对象和中间件对象。

YMS容器提供了web容器的功能，增加了用友二方包的共享功能, 减少了安装盘的体积。

YMS容器在合并工程中为下层的module提供上层共享对象服务，如同JVM跨模块事务、同JVM跨模块数据源、Redis、Mongo连接池共享。

YMS基础中间件
YMS组件框架

YMS框架基于spring boot/cloud的微服务开发自研框架，提供统一的中间件和依赖管理，每个中间件均是可插拔的组件。开发者在集成了YMS框架后，只需引入对应中间件的 starter，YMS即会自动导入所需的依赖并完成必要的配置，也能自动解决后续的健康检查、运维监控等问题，开发者能够更加专注于业务逻辑，有效节约了开发时间及后期维护的成本。

YMS容器

YMS框架提供模块隔离、模块合并部署的相关，这套工具主要服务与基于YMS框架开发的项目，其主要功能如下：

YMS-容器
在合并工程中引入YMS中间件
为YMS中间件提供自动配置功能
提供合并部署的模块间的隔离机制
为YMS框架和下层模块提供管理工具
YMS-打包插件
提供将YMS项目打包为合并module的Maven插件
去除各个module中冗余的依赖
改造合并包的结构使其适用于合并部署
暴露module中的公共类供上层使用

中间件组件

以iuap-boot-starter产品形态提供给开发者使用，上图能看出iuap-boot-starter与YMS中间件的关系，从下往上它们分别是：

1、yms-module-maven-plugin：是YMS中间件的module打包插件，使用iuap-maven-module-plugin插件打包业务工程，输出为iuap module结构的jar

2、iuap-boot-starters：提供统一版本控制和依赖管理以及中间件组件，中间件组件以starter形态提供，赋予业务工程中间件适配集成能力；

3、iuap-yms：是YMS中间件不被用户感知，对用户透明。

组件设计

Iuap-boot-starter封装了一系列的中间件SDK，提供统一的操作面，以starter形态集成；采用抽象接口或对象来隔离底层具体实现（例: redisTemplate、mongTemplate、DataSource）。

配置组件

配置组件在iuap-boot-start组件中，负责与YMS控制台进行交互，然后把从YMS控制台上拉取到的中间件组件信息通知各个中间件。配置组件承担与外部系统进行配置交互的角色，由配置组件统一进行环境、配置的拉取、初始化、解密等功能。

注意：不推荐使用原先配置文件的方式，推荐配置在YMS控制台的中间件或配置项中。

Redis组件

业务操作时，一些场景需要将数据进行缓存提供访问性能、验证码过期、排行榜、计数、限流等；解决此类往往可以通过redis中间件来支持上述描述场景。

Redis 是一个开源的支持网络、可基于内存、分布式、可选持久性的键值对(Key-Value)高性能内存数据库。

iUAP-boot-starter Redis组件提供了统一的Redis的操作方法以及统一的中间件配置管理。

Mongo组件

YMS MongoDB组件，是YMS中间件中对于mongoDB使用的封装，提供了统一的mongoDB的接入实现，简化应用使用mongoDB的操作。

日志组件

日志组件为开发者提供统一的日志格式、日志级别、日志输出文件以及日志隔离控制等功能。

分布式锁组件

业务操作时，一些场景需要将特定的资源进行锁定，防止其他用户、线程或者进程同时操作此资源，造成数据的不一致性。

解决此类问题一般是采取加锁的机制，jdk中提供的锁不能解决多个进程间的同时访问的问题，利用数据库行锁的方式性能上会存在问题。

iUAP-boot-starter分布式锁组件提供了统一的分布式锁的接入实现，支持redis为主的锁方式，简化应用使用分布式锁的操作。

Redis提供分布式的锁的机制，利用其单线程特性实现锁的效果。

MQ组件

业务操作时，一些场景需要异步的消息处理提供报表查询、结果统计、指标聚合等；解决此类可以通过MQ中间件来支持上述描述场景。

MQ是低耦合、可靠投递、广播、流量控制、最终一致性的异步消息组件。

iUAP-boot-starter MQ组件提供了统一的异步消息的操作方法以及统一的中间件配置管理。

文件组件

公有云场景，客户常用阿里云OSS、华为云OBS来存储静态文件；私有化部署场景下，客户采用的主流的文件存储是Fastfds、Minio等开源文件存储。

YMS数据源组件需要的配置，如租户和数据源链接间的映射关系、逻辑数据源和数据源链接间的关系等，由YMS管理控制台提供可视化的展示、编辑。最终存储到DB和配置中心的固定配置文件中，供其使用。

数据源信息的配置发布

运维管理和维护好新的逻辑数据源和连接、数据源连接和租户的映射关系后，点击界面的发布按钮，生成新的YMSDataSource组件需要的两个json配置文件，确定后，发布到confcenter的config表中。

YMSDataSource组件监听配置中心的两个配置文件的变化，拉取最新的配置文件内容，更新组件需要的配置信息，并触发数据源的变化。

后台调用运维人员提供的接口，获取不同的数据中心对应物理数据源下的新租户关系。
发布动态数据源配置

运维人员维护好逻辑数据源、数据源之间的映射关系后，点击发布配置按钮，YMS控制台根据对应环境，查询关系数据库中逻辑数据源和数据源连接间的关系，查询关联关系对应的租户与schema的映射关系，生成YMSDataSource组件需要的json文件，并按照版本保存到配置中心的对应分组的配置文件中。

}

org.apache.shardingsphere

sharding-jdbc-spring-boot-starter

4.1.0-YMS-SNAPSHOT

配置文件中添加如下配置信息。

spring.shardingsphere.datasource.names=prewarning

spring.shardingsphere.datasource.dispatch.logicDataSourceName=warning_shardingDataSource（命名规则同上）

使用自定义数据源

在YMS控制台配置后，YMS datasource会自动替换本地同名beans,本地无需修改。

}

进入全局日志配置项后点击编辑可以修改日志的基础属性

全局配置项说明：

是否隔离：中间件日志是否独立输出，默认是，全局变量，领域不可改
是否启用：默认是，全局变量，领域不可改。
日志格式：全局日志输出格式，管理员统一配置，领域不可改，如需定义日志输出格式，可通过配置自有的[附加器]中的文件格式实现。
基础路径：日志文件输出的路径，全局变量，领域不可改。
全局日志级别：日志级别。

ConsoleAppender附加器配置项说明：

类型：Appender类型，ConsoleAppender为控制台输出日志配置，RolingFileAppender为滚动输出文件配置，AsyncAppender为异步输出文件配置，必须配合RolingFileAppender使用，提高日志记录性能。
附加器名称：按区分度配置。
日志格式：非必填项，如果不设置则使用公共日志格式。
过滤器：过滤器用于在外层日志级别基础上，进一步对日志输出级别进行过滤，假如外层设置为DEBUG，LevelFilter：只输出确定级别的日志，例如只设置输出DEBUG级别，INFO级别就不会输出。ThresholdFilter:输出所设置级别和更高级别的，例如设置为WARN，则会输出WARN和ERROR级别。
OnMatch：符合指定的日志的处理方式(deny,neutral,accept)。
OnMismatch:不符合指定级别的日志的处理方式(deny,neutral,accept).

日志文件名称：日志输出文件名，#{logfile.name}会被spring.application.name的属性值替换掉，也可以自定义文件名。
滚动策略：日志滚动策略
轮转模式：轮转文件名，日志文件会按照规则滚动输出
最长历史保存记录：用来保存多少数量的归档文件，超出数量的文件会被清理，默认为0不清理。

/**

@Bean
public SqlSessionFactory sqlSessionFactory(AutoYmsOidInterceptor autoYmsOidInterceptor) throws Exception {
    PathMatchingResourcePatternResolver resolver = new PathMatchingResourcePatternResolver();
    SqlSessionFactoryBean sessionFactory = new SqlSessionFactoryBean();
    sessionFactory.setDataSource(dataSource());
    sessionFactory.setFailFast(true);
    sessionFactory.setMapperLocations(resolver.getResources("classpath:/mapper/*Mapper.xml"));
    /**
    * 添加插件信息(因为插件采用责任链模式所有可以有多个，所以采用数组

    Interceptor[] interceptors = new Interceptor[1];
    interceptors[0] = autoYmsOidInterceptor;
    sessionFactory.setPlugins(interceptors);
    return sessionFactory.getObject();
}

自定义Invoker进行加载

com.yonyou.cloud.middleware.framework.Invoker

IRIS会根据 @MiddleDescribe 注解里设置的 order 权重来判断加载 Invoker 的实现, IrisInvoker 的权重为0, 在没有wing的情况下IRIS会走自己内部的默认调用逻辑, 在有wing 的情况下

IRIS 会加载权重为1 的IrisWingInvoker, 此时会调用 iris-yms-support 包下的 com.yonyou.cloud.yms.utils.YmsUtils#changeInvoker 方法, 如下:

public static RpcOutput changeInvoker(String targetModule, RpcInput input, Invoker.Type type) throws Throwable {
    ClassLoader targetLoader = YMSModuleManagerV2.getInstance().get(targetModule).getCl();
    ClassLoader sourceLoader = Thread.currentThread().getContextClassLoader();
    try {
        Thread.currentThread().setContextClassLoader(targetLoader);
        MiddlewareLoader<Invoker> loader = MiddlewareLoader.getMiddlewareLoader(Invoker.class, targetLoader);
        return loader.load(type.getName()).invoke(input);
    } finally {
        Thread.currentThread().setContextClassLoader(sourceLoader);
    }
}


如此便会进入用户自定义的Invoker实现, 拦截掉默认的调用, 所以通过wing 进行多云适配首先需要实现 middleware-public 包下的 com.yonyou.cloud.middleware.framework.Invoker 接口,

重写invoke 方法, 并为该类加上注解@MiddlewareDescribe(name ="iris", order =1), 适配TSF 的重写方法示例如下:

public RpcOutput invoke(RpcInput input) throws Throwable {
    if (isInJvm(input.getAppName())) {
        return internalInvoker.invoke(input);
    }
    URI uri = input.getUri();
    if (null != input.getContextPath()) {
        String scheme = uri.getScheme();
        String path = input.getContextPath() + uri.getPath();
        String fragment = uri.getFragment();
        String query = uri.getQuery();
        String userinfo = uri.getUserInfo();
        String host = input.getAppName();
        int port = uri.getPort();
        uri = new URI(scheme, userinfo, host, port, path, query, fragment);
    }
    if (null == template) {
        synchronized (this) {
            if (null == template) {
                template = ApplicationContextHolder.getRestTemplate();
            }
        }
    }
    HttpHeaders requestHeaders = new HttpHeaders();
    requestHeaders.add("Accept", input.getContentType());
    requestHeaders.add("Content-Type", input.getContentType());
    input.getHeaders().forEach(requestHeaders::add);
    HttpEntity<byte[]> requestEntity = new HttpEntity<>(input.getRequestBody(), requestHeaders);
    ResponseEntity<byte[]> entity = template.exchange(uri, HttpMethod.POST, requestEntity, byte[].class);
    RpcOutput output = new RpcOutput();
    output.setCode(entity.getStatusCodeValue());
    if (entity.hasBody()) {
        output.setBody(entity.getBody());
    }
    output.setContentType(entity.getHeaders().getFirst("Content-Type"));
    return output;
}

自定义servlet调用

javax.servlet.http.HttpServlet

最后需要自定义wing 自己的servlet, 重写 doGet 和 doPost方法来处理调用请求, 适配 TSF 的 servlet示例如下:

@Override
protected void doGet(HttpServletRequest req, HttpServletResponse resp) throws ServletException, IOException {
    resp.setCharacterEncoding("UTF-8");
    PrintWriter writer = null;
    try {
        writer = resp.getWriter();
 
        resp.setStatus(200);
        writer.write("ok");
 
    } catch (Exception e) {
        resp.setStatus(500);
        if (writer != null) {
            writer.write("fail");
        }
        logger.error("RemoteCall servlet GET method execute fail!", e);
    } finally {
        if (writer != null) {
            writer.flush();
            writer.close();
        }
    }
}
 
@Override
protected void doPost(HttpServletRequest req, HttpServletResponse resp) throws ServletException, IOException {
 
    Cookie[] cookies = req.getCookies();
    String requestURI = req.getRequestURI();
    String module;
    int start = requestURI.indexOf("/wing/") + "/wing/".length();
    int end = requestURI.indexOf(RemoteCallConstants.SERVLET_PATH);
    if (end > start) {
        module = requestURI.substring(start, end).replace("/", "");
    } else {
        module = req.getHeader("x-iris-module");
    }
    module = YmsModuleImpl.getParent(module);
    String sessionZone = null;
    if (cookies != null) {
        for (Cookie cookie : cookies) {
            String ZONE_KEY = "zone";
            if (ZONE_KEY.equalsIgnoreCase(cookie.getName())) {
                sessionZone = cookie.getValue();
                break;
            }
        }
    }
 
    if (req.getRequestURI().contains(RemoteCallConstants.SERVLET_PATH)) {
        RpcInput input = new RpcInput();
        input.setAppName(module);
        input.setZone(sessionZone);
        requestURI = requestURI.substring(requestURI.indexOf(RemoteCallConstants.SERVLET_PATH) + RemoteCallConstants.SERVLET_PATH.length());
        String[] uris = requestURI.split("/");
        String serviceName = uris[0];
        String contentType = req.getContentType();
        input.setServiceClassName(serviceName);
        input.setContentType(contentType);
        byte[] body = IOUtils.toByteArray(req.getInputStream());
        input.setRequestBody(body);
        RpcOutput output = null;
        try {
            ClassLoader sourceLoader = Thread.currentThread().getContextClassLoader();
            ClassLoader targetLoader = YMSModuleManager.get(module).getCl();
            MiddlewareLoader<Invoker> loader = MiddlewareLoader.getMiddlewareLoader(Invoker.class, targetLoader);
            try {
                Thread.currentThread().setContextClassLoader(targetLoader);
                output = loader.load(Invoker.Type.PROVIDER.getName()).invoke(input);
            } finally {
                Thread.currentThread().setContextClassLoader(sourceLoader);
            }
            resp.setStatus(200);
            resp.setContentType(contentType);
            resp.getOutputStream().write(output.getBody());
            resp.getOutputStream().flush();
        } catch (Throwable throwable) {
            logger.error("provider invoker error, input:{}", input, throwable);
        }
 
    } else {
        resp.sendError(500, "not support");
    }
}
