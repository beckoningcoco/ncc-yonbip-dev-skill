业务场景

在开发中通过工具类获取上下文环境变量，各环境微服务拦截器恢复上下文，通过工具类切换上下文跨租户处理相关业务。

使用上下文的意义和平台提供的API
接入上下文的意义

接入上下文后访问上下文发生在JVM内，替代了频繁访问Redis数据库恢复上下文，整体性能提示10%-50%。提供隔离API避免了产生上下文脏数据产生。统一平台和领域上下文使用规范。

线程内（REQUEST范围）上下文使用

request范围的上下文只对当前线程生效，在发起http、rpc请求时不向下游传递此上下文内容，在发起新线程时也不会传递到子线程内。

API使用：

设置当前线程的request属性键值：

com.yonyou.iuap.context.InvocationInfoProxy.setExtendAttribute(key,value)

获取当前线程的request属性值：

com.yonyou.iuap.context.InvocationInfoProxy.getExtendAttribute(key)

上下文的线程间传递和切换

UserExecutors: 用于前台请求的异步执行，会自动复制登录用户的上下文到回调中, 上下文令牌会随着用户登出失效。 异步API：Future<?> future = UserExecutors.executeInDetachedThread(executorService, runnable); UserOTTExecutors: 用于后台任务的异步执行，会自动复制登录用户的上下文到子线程, 上下文令牌15小时后失效，不受用户登出影响， 可以切换到和登录用户绑定的另一个租户上下文(默认是登录用户所选租户上下文)。 同步API：Future<?> future = UserOTTExecutors.executeInCurrentThread(runnable, invocationBizBuilder); 异步API：Future<?> future = UserOTTExecutors.executeInDetachedThread(executorService, runnable, invocationBizBuilder); RobotExecutors: 用于后台任务的异步执行时根据租户ID生成上下文，生成的上下文携带默认用户上下文(原机器人用户) 同步API：RobotExecutors.runAs("$tenantId", callable); 异步API：RobotExecutors.runAs("$tenantId", callable, executorService);

租户分库

大部分情况下领域数据需要按照租户ID分库，YMS会根据上下文中的租户ID选择DataSource获取数据Connection, 需要保证在操作数据库时有上下文，上下文来源于YmsContextFilter 或YMS三类Executors的恢复 不需要分库的特殊场景（如Quartz）需要备案，在审批后可以新建一个关闭了租户路由的逻辑数据源，将不需要分库的业务指向此逻辑数据源。 需要遍历所有数据场景，提供了以下API：根据逻辑数据源编码和查询SQL执行业务逻辑 T[] result = new FededrateQuery（String logicDatasourceCode, String query).setParameter（...）.execute（ResultSetExtractor extractor); 根据逻辑数据源下的每个物理数据源DataSource或Connection对象执行业务逻辑 T[] result = new FederateOperator(String logicDatasource).query((datasource/connection)->{ // 业务逻辑}); 遍历逻辑数据源执行有事务的业务 List<FededrateResult> resultList = new FederateOperator(String logicDatasource).setPlatformTransactionManager(transcationManger).runWithTransaction(()->{}); List<FededrateResult> resultList = new FederateOperator(String logicDatasource).setPlatformTransactionManager(transactionManager).executeWithTransaction(ExecutorService executorService,()->{});

YMS上下文Filter配置

默认情况下，YmsContextFilter会根据前端或后端请求恢复上下文，拦截路径规则如下

受保护的URL:

说明：如果请求携带的令牌过期或无效，前台请求返回302重定向到登录页, 后端请求401；如果令牌有效，会恢复上下文。
默认值：/*
配置键：yms.session.contextFilterPatterns
配置方式：默认不配置时的值是 /*，如需配置请配置在应用的配置项中（spring类型配置项），配置值需要符合servlet filter的url-pattern, 支持/xxx/*或*.xxx两种形式(不是正则)。如配置多个，使用英文逗号分隔。

白名单：

说明：不拦截不恢复上下文的URL

配置键及默认值如下：

// 按照正则排除不需要恢复上下文路径 新的正则需要带上原来默认正则

yms.session.ymsSessionWhite= ^.*(/ping/?)$

// 按照后缀排除不需要恢复上下文路径 以;分割

yms.session.whiteListSuffixs= .css;.js;.jpg;.png

// 包含方式：以;分割。例："/ping/;health"

yms.session.whiteListContain=

配置方式：如需覆盖默认值请配置在应用的配置项中（spring类型配置项）。

灰名单：

说明：恢复上下文但在恢复失败时(比如YhtAccessToken校验非法) 需要放过的URL：

配置键：正则方式：yms.session.greyListRegex

后缀方式：yms.session.greyListSuffixs,

包含方式：yms.session.greyListContain

配置键及示例如下，默认配置均为空

// 按照正则排除yms.session.greyListRegex=^.\*(/ping/?)$
// 按照后缀排除 以;分割
yms.session.greyListSuffixs=.css;.js;.jpg;.png;.ttf;.ico;.html;.jsx;.gif
// 包含方式：以;分割
yms.session.greyListContain=/ping/;health


配置方式：默认值为空，如需覆盖默认值请配置在应用的配置项中（spring类型配置项）。

健康检查URL：

1：如果使用应用自定义健康检查URL，需要将应用健康检查URL加入白名单
2：如果使用yms rpc的健康检查URL，用于检查yms rpc是否启动， 格式为： http://ip:port/${ctx}/CloudRemoteCall/ ，注意最后的斜线不能去掉，已经默认加入了白名单

3：如果使用yms 的健康检查URL，用于检查spring容器是否启动，格式为：http://ip:port/${ctx}/yms/healthcheck ，已经默认加入了白名单

线程间传递上下文

业务测对线程的使用通常有两种场景,

场景一 : 响应用户界面请求, 如通过forkjoin加速查询并返回结果到前端界面, 这时候使用当前用户的线程上下文即可.

场景二: 执行后台任务,如导入场景, 这种情况是用户执行执行时立即返回用户成功, 并且进入后台执行, 如果使用用户的token将面临用户注销后token失效的问题, 需要更换一次性OTToken进行执行

YMS对这两种场景进行了包装,

针对场景1：响应用户前台界面请求

注意：以下三种方案(3.6.1.x)任选其一，不要叠加使用

使用YMS线程池

YMS 线程池 组件设计_v0.2.docx
YMS线程池默认复制主线程的当前上下文到线程内，应用在Runnable 的run() 或Callable的 call() 实现中使用InvocationInfoProxy.getXxx() API获取上下文的提交任务时的主线程上下文一致，无需使用包装器包装Runnable或Callable对象。

private ExecutorService ymsExecutor = YmsExecutors.getYmsExecutor();
@Test
public void testYmsExecutor() {
Runnable runnable = () -> {
logger.info("上下文中的 租户id:{}, 用户id:{}", InvocationInfoProxy.getTenantid(),
InvocationInfoProxy.getUserid());
};
ymsExecutor.execute(runnable);
Callable<String> callable = () -> {
logger.info("上下文中的 租户id:{}, 用户id:{}",
InvocationInfoProxy.getTenantid(),
InvocationInfoProxy.getUserid());
return "result";
};
Future<String> future = ymsExecutor.submit(callable);
Assert.assertEquals(future.get(), "result");
}

使用UserOTTExecutors执行

用于用户登录后提交的任务在用户登出时无法保证完成时的场景， 基于当前用户会话令牌，换取一次性令牌(生命周期15个小时, 不受用户主动注销影响)

OTT：意思是一次性令牌，这种token友户通只处理23个最小集的属性，使用sdk的算法恢复，不进行服务端的token的信息的落盘存储；
