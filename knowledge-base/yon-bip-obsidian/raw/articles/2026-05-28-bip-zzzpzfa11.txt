主组织配置方案
最后更新时间：2024-01-21
引言

本文档主要讲解客开单据，如何进行主组织配置，以达到主组织的控制效果；包含两种客开方式：Yonbuilder自建应用和基于MDD开发应用。

核心概念
主组织：YonBIP支持多组织（比如一个集团下的多个独立法人公司）共用一套系统。所有的档案和单据，在记录数据时都会记录一个组织的id，用来标记这个数据或者这个业务是在哪里发生的（即用做业务数据存储和业务数据维隔离的范围）。在非多职能的多组织场景下，这个主组织一般指的是法人公司。在多职能的场景下，主组织需要确定具体的职能组织类型。
目前BIP权限体系支持的数据隔离范围包含：组织、会计主体、开票点、管控范围这四个纬度。大部分业务都是使用组织或会计主体，只在特定的场景和产品下，使用的是开票点和管控范围。
**主组织类型：**描述单据可在哪种业务类型的组织下进行业务操作，如采购组织，销售组织等。对于上面四个隔离纬度来说，如果按照组织进行数据隔离，主组织类型就需要指定组织的职能。
组织职能即组织所承担的功能，不同的组织职能具有不同的业务处理规则，通过组织职能为组织明确权责，让不同组织职能的组织间彼此协同又相互制衡。目前组织的职能有采购、销售、库存、会计主体等。组织职能会对应一类的业务领域，组织必须具有这些职能才能做相应的业务，例如会计主体职能，主要在财务领域应用，只有具备会计主体职能的组织才能处理财务业务。
主组织权限：设置角色能干哪个组织的事情。

主组织权限包含两方面含义，一是角色具备的组织权限范围，二是单据的主组织类型（即单据可操作的业务范围，如采购、销售等）。例如为用户的某个角色分配了10个组织的权限，其中6个具有采购职能，4个具有会计主体职能，则该用户登陆后，操作采购单据时（采购订单的主组织类型为采购组织），仅可使用6个具有采购职能的组织，操作财务单据时，仅可使用4个具有会计主体职能的组织。

所有的功能权限和数据权限，前提条件是主组织权限。
一个单据只能有一个主组织字段，仅支持一个主组织类型（如果跨组织职能或者不需要指定组织职能，可以指定为组织单元，指定为组织单元的效果就是主组织的组织职能为全部，主组织权限按照授权的组织过滤，不按照组织职能过滤了）。
单据启用主组织的效果体现在以下几个方面：
1、单据列表页显示的数据会按照主组织权限进行过滤，即仅能查询到有权限的组织下的数据；
2、制单时，仅可见主组织权限范围内的组织；
3、制单时，需要先选择一个主组织，才可录入其他数据；即需要先指定单据产生的业务范围，才可进行业务操作；
4、编辑时，主组织不允许修改；

单据主组织使用介绍

本章节以采购订单为例，讲解单据是如何实现主组织控制，以及主组织的控制效果是怎样的。

在第二章节：核心概念中已介绍，主组织权限包含两方面的含义：一是当前用户的组织权限，二是单据的主组织类型，而主组织类型主要是指组织职能。因此本章节主要介绍如何配置组织职能，以及如何进行组织权限授权，准备好这两项工作后，便可看到主组织效果了。

单据的主组织类型

标准单据在元数据注册时已经指明其主组织类型，如采购订单的主组织类型为采购组织。

组织职能

本节主要介绍如何配置组织职能。

打开【数字化建模】--【组织管理】--【业务单元】；
进入详情界面，可见组织职能区域，每个组织职能具有不同的业务处理规则，选中职能后，会在下方增加该职能的配置区域，可配置该职能的业务属性；

3、配置组织职能。

如，配置【luocl集团】、【luocl集团北京分公司】、【luocl集团天津分公司】、【Q测试集团公司】具有采购职能。

注：

关于组织职能扩展，目前标准产品支持了职能扩展的功能，但是没有成熟的项目验证，如果项目有扩展需求，建议就项目需求先确定方案再实施使用。

用户授权

本节主要介绍为用户授权的过程。

创建角色，为其分配功能权限；

节点路径：【数字化建模】-【权限管理】-【角色管理】；

如：创建角色：luocl采购员，为其分配采购管理的功能。

为用户分配角色；

节点路径：【数字化建模】-【权限管理】-【授权】

如：为luocl用户分配【luocl采购员】角色

为用户分配角色下的组织权限；

如：为luocl用户在【luocl采购员】角色下分配图中的6个组织，其中4个具有采购职能，2个不具备采购职能；

注：

为用户授组织权限的时候，是选择用户的某个角色进行授权的，业务含义为该用户是某组织下的某角色，如：该用户为A组织的采购员。如果用户在不同的组织担负着不同的角色，如某用户既为A组织的采购员，又为B组织的库管员，则为该用户授权的时候，为用户授予采购员角色，对应组织权限为A，授予库管员角色，对应组织为B。这样，该用户登陆系统后，既可以看到采购的节点，也可以看到库存的节点，但是在操作采购单据时，只能使用A组织，操作库存单据时，只能使用B组织。

2、关于【企业账号级】能否做为主组织，单据在注册的时候，会有指定【主组织类型是否覆盖全局】，如果是，则【企业账号级】就可以做为主组织。目前自建应用不支持这个配置，默认都是否，即自建应用里，【企业账号级】不可做为主组织。【企业账号级】做为主组织的应用场景主要在基本档案上，有集团管控的需求，在业务单据上基本不会使用。

主组织效果

本节主要是使用上面准备的数据进行主组织效果的展示。

单据的查询结果按照主组织权限过滤；

如：luocl用户登陆，仅能查到有权限的4个采购组织的单据。

查询区域
主组织参照，会根据主组织权限过滤；

如：luocl用户仅能参照到有权限的这4个采购组织。

新增界面
新增单据时，如果没有选择主组织，则界面上的字段均不可编辑；

注：这是因为单据上的很多字段都是和主组织相关的。做单据时，必须首先指明该业务发生的组织范围，只有指明了这个组织范围，才能明确其他字段的范围。

选择主组织后，其他字段可编辑；
基本档案字段的可见范围会按照主组织进行过滤；

注：

对于启用分级管控的基本档案，档案上有两个组织属性：管理组织和使用组织；管理组织是指档案的创建组织，可以对档案进行增、删、改、用；使用组织是档案的可使用组织，即该档案可给哪些组织使用。

新增过程中，如果切换主组织，会清空数据；

编辑界面
编辑时，不可更改主组织

自建应用的主组织配置

本章节主要介绍使用Yonbuilder创建应用时，如何实现主组织的控制效果。

启用主组织

该配置是为自建单据启用主组织；启用后，单据会自动增加一个主组织字段，并可自动实现：一、主组织参照按照组织权限和组织职能过滤；二、不选择主组织，其他字段不可编辑；三、清空主组织，会自动清空数据，这三个效果。

在创建实体时，引用接口属性里，有一个【主组织】选项，进行勾选；勾选后，会自动增加一个字段：主组织，该字段即为主组织字段。

设置主组织类型

该配置设置发布单据的主组织类型，即指定该单据是哪种业务类型，如采购、销售、会计主体等。

使用应用构建发布的实体默认的主组织类型为业务单元；如果实体需要指定某一个组织职能，可如下操作：

在【数字化建模】--【系统管理】--【我的应用】中，点击某个应用后面的【服务列表】

在服务列表里找到需要更改的实体，点击后面的【编辑】按钮；

进入编辑界面，在高级设置后面，点击【展开】；【主组织类型】字段可设置该实体的主组织类型。

启用数据权限

该配置可实现可见的单据范围按照主组织权限进行过滤。即仅可查询到有权限的组织下的单据。

在应用构建-页面建模里，左侧选中根节点，右侧属性里，启用【数据权限控制】；

注：

数据权限控制，控制的是组织权限和数据权限（组织权限是数据权限一个特例）；
功能权限控制，控制的是按钮权限。
设置基本档案参照类型

该配置可实现参照基本档案时按照主组织权限进行过滤。

在应用构建-页面建模里，选中基本档案字段，点击【参照配置】；

在参照配置里，选中带“过滤主组织”的参照。

注：

自建应用里，默认使用的参照是通用参照；如果档案需要按照主组织过滤，则可按照上面步骤更改下参照。
更改参照后的效果就是，当单据上参照该档案时，会按照当前选中的主组织过滤，即仅显示使用组织为当前主组织的档案。
主组织验证

下面以使用应用构建创建出来的单据【检验单-主组织】为例，验证主组织的效果。

打开节点，默认显示所有有权限的组织的数据。

查询条件中，主组织参照按照主组织权限进行过滤；

点击【采购组织】，仅显示具有采购职能的4个组织

新增界面
未选择主组织时，其他字段不可编辑；点击采购组织参照，仅可使用4个采购组织，选择一个组织确定后，其他字段可编辑。

注：

关于新增时，可带出默认主组织的问题，标准单据已支持可以默认带出上一张保存单据的主组织，目前Yonbuilder自建应用还不支持。

基本档案参照按照主组织过滤

切换主组织会清空数据

编辑界面，主组织字段不可编辑

基于MDD开发应用

本章节主要介绍基于MDD开发应用过程中，主组织如何配置。

主组织配置
启用主组织：

给元数据里对应的组织字段添加标签：isMasterOrg=true，

设置主组织类型：

元数据里，type标签为组织职能对应的元数据地址

UI元数据里进行主组织配置：

（设置后，界面可实现不选择主组织，其他字段不可编辑的效果）

UI元数据涉及的数据库表：billitem_base. isMasterOrg设置为1。

设置基本档案过滤规则

该规则设置后，可实现参照基本档案时按照当前选择的主组织进行过滤。

1、增加参照过滤规则，根据主组织过滤物料：

在表billruleregister中注册该单据对应的规则

规则代码如下：

@Component("referOrgRule")
public class ReferOrgRule extends AbstractCommonRule {

    @Override
    public RuleExecuteResult execute(BillContext billContext, Map<String, Object> paramMap) throws Exception {
        Logger.info("调用物料事务处理单组织过滤规则");
        BillDataDto billDataDto = (BillDataDto) getParam(paramMap);
        List<BizObject> bills = getBills(billContext, paramMap);
        //根据组织id查询对应组织的物料
        String org = (String) bills.get(0).get("org");
        BillInfoUtils.appendCondition(billDataDto, "productApplyRange.orgId", "eq", org);
        putParam(paramMap, billDataDto);
        return new RuleExecuteResult();
    }
}


效果如下：仅显示使用组织为当前选择的主组织的物料。

主组织验证
制单界面，未选择主组织时，不可编辑；

选择主组织时，按照功能权限和组织职能过滤；

基本档案按照主组织过滤，仅显示使用组织为当前主组织的数据；

清空主组织，会清空数据。
