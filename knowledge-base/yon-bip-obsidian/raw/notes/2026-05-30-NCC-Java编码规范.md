## 1 Java编程规范

### 1.1命名约定

命名是 Java 开发中非常重要的一个部分，命名的好坏决定了代码的可读性、可维护性，良好的命名，一定程度上能够取代注释的作用，在Java命名中，我们规定了下面的一些原则：

1.合法的命名字符集包括26个英文字母、0~9的阿拉伯数字和下划线。命名中应该尽量的避免使用数字；

2.命名需要有一定的意义，推荐采用问题域中的术语命名，使命名在一定程度上是自描述的；

3.命名尽量的短，如果命名太长，可以采用别名的方式，或者缩写来简化命名；

4.名称缩写的规则（对于类名、字段名、变量名称、模块名称等适用）

1）删除所有的原音字母，压缩重复字母。如 button，缩写为 btn； 2）如发生命名冲突，则在某一缩写中保留原音。如 batton，为了不与 button 冲突，缩写为 batn。

5.命名过程不要使用前导下划线，也不要在名称的末尾附加下划线。

#### 1.1.2 命名约定-类和接口

类和接口建议以名词进行命名，命名需要具有一定的意义，并且尽量的避免使用缩写，除非缩写得到广泛的采用。进一步的约定如下：

1.采用问题域中的单词或单词组合，采用大驼峰命名，一些特殊单词如 URL，HTML 建议采用 Url、Html 的形式；

2.接口的第一个字符采用 Ｉ；

3.对接口的默认实现类命名方式为接口名称去掉I， 末尾加 Impl；

4.产品业务的内部逻辑实现类，以 BO 结尾；

5.数据访问的类以 DAO 或者 DMO 结尾；

6.测试用例以 Test结尾；

7.对于继承 ValueObject 对象的类，以 VO 结尾，聚合 VO 以 AggVO 结尾；

#### 1.1.3 命名约定-方法

方法命名多数为动宾结构：

1.采用问题域中的单词或单词组合，使用小驼峰命名，没有特别理由不用下划线作为分隔符，一些特殊单词如 URL、HTML 建议采用 Url、Html 的形式；

2.在 Java 中对属性方法命名遵循 JavaBean 的标准：

1）getter 方法：get + 属性名。

2）setter 方法： set + 属性名。

#### 1.1.4 命名约定-变量

   变量的命名包括实例变量、静态变量、函数参数的命名：

1.避免在命名中采用数字，除非命名意义明确，程序更加清晰，对实例变量的命名中不应该有数字；

2.变量名称是名词意义；

3.采用问题域中的单词或单词组合，使用小驼峰命名。第一个单词全部小写，后续的每个单词采用首字母大写，其余小写；

4.命名尽量简短，不要超过16个字符；

5.除了生命周期很短的临时变量外，避免采用单字符作为变量名，实例变量的命名不要用单字符。常用的单字符变量如整型用 i、j、k、m、n，字符型用 c、d、e，坐标用 x、y、z。

**示例：**

1.String getName()；

2.String isStopped()；

3.void connect()；

#### 1.1.5 命名约定-常量

使用常量取代对一些常值的引用，程序更易维护，常量命名约定如下：

1.所有的字符都必须大写。采用有意义的单词组合表达，单词与单词之间以“\_”下划线隔开；

2.命名尽量简短，不要超过16个字符。

**示例：**

**public** **static** String BTN\_AREA\_COMMON = "common";

#### 1.1.5 命名约定-泛型变量

范型中的变量我们规定只允许用一个字符进行表示，常用的字符：

1.T： 表示单一的范型类型；

2.K，V：表示 Key、Value 对应的范型类型；

3.E：表示一种与声明类是聚合关系的泛型类型。

### 1.2 注释

注释目的是为了提高代码的可读性和可维护性，通过注释描述清楚代码的目的、逻辑实现等：

1.注释要简单明了，只要提供能够明确理解程序所必要的信息就可以了；

2.对于一般的 getter、setter 方法可以不用注释；

3.可以用多行对类、接口、成员方法、成员变量、静态字段、静态方法、常量等进行说明；

4.不要在注释中使用格式修饰符，如一大串的 \* 或者 - 进行字符化的格式修饰；

5.符合 Javadoc 语法，可以用 Javadoc 来生成代码的文档；

6.采用一些标签进行文档的特定用途描述，用于帮助 Javadoc 产生文档

7.生成开发文档的需要用中文编写。

块注释

3.可以通过缩进、换行进行格式化；

4.如果注释只是简短的注释，不要使用块注释，除非注释是在行中。

#### 1.2.2 注释-类/接口

类/接口描述，一般比较详细。按照常用的说明顺序排列，主要包括：

1.通常都需要发布为API文档，因此通常采用文档注释

2.类的主要说明，以”。”或”.”结束

3.类设计的目标，完成什么样的功能

4.主要的类使用方式，如何使用该类，包括环境要求，如是否线程安全，并发性要求，及使用约束

5.已知的 bug

6.描述类的修改历史：修改人 + 日期 ＋ 简单说明

7.@author 作者、@version 版本、 @see 参照、@since 开始版本等信息

为了使形成的文档可读性好，注释中经常带有缩进和格式控制。类描述放在类的类定义的紧前面，不能有任何的空行。

成员变量、类静态变量采用文档注释，对成员变量的注释通常包括：

1.变量的意义

2.变量的合法值域

3.对并发访问的限制

### 1.3 代码书写规范

#### 1.3.1 代码书写规范-类编写规范

类的结构组织，一般按照如下的顺序：

1.常量声明

2.静态变量声明

3.成员变量声明

4.构造函数部分

5.finalize 部分

6.成员方法部分

7.静态方法部分

这种顺序是推荐的，在实际开发中可以按照一定的尺度修改，原则是程序更易读。如对方法的排序按照重要性，或按照字母顺序排列或按照方法之间的关系排列。

每个方法（包括构造与 finalize）都是一个段。多个变量声明按照逻辑共同组成一个段，段与段之间以空行分隔。

类声明时，要指出其访问控制，一般为没有修饰符、public 或 private。

方法与方法之间、大的代码块之间都需要以空行隔离。

![](https://community.yonyou.com/datadict/bipbook/assets/2-4.Dqnd5rZm.png)

#### 1.3.2 代码书写规范-缩进换行等格式化约定

1.一个程序文件最好不要超过 2000 行；

2.每个方法代码行数尽量不要超过 100 行（有效代码行，不包括注释），但必须保证逻辑的完整性；

3.避免过多的参数列表，尽量控制在 ５ 个以内，若需要传递多个参数时，应当使用一个容纳这些参数的对象进行传递，以提高程序的可读性和可扩展性；

4.文档注释与其注释部分不能有空行，例如方法的文档注释放在方法的紧前面，不能空一行；除了++、--、^ 这些一元操作符外，操作符和操作数之间具有一个空格；

5.括号之前之后都不能有空格；

6.数组的申明采用 数据类型\[\] + 变量名。

char\[\] buffer; //好；

char buffer\[\]; //不推荐；

![](https://community.yonyou.com/datadict/bipbook/assets/2-5.DcjU2Ufh.png)

#### 1.3.3 代码书写规范-其它算法类注释

算法描述指在实现级别的描述注释，如在方法内的注释，对类实现的注释，这样使得程序更加易懂，方便程序算法的修改和 bug 的修复。一般采用 块/行 注释，对于简短的描述采用行注释，不要用文档注释。注释的主要内容包括：

1.某些局部变量的意义和用途；

2.复杂的控制结构的注释，如 循环、分支、条件表达式。说明控制所要达到的目标，复杂代码段的描述，说明代码完成的功能，以及为什么要这样做。

### 1.4 接口设计规范

名词描述：

1.接口设计：指将本业务模块为其他业务提供的服务方法和进行远程调用的方法提取为接口的设计方式；

2.接口：指上述设计所涉及的类，包括接口类及接口方法中所使用的业务对象（如VO）和业务异常。不包括接口类的具体实现类；

3.代码设计：指代码即为设计文档，要求其注释可生成 JavaDoc 文件；

4.设计阶段：分为总体设计、详细设计；

设计要求：

1.设计工具：采用 Eclipse、Idea 直接进行代码设计；

2.文档格式：JavaDoc 文档，文档中包含接口中使用的接口类、业务对象和业务异常的描述。

### 1.5 编程要求

1.一个方法所完成的功能要单一，不同的功能需要重构封装为不同的方法或者新类；

2.局部变量在使用时刻声明，尽可能缩小变量的作用域，这样变量的可见范围和生存期也都会尽可能地小；

3.局部变量/静态变量在声明时同时初始化，特别是对于静态变量，初始化一定要简单，如果静态变量初始化复杂，特别是还依赖其他的类的静态变量的情况时，应该竭力避免，如果不能避免，应该引入静态方法进行初始化，尽量少用静态段进行初始化；

4.避免变量的定义与上一层作用域的变量同名；

5.尽可能的处理异常或转换异常，不要一味的包装异常；

6.不允许调用 System.gc；

7.如果对象在某个特定范围内必须被清理（而不是作为垃圾被回收），请使用带有 finally 子句的 try 块，在 finally 子句中进行清理；

8.把一些逻辑紧密相关的类组织在一起时可以考虑匿名类，通过匿名类访问被外部类保护的成员，具有简单安全的优点，例如按钮的事件处理监听器，推荐使用内部类；

9.对成员变量的访问最好通过 getter/setter 方法，能够保证访问的合法性，并使代码调整更容易些；

10.使用 java 标准库提供的容器类，方法的返回值、参数、成员变量的声明，都应该使用 List、Set 和Map 进行声明，这样代码具有更大的灵活性和可调整性（使用容器的接口类而不是实现类）；

11.使用这些标准的容器时，必须使用范型；

12.数组是一种效率最高的存储和随机访问对象引用序列的方式，但是当创建了一个数组对象，数组的大小就被固定了，如果在空间不足时再创建新的数组进行复制，这样效率就比 ArrayList 开销大了。所以必须明确使用场景；

13.尽量使用 private、protected 关键字。一旦你把库的特征（包括类、方法、字段）标记为 public，你就再也不可能去掉他们。在这种方式下，实现的变动对派生类造成的影响最小，在处理多线程问题的时候，保持私有性尤其重要，因为只有 private 的字段才会受到保护，而不用担心被未受同步控制的使用所破坏；

14.采用类而不是对象引用静态变量和静态方法；

15.UFDobule 常量必须使用 UFDouble.ZERO、UFDouble.ONE；

16.使用 UFBoolean 必须采用 UFBoolean.valueof(true|false|y|n|Y|N) 的形式进行构造，常量必须用UFBoolean.TRUE、UFBoolean.FALSE。

### 2.1 命名约定

1.文件、目录命名不能用中文，只能使用数字和英文字母，不能以数字开头；

2.文件、目录命名需要有一定的意义，尽量简短，名称符合各种主流操作系统的要求，如不要使用空格，不以“.”开头，不使用过长的文件名；

3.文件名称的分隔，使用下划线或者采用首单词小写，其余单词首字母大写的方式分隔。

4.模块名必须是小写的英文字母，在能表达意义的基础上尽量的简短，模块名和模块的所在目录名相同；

5.服务组件的配置文件必须以.upm 结尾，AOP 相关的配置必须以.aop 结尾；

### 2.2 属性文件

1.采用 UTF-8 编码；

2.文件后缀为.properties；

3.属性文件放在各产品模块的 resources 目录下，按照一定目录层次进行组织；

### 2.3多语资源

1.多语资源文件的字符集必须用 UTF-16 Big Endian 编码；

2.多语资源可以以 jar 包形式存放在 ncchome/langlib 目录下，jar 包名称必须以 \_langres.jar 结尾，或以目录方式存在 ncchome/resources/lang 下；

3.多语资源的结构组织如下：lang/语种编码/多语类别/多语资源文件，语种编码参考ierp/bin/langConfig.xml 文件，如 simpchn。

## 3 异常处理规范

1. 禁止对大段代码进行 try-catch，要明确抛异常的位置，对抛出异常的位置进行 try-catch；
2. 不要在 finally 块中使用 return；
3. 区分 unchecked / checked 异常，避免直接使用 RuntimeException 抛出，更不允许抛出 Exception；
4. 防止 NPE(NullPointException)；
5. 捕获异常是为了处理它，不要捕获了却什么都不处理而抛弃之，如果不处理，请将该异常抛给它的调用者；
6. 不要用异常来做流程控制、条件控制，因为异常的处理效率比条件分支低；
7. 捕获异常与抛异常，必须是完全匹配，或者捕获异常是抛异常的父类；
8. java 类库中定义的一类 RuntimeException 可以通过预先检查进行规避，而不应该通过 catch 来处理；
9. 不要在用户交互界面中泄露敏感信息，如：系统详细信息、会话标识符、帐号信息等；
10. 应对错误、异常消息进行处理，避免显示调试或堆栈跟踪信息；
11. 确保日志记录包含了重要的日志事件数据；

### 3.1异常处理示例

1.在 EJB 的边界（接口的实现方法）中，需要采用以下方式进行异常的 marsh

```
try{
    ......
 }
 catch(Exception e){
      ExceptionUtils.marsh(e);
 }
```
1. 在其他地方
```
try{
     ......
 }catch(Exception e){
     ExceptionUtils.wrappException(e);
 }
```
