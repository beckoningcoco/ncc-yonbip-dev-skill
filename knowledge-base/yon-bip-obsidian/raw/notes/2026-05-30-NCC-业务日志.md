## 1\. 前言 [​](https://community.yonyou.com/datadict/bipbook/chapter1507/chapter1507.html#_1-%E5%89%8D%E8%A8%80)

​ 在YonBIP高级版产品中，很多业务操作，需要记录业务行为的细节日志。这些业务行为日志称为业务日志，业务日志跟踪记录针对业务对象的变更操作，用于管理员跟踪和追溯业务数据的变更过程。

## 2.1 定义元数据操作模型 [​](https://community.yonyou.com/datadict/bipbook/chapter1507/chapter1507.html#_2-1-%E5%AE%9A%E4%B9%89%E5%85%83%E6%95%B0%E6%8D%AE%E6%93%8D%E4%BD%9C%E6%A8%A1%E5%9E%8B)

​ 业务日志需要通过使用日志元数据的信息来做相应的处理，需要使用者在描述业务操作的bpf文件中添加关于业务日志的描述信息。设置bpf如下图所示：

​ 其中，业务接口为调用操作方法的接口。 因为业务日志是定义到业务操作上的，因此业务操作上需要定制一部分业务日志的相关信息，如下图所示：

![](https://community.yonyou.com/datadict/bipbook/assets/1-2.vumGtzG1.png)

**上面谈到的三个设置，会对如下内容产生影响：**

-   所属实体属性：在业务日志配置中显示，选择当前操作，操作的对象属于哪个元数据，如：地点档案
-   是否日志属性：如果勾选，那么在业务日志配置节点，就可以看见该实体下挂着可支持业务日志的业务操作，在实施态，实施或客户可以控制是否真的记业务日志日志，也就是说，业务日志有两重开关，首先开发在模型设计上来确定操作是否支持记日志，而实施可以配置支持记日志的操作是否真的要在客户环境中记日志。（从左边树选到右边树的操作要记录对应选择属性的业务日志：业务日志配置节点）
-   日志类型属性：该属性类型会影响自动构造的日志内容，取值为枚举，目前主要包括： ADD（增加）：新增数据，日志内容记录该数据的关键属性 DELETE（删除）：删除数据，日志内容记录该数据的关键属性 MODIFY（修改）：修改数据，日志内容记录该数据的关键属性，记录修改前后的数据对比

## 2.3 记录业务日志代码的调用 [​](https://community.yonyou.com/datadict/bipbook/chapter1507/chapter1507.html#_2-3-%E8%AE%B0%E5%BD%95%E4%B8%9A%E5%8A%A1%E6%97%A5%E5%BF%97%E4%BB%A3%E7%A0%81%E7%9A%84%E8%B0%83%E7%94%A8)

调用业务日志接口有两种方式：

一、AOP 截获操作接口方式

​ 关于记录业务日志，平台提供nc.bs.aop.log.BusilogAspect用来截获业务操作，提取业务日志并记录之。各业务组需要将需要记录业务日志的服务注册到与upm文件同目录的\*.aop文件中，文件格式如下：

xml

```
<module priority="1025">
      <aops>
          <aspect class="nc.bs.aop.log.BusilogAspect"  component="your service interface "/>
      </aops>
</module>
```

​ 其中：元素aspect的class属性统一使用nc.bs.aop.log.BusilogAspect，Component属性为需要记日志的服务接口 ，BusilogAspect可以截获业务操作，并从中获取该操作的一些基本信息（类名，方法，参数等）， 需要且仅需要根据这些信息在元数据中获得与业务日志相关的所有信息。

二、直接调用nc.bs.sm.busilog.util.BusinessLogServiceUtil

BusinessLogServiceUtil 提供了事务中、事务后，以及异步插入单个、多个日志的六种方法。

方法1：事务成功插入一条日志–后台调用 `insertBusiLogAfterTransaction(BusinessLogContext context)` 方法2：事务成功批量插入日志–后台调用 `insertBatchBusiLogAfterTransaction(List\<BusinessLogContext> listvo)` 方法3：异步插入单条日志–后台调用 `insertBusiLogAsynch(BusinessLogContext context)` 方法4：异步批量插入日志–后台调用 `insertBatchBusiLogAsynch(List\<BusinessLogContext> listvo)` 方法5：事务中插入单条日志，日志失败事务失败（后台，事务中调用） `insertBusilogInTransaction(BusinessLogContext context)` 方法6：在事务中批量插入日志，写日志失败，事务也失败–后台用 `insertBatchBusilogInTransaction(List\<BusinessLogContext> listcontext)`

其中参数 BusinessLogContext 属性意义如下：

java

```
private String typepk_busiobj; //元数据 metaid
private String pk_busiobj; //业务对象主键
private String busiobjcode; //操作业务对象的code
private String busiobjname; //业务对象名称
private Object busiobjvo;//修改后日志，操作的业务对象
private Object oldbusiobjvo;//修改前日志，操作的业务对象,
private String logmsg; //日志信息
private String orgpk_busiobj; //所属组织
private String pk_operation; //操作的主键(如:修改,新增等等之类的)
private boolean hasjudged= false; //是否判断了写日志
```

## 2.4 分表方案 [​](https://community.yonyou.com/datadict/bipbook/chapter1507/chapter1507.html#_2-4-%E5%88%86%E8%A1%A8%E6%96%B9%E6%A1%88)

​ 为了解决并发效率问题，以及数据量增长速度问题，将业务日志的存储采用分表的方式处理。平台提供表结构， 同时提供默认表sm\_busilog\_default，各领域按照各自的需要来创建同构的日志表

自定义日志表配置：

-   文件位置：modules\\模块\\config\\busilog\\busilogproperties.properties
-   格式：uap=sm\_busilog\_uap （业务日志实体所属元数据模块编码=表名）
-   表明规范：长度不得超过30位（oracle限制），以sm\_busilog\_为前缀

服务器读取：

-   服务启动时做内存级缓存

-   查询该业务对象对应的元数据模块

-   内存中查找该模块下是否配了独立日志表，配了则向该表中写数据，否则向默认表写数据

默认表结构如下：

sql

```
 CREATE
      TABLE sm_busilog_default
      (
          typepk_busiobj VARCHAR(36)  DEFAULT '~',
          pk_busilog CHAR(20)  NOT NULL,
          busiobjcode VARCHAR(50) ,
          busiobjname VARCHAR(300) ,
          client VARCHAR(50) ,
          logdate CHAR(19) ,
          logmsg IMAGE,
          operateresult VARCHAR(50) ,
          orgpk_busiobj VARCHAR(36)  DEFAULT '~',
          pk_group VARCHAR(36)  DEFAULT '~',
          pk_operation VARCHAR(36) ,
          pk_user VARCHAR(36)  DEFAULT '~',
          ts CHAR(19)  DEFAULT 'CONVERT([char](19),getdate(),(20))',
          dr SMALLINT DEFAULT 0,
          pk_busiobj VARCHAR(36) ,
          CONSTRAINT pk_sm_busilog_default PRIMARY KEY (pk_busilog)
      )
```

## 3\. 业务日志使用注意事项 [​](https://community.yonyou.com/datadict/bipbook/chapter1507/chapter1507.html#_3-%E4%B8%9A%E5%8A%A1%E6%97%A5%E5%BF%97%E4%BD%BF%E7%94%A8%E6%B3%A8%E6%84%8F%E4%BA%8B%E9%A1%B9)

​ 在写入类型为MODIFY的业务日志时需要传入SuperVO\[\] oldVOs作为参数，生成业务日志需要从oldVO中获取一些相关数据，所以需要保证调用update、enable等涉及到修改业务的方法时要保证原VO的完整性，YonBIP高级版开发过程中出现一些业务日志记录不全，相关信息显示不完整的问题都是因为传入的VO中缺少了一部分信息导致。

​ 目前版本YonBIP高级版后台操作类中已经实现了相应的记录业务日志功能，不需要单独去实现 如批量操作的后台服务的基类： nc.bs.bd.baseservice.md.BatchBaseService，里面提供的增删改方法直接调用即可实现业务日志的记录
