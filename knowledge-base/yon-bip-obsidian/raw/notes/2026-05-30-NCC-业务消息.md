## 1.消息模板设计

NCC统一沟通平台提供了送消息的相关功能，支持消息中心、电子邮件和手机短信 3 种信息发送方式，其中电子邮件和手机短信的方式需要通过系统管理员登录系统，进入【消息管道配置】进行相关配置。我们通过注册消息类型、消息模板、业务消息配置后就可以通过代码去控制发送业务消息了。

**消息模板类型注册：** 在配置消息模板前需要先到【消息模板类型注册】添加消息类型，注册消息类型后消息模板节点左侧树才能找到自己的消息类型。消息模板类型注册界面如下图所示，左侧有三个基本的模板分类：业务消息、预警与后台任务、流程平台，可以根据自己需要在对应分类下新增层级或者添加分类。如果是业务消息则注册在业务消息下面。

![](https://community.yonyou.com/datadict/bipbook/assets/1.B4PKvPeI.png)

类型编码：根据需要填写消息类型编码，不要与已有编码重复即可。

类型名称：根据需要填写，类型名称会显示在消息模板左侧树。

关联实体：可以不关联元数据，则消息模板上无法添加元数据的变量。如果需要关联则选择元数据的主实体，需要注意的是如果新模块的元数据需要往md\_dev\_busi\_module预置业务模块和开发模块映射关系，md\_comp\_busimodule 预置元数据业务模块映射。元数据实体也需要有业务操作bpf文件并勾选是否日志，否则这里可能选择不到元数据实体。

如果消息模板上需要用到计算相关的函数，可以使用新增变量功能。需要先到【单据函数注册】，把对应函数注册之后函数名称填到变量编码上，并补充变量名称、变量名称对应的多语即可，在配置消息模板详情界面“业务函数”页签就可以找到自己注册的变量。

\*\*消息模板：\*\*消息模板分为【消息模板-全局】和【消息模板-集团】两个节点，除了可见范围不同其他都一致。在消息模板节点可以按照不同的领域模块定制消息模板，在模板中配置发送消息的内容和格式。

![](https://community.yonyou.com/datadict/bipbook/assets/2.BxvDUA-p.png)

如果要添加一个消息模板，从左边的消息模板分类中选定一个模板分类，进入消息模板的新增界面，如图所示，输入模板名称、模板描述、消息标题、消息内容，点击保存按钮。即可新增一个消息模板。消息模板的编码只在第一次新增的时候可编辑，后续修改时不允许修改。

![](https://community.yonyou.com/datadict/bipbook/assets/3.BogIxad3.png)

模板语种：模板支持的语种跟系统安装的语种类型有关，目前系统支持简体中文、繁体中文和 英文 三个语种，模板也支持简体中文、繁体中文和英文三种；

模板编码：当首次新增某一个模板分类下的模板时，模板编码需要手工输入；当已存在一个 语种的模板，新增另一个语种的模板时，模板编码不允许编辑，同一个消息模板的不同语种 的模板编码是相同的；

消息内容：可以设置要发送的消息内容，消息内容中可以引用变量。例如：“新生成的收入合同@&m name@等待审核！” 其中@......@即是所引用的变量。具体操作是从右边的系统变量页签、元数据页签或者业务函数页签中选择一个变量，如收入确认单的“合同名称”，双击该变量，则相应的变量@&m name@就作为消息内容的一部分。注意点：消息模板所属的模板分类不同，可以应用的变量也可能不同，有些模板可以应用系统变量、元数据设置消息内容，有些可以应用系统变量、业务函数设置消息内容。 某些情况下变量可能翻译不过来可以到nc.message.templet.bs.MsgTextConvertor里的convertMsgVar进行排查。

\*\*业务消息配置：\*\*业务消息配置是为系统发送的通知配置消息模板、发送方式，配置接收人等，根据不同的交易类型配置这种交易类型的业务消息。支持根据不同的规则配置不同的接收人，也可以支持自定义接收人。

![](https://community.yonyou.com/datadict/bipbook/assets/4.jzFov4sJ.png)

左树是消息源信息，通过后台预置插入进数据库，后台表为pub\_msgres\_reg，该表各个字段含义如下所示。

关于pub\_msgres\_reg重要属性说明:

moduleid 模块号必须是dap\_dapsystem表中的四位模块号主要起到界面的分类管理作用，若该模块号不合法，整条数据不会被展现；

code 消息源编码，该编码始终用于编码和关联消息模板、接收人、自定义接收人数据；

metaid 实体主键(元数据ID) 需和单据实体保持一致，因为该元数据会应用到接收人规则的解析；

msgtemptype\_code 消息模板分类编码(pub\_msgtemp\_type表中字段tempcode的值并且必须是末级分类编码)，直接关系到消息模板配置中消息模板的选取范围；

pk\_billtypecode (bd\_billtype表中字段pk\_billtypecode的值并且是单据类型)单据类型编码，关系到消息模板配置页签和接收人页签中交易类型的选取范围(取该单据类型下的交易类型)。

配置消息模板页签主要用来引用消息模板配置中定义好的消息模板，页签下面可以配置多条但需要保证交易类型编码不能重复。如下是配置消息模板页签界面：

![](https://community.yonyou.com/datadict/bipbook/assets/5.-5ZyBWpA.png)

交易类型：该消息源支持的交易类型。如果该消息源注册时只注册了单据类型，则参照该单据类型下的交易类型以及用户扩展出来的交易类型；如果该消息源注册时注册了单据类型和交易类型，则只能选择该交易类型；

发送方式：下拉选择各个消息源预置能支持的发送方式，支持系统、邮件、短信三种及其组合方式；

消息模板：参照该消息源下定义的消息模板，有全局级和集团级的消息模板。

启用状态：该条配置是否生效。

创建人及创建时间：审计信息系统自动填充，不需要过多关注。

配置接收人页签可以定义消息接收人，页签里面可以配置多条。配置接收人的界面如下图所示：

![](https://community.yonyou.com/datadict/bipbook/assets/6.C1f73nLP.png)

所属组织：消息的接收组织，根据当前登录用户过滤，参照该用户有权限的组织；

交易类型：该消息源支持的交易类型。如果该消息源注册时只注册了单据类型，则参照该单据类型下的交易类型以及用户扩展出来的交易类型；如果该消息源注册时注册了单据类型和交易类型，则只能选择该交易类型；为空标识所有的交易类型都按该规则发送；

接收规则：可以利用单据上的属性定义规则，参照单据的元数据；为空表示所有消息都发送给接收人；可以设置不同的规则发送给不同的接收人；

接收人：接收人字段支持多接收人类型(用户、用户组、角色、自定义接收人等多种)多接收人的离散选取。增行表示增加一种接收人类型，该处接收人类型不允许重复。其中自定义接收人需要相应自定义接收人提供者进行后台注册。

自定义接收人区别于用户类型等一般的接收人类型，它是为了让开发者能够自己去定义一种接收人。由于NCC的消息最终的接收人字段必须是用户主键，那么自定义接收人必须能够转化为对应的用户，这个必须由自定义接收人注册者做转换，也就是对应的到底是哪些用户。首先需要到表pub\_msgres\_sfdfrcv 自定义接收人后台注册一条数据，pub\_msgres\_sfdfrcv对应的字段说明如下：

| 字段名称 | 字段描述 |
| --- | --- |
| pk\_msgres\_sfdfrcv | 主键 |
| msgres\_code | 所属消息源编码 |
| code | 编码 |
| name | 名称 |
| resid | 多语资源号 |
| foldername | 多语文件夹名称 |
| cnvtorimplfullpath | 自定义接收人转换实现类全路径 |
| creator | 创建人 |
| creationtime | 创建时间 |
| modifier | 最后修改人 |
| modifiedtime | 最后修改时间 |

重要属性说明:

msgres\_code 同消息源注册表pub\_msgres\_reg种的该字段，用于过滤自定义接收人的参照。

cnvtorimplfullpath 自定义接收人转换实现类全路径，实现类必须实现nccloud.buzimsg.itf.ISelfDefRcvConvertor接口并实现convert方法，可以参考nc.itf.scmpub.message.convertor.PositionsConvertor类。

## 2.代码实现

ncc的消息分为业务消息、通知消息、审批消息和预警消息，这四个类型的消息分别存在四个表，如下所示：

| 表名 | 描述 |
| --- | --- |
| sm\_msg\_todo | 业务消息记录表 |
| sm\_msg\_notice | 通知消息记录表 |
| sm\_msg\_approve | 审批消息记录表 |
| sm\_msg\_prealert | 预警消息记录表 |
| sm\_msg\_content | 消息发送实体表 |
| sm\_msg\_user | 消息接收人记录表，MESSAGE\_TYPE区分消息类型 todo:业务消息 notice:通知消息 approve：审批消息 prealert：预警消息 |

（1）业务消息又为待办消息，需要代码处理消息状态未已读，如果业务回退，需要处理删除消息；

发消息：IBuziMsgSending.sendBuziMsgReturnReceiver

删除消息：ITodoMessageManageService.setDelete 更改消息为已读/未读：ITodoMessageManageService.setRead

（2）通知消息，无需处理消息状态，只需要关注发送和删除；

发送消息：IBuziMsgSending.sendBuziMsgReturnReceiver

删除消息：ITodoMessageManageService. setDelete

注意点： 1.发送消息，第一个参数，业务消息类型为：todo，通知消息类型为：notice； 2.消息如果需要跳转到具体页面，发送消息最后一个参数可构造URL，示例代码见：nc.bs.pu.m20.maintain.rule.approve.SendMsgToOrderRule; 3.业务中心，通知中心支持按单据号查询消息，在构造参数中必须要输入单据号，平台承接字段：MessageVO.setDetail； 4.消息删除或者状态更改参数参数必须构造成以下格式； JSONObject params=new JSONObject(); JSONArray userIds=new JSONArray(); JSONArray pk\_messages=new JosnArray(); params.put(BuziMsgConst.PK\_MESSAGES,pk\_messages); params.put(BuziMsgConst.USERIDS,userIds);

```java
        UFBoolean bsctype = header.getBsctype(); //单据类型
        param.putAttachment(BuziMsgConst.BILLTYPE, POBillType.PrayBill.getCode());               if(bsctype.booleanValue()) {
            param.putAttachment(BuziMsgConst.APPCODE, AppCodeConst.SCORDER_APPCODE); //不分配请购单权限时，消息跳转会无权限，修改为委外订单appcode

        BuziMsgSender sender = new BuziMsgSender();  //按组织分组构造请购单数据
        PraybillVO[] billVOs = praybillVOGroupByOrg(vo);//表体都是同一个组织时，billVOs没有值，走原逻辑
        if(billVOs.length > 0){
            for(PraybillVO billVO : billVOs){
                sender.send(billVO, new String[]{(String) billVO.getChildrenVO()[0].getAttributeValue(PraybillItemVO.PK_PURCHASEORG)} , param, true);
            }
        }else{
            sender.send(vo, new String[]{(String) vo.getChildrenVO()[0].getAttributeValue(PraybillItemVO.PK_PURCHASEORG)} , param, true);
        }
}
```

```java
javaprivate void updateMsgStatus(TodoMsgVO[] todoMsgVOs) {
    BuziMsgSender sender = new BuziMsgSender();
    try {
        List<TodoMsgVO> msgVOs = new ArrayList<TodoMsgVO>();
        for (int i = 0; i < todoMsgVOs.length; i++) {
            // 来源单据子表主键和消息子表主键相同才进行修改消息中间表
            todoMsgVOs[i].setFmsgstatus(BuziMsgStatusEnum.READ);
            msgVOs.add(todoMsgVOs[i]);
        }
        // 修改消息中间表消息状态，处理为已读
        TodoMsgVO[] updatVOs = NCLocator.getInstance().lookup(IToDoMsgService.class)
                    .updateTodoMsg(msgVOs.toArray(new TodoMsgVO[0]));
        if (updatVOs != null && updatVOs.length > 0) {
            JSONObject params = new JSONObject();
            // 构造消息更新参数
            params = sender.constrMsgParams(updatVOs);
            NCLocator.getInstance().lookup(ITodoMessageManageService.class).setRead(params, Boolean.TRUE, null);
            }
        } catch (BusinessException e) {
            ExceptionUtils.wrappException(e);
        }
    }
```

删除消息参考nc.vo.scmpub.msg.rule.DeleteMessageRule

```java
javapublic void process(T[] vos) {
        if(vos == null || vos.length <=0){
            return;
        }
        try {
            // 构造查询消息表数据
            List<String> billbids = new ArrayList<String>();
            for (AbstractBill vo : vos) {
                CircularlyAccessibleValueObject[] itemVOs = vo.getChildrenVO();
                for (CircularlyAccessibleValueObject itemVO : itemVOs) {
                    String billbid = itemVO.getPrimaryKey();
                    billbids.add(billbid);
                }
            }
            BuziMsgSender sender = new BuziMsgSender();
            //查询消息中间表
            TodoMsgVO[] todoMsgVOs = sender.queryMsgVOs(billbids.toArray(new String[0]), TodoMsgVO.PK_BILLBID);
            if (todoMsgVOs != null && todoMsgVOs.length >0) {
                //构造消息更新参数
                JSONObject params = sender.constrMsgParams(todoMsgVOs);
                sender.deleteTodoMsgVOs(todoMsgVOs);
                for (AbstractBill vo : vos) {
                    sender.deleteMsg(params, vo);
                }
            }
        } catch (BusinessException e) {
            ExceptionUtils.wrappException(e);
        }
    }
     /**
     * 构造更新消息参数
     *
     * @param ids
     * @return *
     */
     public JSONObject constrMsgParams(TodoMsgVO[] todoMsgVOs) {
         JSONObject params = new JSONObject();
         JSONArray userIds = new JSONArray();
         JSONArray pk_messages = new JSONArray();
         for (TodoMsgVO todoMsgVO : todoMsgVOs) {
             String pk_message = todoMsgVO.getPk_message();
             pk_messages.add(pk_message);
             String pk_receiver = todoMsgVO.getPk_receiver();
             userIds.add(pk_receiver);
```
