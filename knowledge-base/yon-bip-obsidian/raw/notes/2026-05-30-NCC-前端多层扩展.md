# 前端分层扩展 | YonBIP高级版

## 前端分层扩展

## 1 介绍

**分层扩展开发方案适用于对已经完成的标准业务节点进行简单的功能扩展，不建议在开发新应用节点页面时使用** 。

扩展开发具备的能力有：

- **多层、可控、有序** 。支持对同一个节点的多次开发，通过配置项等可以控制每层事件的执行顺序。
- **组件的事件和属性拦截** 。比如，拦截按钮点击时的事件 `onButtonClick` ，修改表格创建时的属性 `showCheck` 来控制复选框显隐性，等等。
- **页面级预制钩子** 。比如扩展页面 UI（ `UIExtend` ），页面模板加载完成（ `templateAlreadyLoaded` ），业务数据加载完成（ `dataAlreadyLoaded` ），等等。

开发前需要注意以下几点：

- **目前多层扩展开发仅支持YonBIP3 高级版** ，可以对同一个节点进行多次可维护的有层次扩展，其他版本请参考【 [单层扩展开发](https://nccdev.yonyou.com/article/detail/93) 】。
- 开发人员应对已完成的标准页面进行扩展。
- 标准页面通常指使用前端框架进行开发的界面，注意甄别是否使用该方案。
- 新应用的节点的开发流程请查看社区相关课程，在此不做说明。

## 2 快速开始

下面以 “客户申请单” 为例，介绍如何快速对节点添加扩展开发代码。

### 2.1 节点配置

**NC Cloud 2020.05版本及以后版本** 进行标准节点的扩展开发时，需要在应用注册节点找到该节点，在参数注册中添加参数：sec\_develop，参数值：Y。

![image-20230320105246213](https://community.yonyou.com/datadict/bipbook/assets/image-20230320105246213.CWvtOaaw.png)

### 2.2 前端工程结构

在前端工程脚手架的 `src` 目录下，扩展开发主要分为两个部分，多层控制配置（ `front_extend_config` ）和导出模块配置（ `FrontExtend` 、 `LocalExtend` 、 `NapoleonCakeExtend` ）。目录规范由领域名（ `domainCode` ）、模块名（ `moduleCode` ）、应用编码 （ `appCode` ）、节点名称（ `nodePath` ）、页面编码（ `pageCode` ）组成。

**多层控制配置** ：文件路径 `front_extend_config\${domainCode}\${moduleCode}\${appCode|nodePath}\${pageCode}.json` ，其中 `front_extend_config` 为固定目录名，放在前端工程脚手架 `src` 目录下， `appCode` 和 `nodePath` 可选择其中一种作为目录名，推荐使用 `nodePath` 。

**导出模块配置** ：文件路径 `${layerName(层级名)}/${domainCode}/${moduleCode}/${nodePath}/${pageCode}/index.js` ，放在前端工程脚手架 `src` 目录下。

例如，客户申请单列表页面的页面路径是`../../../../uapbd/customer/custapply/main/index.html#/list` ，应用编码为 `10140CUSTPF` ，页面编码为 `10140CUSTPF_custpflist` ，那么多层控制配置路径为 `\src\front_extend_config\uapbd\customer\custapply\10140CUSTPF_custpflist.json` 或 `\src\front_extend_config\uapbd\customer\10140CUSTPF\10140CUSTPF_custpflist.json` ，导出模块配置路径为 `\src\FrontExtend\uapbd\customer\custapply\10140CUSTPF_custpflist\index.js`

![image-20240118135138727](https://community.yonyou.com/datadict/bipbook/assets/image-20240118135138727.kCNwaaT5.png)

#### 2.2.1 多层控制配置

多层控制配置为一个 json 文件，描述扩展开发文件的加载配置。必须包含 `layers` 配置项， `alias` 是层级别名，用于配置事件关系时指定事件所属层级， `path` 是当前扩展开发的模块入口文件，详情请查看【\[3.1 多层控制配置\](#3.1 多层控制配置)】。

```json
json{
    "layers": [
        {
            "alias": "FrontExtend",
            "path": "FrontExtend\\uapbd\\customer\\custapply\\10140CUSTPF_custpflist\\index.js"
        },
        {
            "alias": "LocalExtend",
            "path": "LocalExtend\\uapbd\\customer\\custapply\\10140CUSTPF_custpflist\\index.js"
        },
        {
            "alias": "NapoleonCakeExtend",
            "path": "NapoleonCakeExtend\\uapbd\\customer\\custapply\\10140CUSTPF_custpflist\\index.js"
        }
    ]
}
```

#### 2.2.2 导出模块配置

每个扩展层级都有一个导出模块配置，即多层控制配置 json 文件 `layers` 中配置的 `path` 入口文件。导出模块配置通过 `export default` 导出一个对象，在对象中添加页面级事件配置 `currentPageDidMount` ，当页面加载完成后，将调用各层级配置的 `currentPageDidMount` 函数。导出模块配置还支持很多其他页面级和组件级函数，详情请查看【\[3.2 导出模块配置\](#3.2 导出模块配置)】。

```js
js// 路径：FrontExtend\uapbd\customer\custapply\10140CUSTPF_custpflist\index.js
import currentPageDidMount from "./events/currentPageDidMount";

export default {
    page: {
        // 页面加载完成
        currentPageDidMount: currentPageDidMount
    }
}
```

在各层级导出模块配置同级目录下创建 `events` 文件夹，并创建 `currentPageDidMount.js` 文件定义 `currentPageDidMount` 函数，在函数中通过 `toast` 组件弹出提示，提示当前层级扩展内容加载成功。注意： `toast` 组件需要添加参数 `mark` ，通过 `toast` 组件弹出提示时，相同 `mark` 值只会显示一个，所以为了更好的看到效果，请将不同层级的 `mark` 设置为不同的值，如 `Front` 、 `Local` 、 `Napoleon` 。

```js
js// 路径：FrontExtend\uapbd\customer\custapply\10140CUSTPF_custpflist\index.js
import {toast} from 'nc-lightapp-front';

export default function () {
    // 提示 扩展内容加载成功！
    toast({color: 'success', content: 'FrontExtend扩展内容加载成功！', mark: 'Front'})
}
```
```js
js// 路径：LocalExtend\uapbd\customer\custapply\10140CUSTPF_custpflist\index.js
import {toast} from 'nc-lightapp-front';

export default function () {
    // 提示 扩展内容加载成功！
    toast({color: 'success', content: 'LocalExtend扩展内容加载成功！', mark: 'Local'})
}
```
```js
js// 路径：NapoleonExtend\uapbd\customer\custapply\10140CUSTPF_custpflist\index.js
import {toast} from 'nc-lightapp-front';

export default function () {
    // 提示 扩展内容加载成功！
    toast({color: 'success', content: 'NapoleonExtend扩展内容加载成功！', mark: 'Napoleon'})
}
```

#### 2.2.3 注意事项

另外，有以下几点注意事项：

- 请按照推荐的方式创建配置文件及编码相关内容，这有利于后续升级和维护。
- 配置文件路径在保证和页面对应的情况下，如果出现配置加载出现404的情况，反馈到节点对应领域分析，一般这种情况是领域没有适配 `billinfo` 。
- 不是所有的事件都有领域默认逻辑 `_DEFAULT` ，有些事件领域干预不了，有些事件领域默认没有干预。
- 推荐的模块导出方式是 `export` 或者 `export default` 。

### 2.3 启动前端工程

启动前，需要在脚手架根目录与 `src` 同级目录下的 `config.json` 文件中，添加 `freeExtendBuildEntryPath` 配置项， `freeExtendBuildEntryPath` 为一个数组，配置各层级导出模块配置的模块入口文件。

```json
json{
    "buildEntryPath": [
        "./src/domain/*/*/*/index.js"
    ],
    "freeExtendBuildEntryPath": [
        "./src/FrontExtend/*/*/*/*/index.js",
        "./src/LocalExtend/*/*/*/*/index.js",
        "./src/NapoleonCakeExtend/*/*/*/*/index.js"
    ],
    "proxy": "http://127.0.0.1:221/",
    "buildWithoutHTML": [
        "uapbd/refer",
        "uap/refer"
    ],
    "devPort": 3006,
    "patch": {
        "path": [],
        "provider": "",
        "department": "",
        "project": "",
        "branch": ""
    },
    "eslint": {
        "path": [],
        "enable": true,
        "autoFix": false
    }
}
```

在终端中输入以下命令并运行

```shell
shellnpm run dev
```

登录Yon BIP高级版系统后，打开 “客户申请单” 节点将看到弹出了三个提示：FrontExtend扩展内容加载成功！、LocalExtend扩展内容加载成功！、NapoleonExtend扩展内容加载成功！

![image-20230320195055911](https://community.yonyou.com/datadict/bipbook/assets/image-20230320195055911.8tBc9BDz.png)

### 2.4 打包&部署

在终端中输入以下命令并运行

```shell
shellnpm run build
```

或

```shell
shellnpm run test
```

`test` 命令编译后包含 `sourcemap` 可在测试阶段使用， `build` 可在生产上线使用。等待编译完成后，在脚手架根目录与 `src` 同级目录下的 `dist` 文件夹中，将会生成 `front_extend_config` 、 `FrontExtend` 、 `LocalExtend` 、 `NapoleonCakeExtend` 四个文件夹，分别对应多层控制配置和各层级导出模块配置。 **将这四个目录拷贝并粘贴到 `home\hotwebs\nccloud\resources` 目录下，重启服务后登录系统** ，打开 “客户申请单” 节点将看到开发环境相同的效果，注意也需要在应用注册节点注册参数 `sec_develop` 。

**layers** ：固定名称，描述这里是用来配置层级入口关系的，其值是个有序数组，用于指定扩展开发层级的加载顺序，也用于规定相同名称钩子的默认执行顺序。 `alias` 是层级别名，用于配置事件关系时指定事件所属层级。 `path` 是当前扩展开发的模块入口文件，理论上这个路径是可以任意定义的。

**page** ：固定名称，描述当前是页面级事件。 `UIExtend` 、 `templateAlreadyLoaded` 等是页面级事件预置固定名称，其值是个有序数组，规定了当前事件的层级执行顺序。 `layer` 标识层级别名， `enable` 标识是某个事件某个层级是否禁止执行， `description` 可以填写当前层级逻辑描述（仅描述，可以为后来扩展层级提供说明指导）。

**component** ：固定名称，描述当前是组件级配置。第一层 `card_head` 、 `adjustprice` 等，这个值是某个组件或者区域的id（这两者一般相同）。 `onButtonClick` 、 `onAfterEvent` 等是组件create时的事件钩子，这里有两种写法。一是对象形式，此时可以指定组件下某个子元素的执行顺序，例如：指定按钮区域 `card_head` 下的保存按钮（编码 `Save` ），指定表格区域 `adjustprice_b` 下的物料列（编码 `pk_material` ）。那么如果没有子元素或者不想指定子元素，此时可以直接使用数组模式，直接指定事件的执行顺序，例如： `onButtonClick: [(等同子元素数组配置)...]` 。通过数组来指定每个层级的执行顺序，规则与页面级钩子相同。值得注意的是组件事件可能领域已经有了默认钩子，这时候就需要管理默认业务事件和扩展事件的关系，为了控制默认业务事件引入了一个 **业务默认（ `_DEFAULT` ）层，固定名称 `_DEFAULT`** ，可以像其他扩展层级一样配置参数和顺序。如果没有配置 `_DEFAULT` ，那么 `_DEFAULT` 在所有扩展层级之前执行。 `layer` 、 `enable` 、 `description` 和页面级配置功能一致。

另外，在多层控制配置文件中是没有事件执行函数的，事件执行函数在导出模块配置文件中。事件执行函数是和多层控制配置一一对应的，如果每一层的每一个事件都在这里配置顺序的话，整个工作量也不小。 为了简化这份工作，就要用到默认的加载执行顺序。规则如下，如果多层控制配置文件里没有指定某个事件的某一层顺序并且其他层也导出了扩展事件函数，那么这个事件会按照各层级加载顺序放到执行数组中。如果所有层级都没有配置，则执行顺序就是各层级加载顺序。例如： `UIExtend` 函数里只配置了 `NapoleonCakeExtend` 层，那么最终的执行顺序是 `NapoleonCakeExtend` => `FrontExtend` => `LocalExtend` 。如果默认是空，那么执行顺序就是 `FrontExtend` => `LocalExtend` => `NapoleonCakeExtend` 。 值得注意的是默认情况下 `_DEFAULT` 层最先执行。 说在最后，无论页面级还是组件级都支持这个规则， **工作量不大的情况下，推荐在配置文件标识清楚，这样后面各层级的扩展也更好理解** 。

## 4 单层改造

在使用多层扩展前，如果原来已经进行了单层扩展开发，需要先确定当前版本支持多层扩展开发方式，并且申请到单层扩展开发的源码，然后再将单层扩展内容作为一个层级改造到多层扩展结构中，实现单层到多层的改造支持。

### 4.1 创建多层控制配置

参照【\[2.2 前端工程结构\](#2.2 前端工程结构)】和【\[3.1 多层控制配置\](#3.1 多层控制配置)】内容，根据规范创建多层控制配置 `front_extend_config` 。例如，在单层扩展中对客户申请单列表和卡片页面进行了扩展，则需要创建客户申请单列表和卡片页面对应的多层扩展配置。

```
src
│
├─extend_uapbd 单层扩展目录
│  └─uapbd
│      └─customer
│          └─custapply
│              ├─10140CUSTPF_custpfcard 客户申请单卡片
│              │      index.js
│              │
│              └─10140CUSTPF_custpflist 客户申请单列表
│                     index.js
│
└─front_extend_config 多层扩展配置
    └─uapbd
        └─customer
            └─custapply
                    10140CUSTPF_custpfcard.json 客户申请单卡片
                    10140CUSTPF_custpflist.json 客户申请单列表
```

不难看出，其实单层扩展和多层扩展的区别就是缺少了多层控制配置，多层级结构也是由此来实现的，单层扩展的入口文件也就相当于导出模块配置，将单层扩展入口文件作为导出模块配置直接添加到多层控制配置中，如果单层扩展文件路径不是 `${layerName(层级名)}/${domainCode}/${moduleCode}/${nodePath}/${pageCode}/index.js` ，则修改成这种格式。

```json
json{
    "layers": [
        {
            "alias": "SingleExtend",
            "path": "extend_uapbd\\uapbd\\customer\\custapply\\10140CUSTPF_custpfcard\\index.js"
        }
    ]
}
```

### 4.2 导出模块配置改造

单层扩展中的入口文件的写法是通过将扩展内容绑定到 `window.NCCExtend` 上面，平台组件获取调用实现的。

```js
jswindow.NCCExtend = {
    UIExtend: UIExtend,
    dataAlreadyLoaded: dataAlreadyLoaded,
    afterBtnClick: afterBtnClick,
    beforeBtnClick: beforeBtnClick,
    currentPageDidMount: currentPageDidMount,
    beforeUnMountEvent: beforeUnMountEvent,
    templateAlreadyLoaded: templateAlreadyLoaded,
    pageRenderExtend: pageRenderExtend,
    beforeMetaUpdate: beforeMetaUpdate,
    onCardTableAfterEvent: onCardTableAfterEvent,
    onCardTableBeforeEvent: onCardTableBeforeEvent,
    onFormAfterEvent: onFormAfterEvent,
    onFormBeforeEvent: onFormBeforeEvent
}
```

那么，改造成导出模块配置就需要以 `export default` 的方式导出扩展内容。平台给单层扩展预留的钩子都是页面级别的，所以这些在改造时都是添加在 `page` 配置中导出的。具体的导出模块配置细节，请参照【\[3.2 导出模块配置\](#3.2 导出模块配置)】。

另外一下几点 **在改造时需要注意** ：

- `afterBtnClick` 、 `beforeBtnClick` 、 `onFormAfterEvent` 、 `onFormBeforeEvent` 、 `onCardTableAfterEvent` 、 `onCardTableBeforeEvent` 、 `onEditTableAfterEvent` 、 `onEditTableBeforeEvent` 这几个钩子已在多层扩展的页面级事件中删除，改为通过组件级事件来实现。
- 由于按钮组件只有 `onButtonClick` 事件，没有点击前后事件，在改造时如果同时扩展了 `afterBtnClick` 和 `beforeBtnClick` ，需要将这两个事件扩展的内容合并为一个。
- `defaultFxConfig` 配置通过在多层控制配置中配置不同层级事件执行顺序实现。
```js
jsexport default {
    page: {
        UIExtend: UIExtend,
        dataAlreadyLoaded: dataAlreadyLoaded,
        currentPageDidMount: currentPageDidMount,
        beforeUnMountEvent: beforeUnMountEvent,
        templateAlreadyLoaded: templateAlreadyLoaded,
        pageRenderExtend: pageRenderExtend,
        beforeMetaUpdate: beforeMetaUpdate
    },
    component: {
        'header-button-area': { // 按钮区域
            onButtonClick: onButtonClick
        },
        'customer': { // 客户基本信息表单区域
            onBeforeEvent: onFormBeforeEvent,
            onAfterEvent: onFormAfterEvent
        },
        'finance': { // 财务信息表格区域
            onBeforeEvent: onCardTableBeforeEvent,
            onAfterEvent: onCardTableAfterEvent
        }
    }
}
```

#### 5.1.2 前端注册组件

21.05及以后版本在使用高阶组件 `form` 、 `table` 、 `cardTable` 、 `editTable` 、 `search` 创建组件前，需要调用 `use` 中的函数先注册组件。扩展开发时在页面级事件 `beforeMetaUpdate` 中调用 `use.form()` 注册表单组件，参数传表单区域的区域编码 `modal_form` ，并设置一个标志位标记已经注册过组件防止多次注册。其他通过扩展开发方式实现额外增加这类组件的渲染也可以通过此方式注册组件，如果没有使用到这些组件则不需要注册组件。

```js
js// 路径：FrontExtend\uapbd\customer\custapply\10140CUSTPF_custpfcard\index.js
import beforeMetaUpdate from "./events/beforeMetaUpdate";

export default {
    page: {
        // 页面模板meta更新前
        beforeMetaUpdate: beforeMetaUpdate,
    }
}
```
```js
js// 路径：FrontExtend\uapbd\customer\custapply\10140CUSTPF_custpfcard\event\beforeMetaUpdate.js
export default function (meta, output, type) {
    // 没有注册组件则通过use.form注册
    !output.frontExtendUsed && output.use.form('modal_form');
    // 标记已经注册组件
    output.frontExtendUsed = true;
}
```

#### 5.1.3 渲染弹窗

扩展页面级事件 `pageRenderExtend` ，通过高阶组件 `modal.createModal()` 创建并返回弹窗。配置弹窗的 `content` 属性，通过 `from.createForm()` 创建表单组件实现弹窗中显示表单。

```js
js// 路径：FrontExtend\uapbd\customer\custapply\10140CUSTPF_custpfcard\index.js
import beforeMetaUpdate from "./events/beforeMetaUpdate";
import pageRenderExtend from "./events/pageRenderExtend";

export default {
    page: {
        // 页面模板meta更新前
        beforeMetaUpdate: beforeMetaUpdate,
        // 页面扩展渲染
        pageRenderExtend: pageRenderExtend,
    }
}
```
```js
js// 路径：FrontExtend\uapbd\customer\custapply\10140CUSTPF_custpfcard\event\pageRenderExtend.js
export default function (output) {
    return output.modal.createModal('demoModal', {
        title: 'Demo弹窗',  // 弹窗头部信息
        content: (
            <div>
                {output.form.createForm('modal_form', {
                    // onAfterEvent: onAfterEvent,  // 编辑后事件
                    // onBeforeEvent: onBeforeEvent,  // 编辑前事件
                })}
            </div>
        ),  // 弹窗内容，可以是字符串或dom
        closeModalEve: () => {
            // 调用modal.close()关闭弹窗
            output.modal.close('demoModal');
        },  // 关闭按钮事件回调
        size: 'lg',  // 关闭按钮事件回调
        noFooter: true  // 是否需要底部按钮，默认有footer，有false，没有true
    })
}
```

#### 5.1.4 按钮事件

扩展头部按钮区域的 `onButtonClick` 函数，在函数中通过调用 `modal.show()` 函数显示弹窗，注意 `show` 函数接收一个弹窗编码，编码是调用 `createModal` 时的第一个参数，此处为 `demoModal` 。可以在多层控制配置中配置当前层级此扩展函数的执行顺序。

```js

import beforeMetaUpdate from "./events/beforeMetaUpdate";
import onButtonClick from "./events/onButtonClick";
import pageRenderExtend from "./events/pageRenderExtend";

export default {
    page: {
        // 页面模板meta更新前
        beforeMetaUpdate: beforeMetaUpdate,
        // 页面扩展渲染
        pageRenderExtend: pageRenderExtend,
    },
    component: {
        // 头部按钮区域编码
        'header-button-area': {
            // button组件的点击事件
            onButtonClick: onButtonClick
        }
    }
}
```
```js
js// 路径：FrontExtend\uapbd\customer\custapply\10140CUSTPF_custpfcard\event\onButtonClick.js
export default function (props, id) {
    switch (id) {
        // 点击的按钮是显示弹窗按钮
        case 'showModal':
            // 调用modal.show()显示弹窗
            props.modal.show('demoModal');
            break;
        default:
            break;
    }
}
```

#### 5.1.5 按钮状态控制

关于按钮状态控制， **目前没有一个很好的控制机制，页面按钮状态都是由领域代码控制，平台无法监控到何时需要更新按钮状态，所以需要自己根据业务结合扩展开发能力来考虑怎么实现** 。如，点击新增按钮进入编辑态时设置按钮不可见，点击保存按钮进入浏览态时设置按钮可见，这些是最简单情况的实现。但是如果按钮事件是标准产品事件且内部存在异步调用，或者点击按钮弹出一个确认窗口，对于确认窗口中的按钮平台并没有提供扩展入口，这些情况对于按钮状态控制就会变得困难，开发人员在开发过程中需要自己根据具体情况具体分析。 **不推荐** 在 `pageRenderExtend` 方法中设置按钮状态，在 React 的生命周期中， `state` 控制的页面渲染，当 `state` 改变时就需要重新调用 `render` 函数，而设置按钮状态就是改变 `state` 操作， `pageRenderExtend` 会被 `render` 函数调用。

```js
js// 路径：FrontExtend\uapbd\customer\custapply\10140CUSTPF_custpfcard\event\onButtonClick.js
// 对于最简单情况的实现，复杂情况如存在异步、弹窗确认等需要具体分析
export default function (props, id) {
    switch (id) {
        case 'showModal':
            props.modal.show('demoModal');
            break;
        case 'Add':
        case 'Edit':
        case 'Copy':
            props.button.setButtonVisible('showModal', false);
            break;
        case 'Save':
        case 'Cancel':
            props.button.setButtonVisible('showModal', true);
            break;
        default:
            break;
    }
}
```

另外，当应用注册添加了按钮但是页面未显示，请在页面级事件 `templateAlreadyLoaded` （页面模板加载完成）中进行断点调试，确认按钮是否存在。如果不存在请咨询领域是否有权限要求；如果存在但未显示，可能是业务逻辑隐藏了按钮，确认隐藏逻辑后，在合适的扩展事件位置调用 `button` 组件API显示按钮。

### 5.2 参照过滤

#### 5.2.1 页面初始化过滤

通过扩展页面级事件 `templateAlreadyLoaded` 实现页面初始化时对字段添加参照过滤， `templateAlreadyLoaded` 会在页面初始化向后端请求并接收到模板信息后调用，遍历模板字段，对需要添加参照过滤的字段设置 `queryCondition` 属性， `queryCondition` 属性接收一个函数，在函数中组装参数并返回。 **存在一种情况不适用页面初始化过滤需要注意** ，当表格区域需要根据当前行的某个字段进行参照过滤时， `templateAlreadyLoaded` 中无法获取到当前编辑的行，则无法实现参照过滤，需要使用编辑前事件过滤的方式实现，请查看下一小节内容。

例如，对 “客户申请单” 卡片页面客户基本信息区域的上级客户字段添加参照过滤，当客户基本分类字段选中了数据时，根据选择的客户基本分类过滤上级客户字段，只显示选中客户基本分类下的客户，否则不过滤。

```js
js// 路径：FrontExtend\uapbd\customer\custapply\10140CUSTPF_custpfcard\index.js
import templateAlreadyLoaded from "./events/templateAlreadyLoaded";

export default {
    page: {
        // 页面模板加载完成
        templateAlreadyLoaded: templateAlreadyLoaded
    }
}
```
```js
js// 路径：FrontExtend\uapbd\customer\custapply\10140CUSTPF_custpfcard\event\templateAlreadyLoaded.js
export default function (params, output) {
    // 结构参数
    let {button, template, context} = params;
    // 从模板中获取上级客户字段信息并设置queryCondition
    template['customer'].items.find(item => item.attrcode == 'pk_customer_main').queryCondition = () => {
        // 获取客户基本分类
        let pk_custclass = output.form.getFormItemsValue('customer', 'pk_custclass');
        // 返回参数
        return {
            // 客户基本分类主键
            pk_custclass: (pk_custclass && pk_custclass.value) ? pk_custclass.value : '',
            // 树型参照自定义过滤类，此处过滤客户参照的左树
            TreeRefActionExt: 'nccloud.web.uapbd.ref.customer.builder.CustomerTreeSqlBuilder'
        }
    }
}
```

#### 5.2.2 编辑前事件过滤

编辑前事件实现参照过滤的方式需要扩展字段对应区域的编辑前事件，在编辑前事件中通过调用 `meta.getMeta()` 获取页面模板信息，遍历模板字段，对需要添加参照过滤的字段设置 `queryCondition` 属性， `queryCondition` 属性接收一个函数，在函数中组装参数并返回，最后通过调用 `meta.setMeta()` 将修改后的页面模板信息重新进行设置。

例如，使用编辑前事件过滤实现页面初始化过滤示例相同的效果。“客户申请单” 卡片页面客户基本信息区域的区域编码为 `customer` ，扩展该区域的编辑前事件。在编辑前事件中可以获取到当前区域的数据 `data` ，也可以通过调用高阶组件API获取不同区域的数据。如果是对表格扩展编辑前事件，在编辑前事件中还可以获取到当前编辑行下表及行数据，方便进行数据获取。

```js
js// 路径：FrontExtend\uapbd\customer\custapply\10140CUSTPF_custpfcard\index.js
import onFormBeforeEvent from "./events/onFormBeforeEvent";

export default {
    component: {
        'customer': {
            // form编辑前事件
            onBeforeEvent: onFormBeforeEvent
        }
    }
}
```
```js
js// 路径：FrontExtend\uapbd\customer\custapply\10140CUSTPF_custpfcard\event\onFormBeforeEvent.js
export default function (props, moduleId, key, value, data) {
    if (key == 'pk_customer_main') {
        // 通过meta.getMeta获取模板
        let meta = props.meta.getMeta();
        // 获取客户基本分类
        let pk_custclass = data['pk_custclass'];
        // 获取客户基本分类， 也可以通过这种方式获取
        // let pk_custclass = output.form.getFormItemsValue('customer', 'pk_custclass');
        // 从模板中获取上级客户字段信息并设置queryCondition
        meta['customer'].items.find(item => item.attrcode == 'pk_customer_main').queryCondition = () => {
            // 返回参数
            return {
                // 客户基本分类主键
                pk_custclass: (pk_custclass && pk_custclass.value) ? pk_custclass.value : '',
                // 树形参照自定义过滤类，此处过滤客户参照的左树
                TreeRefActionExt: 'nccloud.web.uapbd.ref.customer.builder.CustomerTreeSqlBuilder'
            }
        }
        // 重新设置模板
        props.meta.setMeta(meta);
    }
    return true;
}
```

#### 5.2.3 自定义参照过滤

在对 “客户申请单” 节点卡片页面上级客户字段添加参照过滤的示例中，可以看到在 `queryCondition` 函数中返回了 `TreeRefActionExt` 参数，这个参数配置树型参照的自定义参照过滤类的全限定类名。不同类型的参照配置不同的参数，自定义参照过滤配置有：

- TreeRefActionExt：树型参照后端自定义过滤扩展属性
- GridRefActionExt：表型参照后端自定义过滤扩展属性

自定义参照过滤类编写在后端 `src` 目录下 `client` 中，需要实现 `nccloud.framework.web.processor.IRefSqlBuilder` 接口，并重写 `getExtraSql` 方法返回过滤 `sql` ，重写 `getExtraSqlParameter` 方法返回条件中的参数值。

```java
javapackage nccloud.web.uapbd.ref.customer.builder;

import nccloud.framework.web.processor.IRefSqlBuilder;
import nccloud.framework.web.processor.refgrid.RefQueryInfo;
import nccloud.framework.web.ui.meta.RefMeta;
import nccloud.pubitf.platform.db.SqlParameterCollection;

public class CustomerTreeSqlBuilder implements IRefSqlBuilder {

    @Override
    public String getExtraSql(RefQueryInfo refQueryInfo, RefMeta refMeta) {
        // 获取前端queryCondition返回的pk_custclass参数
        String pk_custclass = refQueryInfo.getQueryCondition().get("pk_custclass");
        if (pk_custclass != null && pk_custclass.length() > 0) {
            // 存在pk_custclass参数返回过滤条件sql
            return " and pk_custclass = ? ";
        }
        return "";
    }

    @Override
    public SqlParameterCollection getExtraSqlParameter(RefQueryInfo refQueryInfo, RefMeta refMeta) {
        // 创建sql参数对象
        SqlParameterCollection sqlParam = new SqlParameterCollection();
        // 获取前端queryCondition返回的pk_custclass参数
        String pk_custclass = refQueryInfo.getQueryCondition().get("pk_custclass");
        if (pk_custclass != null && pk_custclass.length() > 0) {
            // 存在pk_custclass参数添加值到sql参数中
            sqlParam.addVarChar(pk_custclass);
        }
        return sqlParam;
    }

    @Override
    public String getOrderSql(RefQueryInfo refQueryInfo, RefMeta refMeta) {
        return null;
    }
}
```

另外，部分标准参照中存在预留过滤参数，可以在前端 `queryCondition` 中直接返回预留参数值进行过滤，预留过滤参数需要自行分析，可以通过参照后端处理类代码是否从 `queryCondition` 获取参数值并添加到过滤条件 `sql` 中确定。例如，物料参照左树 `nccloud.web.uapbd.ref.pub.MaterialClassTreeRef` ，通过 `paramRefQueryInfo.getQueryCondition()` 获取前端 `queryCondition` 返回值，并判断是否存在 `pk_marbasclass` 参数值，存在则设置到过滤条件 `sql` 中。

```java
java// 只展示部分代码片段
package nccloud.web.uapbd.ref.pub;

public class MaterialClassTreeRef extends DefaultTreeRefAction {
    public String getExtraSql(RefQueryInfo paramRefQueryInfo, RefMeta paramRefMeta) {
        String sql = "";
        // 获取前端queryCondition返回值
        Map<String, String> map = paramRefQueryInfo.getQueryCondition();
        // 获取pk_marbasclass参数值
        if (map != null && map.get("pk_marbasclass") != null && !"".equals(((String)map.get("pk_marbasclass")).trim())) {
            // 存在pk_marbasclass参数值添加到过滤条件sql中
            sql = sql + " and pk_marbasclass = '" + ((String)map.get("pk_marbasclass")).trim() + "' ";
        }
        return sql;
    }
}
```

### 5.3 赋值联动

实现赋值联动扩展字段对应区域的编辑后事件，通常是在一个字段编辑后，给其他字段赋值。

例如，“客户申请单” 卡片页面客户基本信息区域的客户基本分类字段值清空后，同时清空上级客户字段值。扩展区域编码为 `customer` 的编辑后事件，在编辑后事件中判断是清空客户基本分类字段值时，通过调用 `form.setFormItemsValue` 设置上级客户字段值为空。

```js
js// 路径：FrontExtend\uapbd\customer\custapply\10140CUSTPF_custpfcard\index.js
import onFormBeforeEvent from "./events/onFormAfterEvent";

export default {
    component: {
        'customer': {
            // form编辑后事件
            onAfterEvent: onFormAfterEvent
        }
    }
}
```
```js
js// 路径：FrontExtend\uapbd\customer\custapply\10140CUSTPF_custpfcard\event\onFormAfterEvent.js
export default function (props, moduleId, key, value, oldValue) {
    // 编辑的是客户基本分类
    if (key == 'pk_custclass') {
        // 清空客户基本分类同时清空上级客户
        if (!(value && value.refpk)) {
            props.form.setFormItemsValue('customer', {pk_customer_main: {value: '', display: ''}});
        }
    }
}
```
