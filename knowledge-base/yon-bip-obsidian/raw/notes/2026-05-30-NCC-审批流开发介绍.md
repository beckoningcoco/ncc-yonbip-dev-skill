### 2.1 单据类型和交易类型基本概念 [​](https://community.yonyou.com/datadict/bipbook/chapter2302/chapter2302.html#_2-1-%E5%8D%95%E6%8D%AE%E7%B1%BB%E5%9E%8B%E5%92%8C%E4%BA%A4%E6%98%93%E7%B1%BB%E5%9E%8B%E5%9F%BA%E6%9C%AC%E6%A6%82%E5%BF%B5)

从 NC6.x 开始，产品中的单据类型用交易代替，业务类型用交易类型代替。交易下面可再细分交易类型。

交易一般指一个完整的业务交易场景，也对应为一个有明显过程边界的功能，例如入仓单、采购发票、应收单、应付单。产品中一个交易用一种单据类型来代表，例如采购入库单、销售出库单。所以交易通常是出厂时预置或二次开发提供的，用户不能随意编辑。

交易类型是一个交易的分类。扩展交易类型只是在交易支撑的功能模型基础之上再描述一些规则、模板、流程的差异，所以是用户定义的。

## 3 单据动作脚本 [​](https://community.yonyou.com/datadict/bipbook/chapter2302/chapter2302.html#_3-%E5%8D%95%E6%8D%AE%E5%8A%A8%E4%BD%9C%E8%84%9A%E6%9C%AC)

注册单据动作脚本，在单据类型下，若有对下游的推单动作则注册成推式动作，后面三个选项看情况勾选。单据类型和交易类型都有对应的动作配置。特别要注意的是：当在交易类型下修改对应动作脚本后，到pub\_billaction表查询对应的记录pk\_billtype是否变更为对应的交易类型编码，如果改变，需要手动改掉，将pk\_billtype改为单据类型对应的编码，否则后续执行脚本时会找不到对应的处理类。

![image6.png](https://community.yonyou.com/datadict/bipbook/assets/image5.TFTkDZn1.png)

在完成单据动作脚本的注册后，需要在后端创建对应的动作脚本。其中类的报名必须是nc.bs.pub.action ；类名必须是N开头\_单据类型编码\_动作名称.java ；这个类需要继承nc.bs.pub.compiler.AbstractCompile2 。 要注意的是，有五个是必须注册且规定动作名称的单据动作，分别是SAVEBASE保存，SAVE提交，UNSAVE收回，APPROVE审批，UNAPPROVE 弃审。动作名称要注意必须是大写。

![image7.png](https://community.yonyou.com/datadict/bipbook/assets/image7.CwPLnUV3.png)

单据的审批状态只有五种，一般由流程平台控制，无需更改。

单据状态是各领域根据业务要求自己定义的，与流程平台无关，同时也需涵盖了审批流的状态，可根据操作去维护

## 4 指派 [​](https://community.yonyou.com/datadict/bipbook/chapter2302/chapter2302.html#_4-%E6%8C%87%E6%B4%BE)

审批流当中的某个审批环节可以指定某个业务角色参与，类似于这种情况下，上个环节到本环节的时候就会存在多个参与者存在。在这种情况下，指定下个环节的具体人员的过程就是指派的过程。通常情况下业务单据的研发人员，只需要处理提交环节的指派即可；提交之后的审批环节都是在审批中心节点进行操作的。

![image13.png](https://community.yonyou.com/datadict/bipbook/assets/image13.BuoBaeHn.png)

指派操作的研发过程当中有如下几个注意点：

1.  首先如果单据如果存在指派的话，需要两次发送网络请求——调用两次提交的Action。

2.  存在指派时，第一次请求平台API会返回相应地指派信息：下个环节包含具体哪些人员等等。

3.  前端界面显示用户详情信息后，客户选择具体的人员，点击确定，做二次提交操作。

指派操作的后端逻辑：

1.  第一次提交操作的时候，可以加下判定，是否存在指派信息，如果有的话，后台需要直接将指派信息返回到前端；指派信息在后端是一个Map的形式存在的。

java

```
Object[] exeScriptPFlow = service.exeScriptPFlow(context);
if (content == null) {
    Object obj = exeScriptPFlow[0];
    // 下面是判定如果有指派信息的话，那么将指派信息传到前端，然后二次提交
    if (obj instanceof Map && ((Map<?, ?>) obj).get(PfUtilBaseTools.PARAM_WORKFLOWTYPE) != null
        && (PfUtilBaseTools.PARAM_APPROVEFLOW
            .equals(String.valueOf(((Map<?, ?>) obj).get(PfUtilBaseTools.PARAM_WORKFLOWTYPE)))
            || PfUtilBaseTools.PARAM_WORKFLOW.equals(
                String.valueOf(((Map<?, ?>) obj).get(PfUtilBaseTools.PARAM_WORKFLOWTYPE))))) {
        Map<String, Object> map = (Map<String, Object>) obj;
        if (map.get(PfUtilBaseTools.PARAM_WORKFLOWTYPE) != null) {
            return map;
        }
    } else if (obj instanceof PfProcessBatchRetObject) {
        // 后端组装一下错误信息
        PfProcessBatchRetObject error = (PfProcessBatchRetObject) obj;
        if(error.getExceptionInfo().hasError()) {
            HashMap<Integer, String> errMap = error.getExceptionInfo().getErrorMessageMap();
            // return error message
            for(Entry<Integer, String> entry : errMap.entrySet()) {
                return entry.getValue();
            }
        }
    }
}
```

2.  第二次提交操作的时候，前端会传入客户选定的审核人等信息，此处需要将相关信息提供到审批流提供的接口当中。

java

```
CloudPFlowContext context = new CloudPFlowContext();
HashMap<Object, Object> paraMap = new HashMap<Object, Object>();
JSONObject contents = JSONObject.parseObject(read);

if (content != null) {	// 指派信息，content其实是com.alibaba.fastjson.JSONObject，由前端提供

}
```

指派的前端逻辑：

1.  当调用了提交的操作之后返回的信息包含指派信息时，需要将指派信息保存到前端，并且显示平台的ApprovalTrans组件——展示指派信息，代码如下：

javascript

```
ajax({
	url: URL.commit,
	data: data,
	success: (res) => {
		if (
			res.data &&
			res.data.workflow &&
			(res.data.workflow == 'approveflow' || res.data.workflow == 'workflow')
		) {
			this.setState({
				compositedata: res.data,//指派信息
				compositedisplay: true
			});
			return;
		}
	}
});
createApprovalTrans=()=>{
	return (
		<div>
			{this.state.compositedisplay && (
				<ApprovalTrans
					title={'指派'} /* 国际化处理： 指派*/
					data={this.state.compositedata} // 指派信息
					display={this.state.compositedisplay}
					getResult={this.getAssginUsedr.bind(this)} //确定的回调操作
					cancel={() => {
						this.setState({ compositedisplay: false });
					}}
				/>
			)}
		</div>
	);
}
```
