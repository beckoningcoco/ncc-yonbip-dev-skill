# 导入导出功能 | YonBIP高级版

## 导入导出功能

## 1\. 导入导出功能总体设计

数据导入功能的展现形式是在应用界面上通过注册导入导出按钮的方式进行，用户可以根据单据模板动态选择需要的导入导出项。同时系统支持多种格式文件（xls,xlsx,csv）的导入，由于导入的翻译和保存接口都是在后台调用，系统会将导入过程中出现的问题回写到导入文件中，用户可以通过导入日志界面很方便的对本次导入的状态进行查看。

![](https://community.yonyou.com/datadict/bipbook/assets/001.Cjzb0ml0.jpg)

#### 1.1 导入导出设计思路

1、配置文件：所有需要数据导入导出功能的档案必须要在home中注册对应档案的配置文件。

2、确定导出项：通过输出的格式文件规范导入数据的格式，以便于进行数据的转换（即通信协议是底层程序已经确定的）。通过前端传入的appcode和pagecode及配置文件record中属性arecord找到对应单据模板区域需要的导出项。

3、导入数据处理：将格式文件中的数据解析后传到数据导入器中，数据导入器根据其传入的参数以及配置文件找到对应的单据模板，并通过单据模板对解析后导入数据进行翻译，完成了和业务逻辑的衔接（这一步操作实际上模拟的是界面录入数据的过程）。循环对导入的所有数据完成这个操作。如果档案不允许导入则直接抛出异常给用户提示。

4、 错误信息提示和日志功能：数据导入工具提供导入时的错误信息提示和日志功能，能够将调用各个档案（或单据）的保存方法时产生的业务异常信息回写到导入的excel文件中，同时提供导入日志查询节点对回写之后的导入文件进行下载查看，因为只是对错误信息进行回写，并不会改变文件格式，可以很方便的修改之后重新导入。

实现导出按钮点击事件：在按钮点击事件方法中添加。

```javascript
javascriptcase '导出按钮编码': // 导出按钮点击事件

    let pks = [];
    /**
     * 1.从页面上获取导出单据主键集合。
     * 如果是表格，获取勾选数据，从中解析出主键。
     * 如果是卡片页面，只能导出一个单据。获取主键pk, pks=[pk]
     * 具体用哪个方法获取，根据使用的组件，查看前端API文档。

    // 2.更新state中pks,弹出导出弹框
    this.setState({pks: pks},()=>{
        this.props.modal.show('exportFileModal'); // 'exportFileModal'为固定写法
    });
    break;
```

如果是不导出数据，只导出模板时直接打开导出弹框即可：

```javascript
javascriptcase '导出按钮编码': // 导出按钮点击事件
    this.props.modal.show('exportFileModal'); // 'exportFileModal'为固定写法

```

### 2.3 后端适配

#### 2.3.1 后端 XML 配置文件

功能说明：配置文件实现导入档案主子vo关系配置，导入字段顺序设置，动态为参照字段设置参数等功能。

**生成方法：** 1.导入导出的xml文件配置头文件都是相同的，不同的就是record标签里关于实体的配置和field标签里关于字段的配置。可以复制已有的xml配置文件，根据需求简单调整即可。 2.如果有重量端环境，可以在发布元数据后在重量端“插件开发向导”节点生成和外部交换平台类似的xml，部分内容还需要手动修改。 3.如果用低代码平台做的节点，导入导出相关的代码和配置文件都会自动生成。

**XML 配置文件命名：** 单据类型编码.xml **位置：** 统一放到 "NCC Home\\resources\\excel\\billdefine\\模块编码" 目录下。（对应前端传对的模块名）

注意：这里的单据类型编码和模块编码跟前端渲染 <ExcelOutput />组件时传的单据类型编码和模块编码保持保持一致。

**配置文件内容说明：** 首先是表头record，以客户应收单为例：

```xml
xml<record displayName="客户应收单" displayNameResid="excel_araprecbillhead_00001" metadataid="ca162462-078f-46e8-af2d-05c4580c8af7" stabletag="header" tablename="recbill" tabletag="recbill" tabletype="head" typeName="表头" areaname="head">
```

| 属性 | 说明 |
| --- | --- |
| displayName | record的名称，无实际意义。 |
| displayNameResid | 多语id，保证唯一，规范：excel\_ + 单据类型+billhead(标识表头)+\_0001(编码可不写)。将对应值统一添加到多语文件中。多语文件位置：nchome\\langlib\\riart\_nccloud\_sim\_langresLevel1.jar\\lang\\simpchn\\excel\_import\\excelrecord.properties；（如果不需要做多语，displayNameResid 直接填写名称，跟 displayName保持一致即可） |
| metadataid | 对应vo的元数据id 可以从md\_class表中进行查询 |
| stabletag | 值为"header"，是固定写法 |
| tablename | 对应导出excel表头编码。 |
| tabletype | record下存储的是表头信息时候 ，设为tabletype = “head” ，是表体信息时tabletype = “body” |
| typeName | 标识record类型，无实际意义。一般写"表头" |
| areaname | 区域编码，对应单据模板区域。导入导出时会顺序遍历该record下的所有field字段，根据filed字段中的name属性值查询模板区域下字段值得是否可见、必输、字段类型等信息用于导出字段展示及导入字段的翻译校验的功能。 |

**Field字段说明：**

| 属性 | 说明 |
| --- | --- |
| matchtag | 可以和name相同，但是需要在改配置文件下保证field中的唯一性 |
| name | 对应vo中字段编码，保证和模板中字段编码保持相同 |
| type | 字段类型该字段在vo中的类型 String, UFDate...，其中标识关联关系的filed type类型为 `<type>ref(item)</type>` 实例如下：refitem对应表体record中的元数据metadataid； |
| variablename | 如果一个字段，作为其他字段的翻译条件，则这个字段需要设置variablename属性。如：部门字段翻译时，组织字段作为条件传进去，这时组织字段需要设置variablename属性。当设置 pk\_org\_x 时，字段翻译后pk记录在上下文中，pk\_org\_x 相当于缓存的key，后续其他地方翻译时从内存中通过key获取该值，需要保证这个key在配置文件内唯一。 |
| translatorProperty | 走平台参照翻译（默认翻译方法），并且需要向参照中传入条件时需要配置 translatorProperty 属性。如：部门翻译时需要传组织，应该配置为： `<translatorProperty>pk_org_x=pk_org</translatorProperty>` 其中pk\_org\_x跟variablename 属性配置的key对应；pk\_org是部门参照接收的条件字段编码。这样，导入部门时会从上下文中获取key为pk\_org\_x的值，作为组织信息pk\_org完成属性注入，然后再进行翻译。如果translatorProperty配置多个属性，则用分号 ";" 隔开。 |
| translator | 需要通过自定义翻译器翻译的字段，配置translator属性。值为翻译器注册表excel\_translator中翻译器主键。如： `<translator>1001Z010000000012DEF</translator>` |
