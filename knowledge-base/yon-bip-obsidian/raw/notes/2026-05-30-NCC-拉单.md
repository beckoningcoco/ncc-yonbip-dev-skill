# 拉单 | YonBIP高级版

## 拉单 [​](https://community.yonyou.com/datadict/bipbook/chapter2305/chapter2305.html#%E6%8B%89%E5%8D%95)

---

拉单操作是企业管理的当中比较常见的一种单据生成方式。通常情况下，企业管理过程当中会存在一部分重复数据，为了展示或者查看单据的方便会在流程当中一直流转下去，例如销售合同->销售订单。重复录入数据过程对于客户来说容易出错并且太过繁琐，通过拉单的方式能够将上游单据的相关信息带入到下游当中。拉单承接了上文当中的业务流，也需要有单据转换规则以及业务流程定义。

拉单的过程按照上述的描述，应当是参照上游的单据，展示相关信息给用户；程序将用户选择的上游单据转换成当前单据。因此拉单开发需要前后端相互配合，前端需要有上有单据信息展示界面，同时后台包含查询上游单据数据并返回前端以及将上游单据转换成当前单据的逻辑。下面我们将分前后端两个部分讲述拉单的开发过程。

## 4.1 拉单前端界面开发 [​](https://community.yonyou.com/datadict/bipbook/chapter2305/chapter2305.html#_4-1-%E6%8B%89%E5%8D%95%E5%89%8D%E7%AB%AF%E7%95%8C%E9%9D%A2%E5%BC%80%E5%8F%91)

拉单前端界面，平台提供了相关的高阶组件。该高阶组件需要配合单据模板来使用，因此我们前端界面的开发，我们需要有配置单据模板、前端界面编写以及相应的后台请求——请求查询上游数据以及将上游数据转换成当前单据。

### 4.1.1 单据模板配置 [​](https://community.yonyou.com/datadict/bipbook/chapter2305/chapter2305.html#_4-1-1-%E5%8D%95%E6%8D%AE%E6%A8%A1%E6%9D%BF%E9%85%8D%E7%BD%AE)

拉单的单据模板配置也是在应用注册节点进行，同普通的单据模板配置没有太大的差距，只不过是不需要做按钮注册。因为我们需要参照上游的单据，并且拉单界面用于展示上游的单据，因此单据模板配置当中需要引用上游单据的元数据配置模板。

### 4.1.2 前端代码编写 [​](https://community.yonyou.com/datadict/bipbook/chapter2305/chapter2305.html#_4-1-2-%E5%89%8D%E7%AB%AF%E4%BB%A3%E7%A0%81%E7%BC%96%E5%86%99)

前端代码主要是拉单界面的渲染工作。如果是采用了平台推荐提供的单页应用方式的话，可以新做一个拉单页面然后做单页应用的跳转——当点击拉单按钮的时候跳转到拉单页面。

前端界面渲染的逻辑，主要是通过createTransferTable这个平台提供的API渲染界面，相关代码如下所示：

javascript

```
const { transferTable } = this.props;
const { createTransferTable } = transferTable;
return (
    <div>
    	// ... 查询区的渲染等操作
    	{createTransferTable({
                headTableId: 'head', //表格组件id
                bodyTableId: 'body', //子表模板id
                fullTableId: '', //主子拉平模板id
                transferBtnText: '', //转单按钮显示文字/* 国际化处理： 生成下游单据*/
                onTransferBtnClick: (ids) => {
             // 此处跳转到卡片界面之后，可以通过相关参数来发送后台请求将上游单据数据转换成当前单据并赋值到卡片界面上
             this.props.pushTo('/card', {
             status: 'transfer',
             type: 'transfer',
             srcBilltype: '',
             dataSource: dataSource,
             pagecode: this.props.getUrlParam('dest_tradetype')
    		});
    	},
        onChangeViewClick: this.changeViewType
    })}
    </div>
)
```

在渲染了拉单界面的时候，通常都会有一个查询区，用于查询上游单据。当用户点击查询按钮的时候会发送请求查询上游单据信息，返回数据的时候我们需要将数据赋值到界面上。相关API如下：this.props.transferTable.setTransferTableValue(id, childCode, newData, rowId, childId)，各参数含义如下——id:主表id；childCode:子表id；newData:放入表格的数据；rowId:主表主键字段，如果是全部多来源转单需要传数组，其他页签主表主键放到数组中childId:子表主键字段，如果是全部多来源转单需要传数组，其他页签子表主键放到数组中。

目前平台提供了三种转单的高阶组件：

1.  createTransferTable，主要是单来源转单。即上游只有一个单据来源的转单操作。
2.  createTransferList，带左侧缩略图的转单界面。
3.  createMultiTransferTable，多来源转单界面，亦即有多个上游来源的数据混合展示在界面上。

## 4.2 拉单后台逻辑开发 [​](https://community.yonyou.com/datadict/bipbook/chapter2305/chapter2305.html#_4-2-%E6%8B%89%E5%8D%95%E5%90%8E%E5%8F%B0%E9%80%BB%E8%BE%91%E5%BC%80%E5%8F%91)

拉单后台逻辑包含了上游单据数据的查询以及将上游单据数据转成当前单据两个部分。其中第一个部分同普通的单据查询操作比较类似。将上游单据转成当前单据的逻辑是通过单据转换规则来实现的，也就是上一个章节——《业务流开发介绍》——当中描述单据转换规则，我们调用平台接口实现单据的转换操作。

首先是上游单据的查询类，此处同普通的单据列表查询接口相同，接受查询区传到后端的数据，然后返回列表数据。

java

```

IJson json = JsonFactory.create();
QueryTreeFormatVO info = json.fromJson(read, QueryTreeFormatVO.class);
INCCloudQueryService schemaService = ServiceLocator.find(INCCloudQueryService.class);
IQueryScheme schema = schemaService.convertCondition(info);
String where = schema.getWhereSQLOnly();

// 调用转换工具类将VO转换成前端展现格式
```

其次是用户在拉单界面选择了相关的单据之后，发送一次请求将上游单据转换成当前单据，调用nc.itf.uap.pf.IPfExchangeService类的相关接口走单据转换规则。将转换之后的单据返回前端，展示在当前单据的卡片界面上，供用户做调整操作。
