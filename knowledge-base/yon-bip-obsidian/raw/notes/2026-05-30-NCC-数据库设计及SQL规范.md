# 数据库设计规范及SQL编程规范 | YonBIP高级版

## 数据库设计规范及SQL编程规范

## 1 数据库设计强制规范

1. 模块名称规则表，表名前缀，例如：系统管理 sm、组织 org、基本档案 bd
2. 表名称长度，不超过 30 个字符（oracle 限制）
3. 业务中主表的命名规则：系统规则名\_名称或系统规则名\_名称\_h
4. 业务中子表的命名规则：子表：主表\_b，孙表：系统规则名\_英文名称\_bs
5. 派生表命名规则，派生规则作为表名后缀如：按集团分表\_xx，历史数据表\_h，如：01集团凭证明细历史数据表gl\_detail\_01\_h
6. 临时表，tmp\_系统规则名\_名称，例如 tmp\_bd\_psndoc

## 3 SQL 书写规范

### 3.1 SQL书写风格

1. 【强制】SQL 语句全部使用小写；
2. 【强制】引用字符时用单引号，如：update testable set idcol=’abcd’；
3. 【强制】连接符或运算符 or、in、and、＝、<=、>=， +，- 等前后加上一个空格；
4. 【强制】严禁使用 select \* …… 形式的语句，必须指出 select 的具体字段，即 select col1,col2,…from tablea where ….；
5. 【强制】严禁使用 insert into table values(?,?,?)，必须指出具体要赋值的字段，即 insert into tablea (col1, col2,…) values(?,?,…)；
6. 【强制】避免隐含的类型转换。例如在 where 子句中 numeric 型和 int 型的列的比较或相加。在 pg 体系的数据库（polardb、金仓、gbase）中 varchar 和 char 之间也会出现隐式转换；
7. 【强制】在子查询中前后必须加上括号，select col1, col2 from tablea where col3 in (select col4 from tableb where col4>0)；
8. 【推荐】SQL语句包含多表连接时，建议对每个表命名别名，对每个字段的使用都要带上表别名，即 select a.col1, a.col2, b.col3 from tablea a, tableb b where a.col4=b.col5；

### 3.2 注意事项

一、存储过程：

1. 【强制】为了减少移植的困难，不要使用存储过程。如必须使用，需审批；

二、触发器：

1. 【强制】尽可能的不使用触发器，尽量通过程序实现，有利于多数据库移植；
2. 【强制】不允许动态创建触发器；

三、视图

1. 【强制】使用静态视图，不允许动态创建视图；
2. 【强制】视图中不能包含 order by 语法，影响效率；
3. 【强制】尽量不要出现视图嵌套视图的情况；

四、索引

1. 【强制】为了保证索引具有提高效率的作用，估计记录数不大于 100 的表一律不允许建索引；
2. 【强制】只用于 distinct 或 group by 子句引用的列，不用建立索引；
3. 【强制】对于数据值变化不大的列，比如只有 yes 和 no，那么不要建索引，否则效率降低；
4. 【强制】不允许动态创建索引；
5. 【强制】当创建复合索引时，复合索引的列不超过 7 列，复合索引的列的总长度不能超过 128 字节；
6. 【强制】在复合索引中，所有的列必须来自同一个表中，不能跨表建立复合列；
7. 【强制】复合索引为多列，同复合主键一样将变化显著的列放到复合索引的首位；
8. 【强制】规范索引的管理，设计表的时候需要将表的主要查询场景整理形成相应文档，然后根据相应场景设计索引；
9. 【强制】某些字段（如 dr、ts 等）设计时不可见。在这些字段上建索引，需要申报审批；
10. 【推荐】在复合索引中，列的排列顺序是非常重要的。例如在（col1，col2）上的索引与在（col2，col1）上的索引是不相同的，因为两个索引的列的顺序不同；

五、表及外键

1. 【强制】表的具体设计规范原则。以第三范式的标准严格设计，然后再考虑开发与运行的效率进行反范式规范的冗余设计；
2. 【强制】不允许创建外键；
3. 【强制】每个表必须有主键（primary key），不推荐复合主键；
4. 【强制】复合主键必须遵守以下规则：越能成为唯一标识的字段（同值重复少）就越放在前面；
5. 【强制】对于已经存在的表，如果要增加字段，建议一般使用可空字段，如果是非空字段，必须给出默认值；
6. 【推荐】一个表的字段在引用另外一个表带编码、名称的字段时，只需引用该表主键字段；

六、函数规范

1. 【强制】不允许使用自定义函数；

## 4 高效 SQL

**强制使用：**

1. 【强制】避免在索引字段上使用表达式或函数；
2. 【强制】禁止使用 not in 语句，建议用 not exist；
3. 【强制】使用 not in 的时候，not in 两边的字段都要有 not null 条件；
4. 【强制】子查询与 or 连用时，改成 union；
5. 【强制】不要在子查询中引用主查询的条件；
6. 【强制】update 的时候注意写 where 条件；
7. 【强制】杜绝不合理的 SQL（避免笛卡尔积）；
8. 【强制】在进行多条记录的增加、修改、删除时，建议使用批处理功能，批处理的次数以整个 SQL 语句不超过相应数据库的 SQL 语句大小的限制为准；
9. 【强制】尽量避免在一条 SQL 语句中从 >=4 个表中同时取数，对于仅是作为过滤条件关联，但不涉及取数的表，不参与表个数计算；如果必须关联 4 个或 4 个以上表，尽量采用子查询的方式；
10. 【强制】对索引列的比较，尽量避免使用 not 或!=，可以考虑拆分为几个条件。如col2 是索引列，条件col2!= 0 可以拆分为 col2 > 0 or col2 < 0；
11. 【强制】建议每条 SQL 语句中 in 中的元素个数在 200 个以下，如果个数超过时，使用临时表；
12. 【强制】禁止使用 col2 in('',''….) or col2 in('','','')；
13. 【强制】禁止使用 or 超过 500，如 col2='123' or col2='456'；
14. 【强制】禁止在一条 SQL 语句中使用 3 层以上的嵌套查询，如果有，请考虑使用临时表或中间结果集；

**推荐使用：**

1. 【推荐】程序尽量使用 PrepareStatement，利用预处理功能；
2. 【推荐】任何对列的操作都将导致表扫描，所以应尽量将数据库函数、计算表达式写在逻辑操作符右边；
3. 【推荐】在对 char 类型比较时，建议不要使用 rtrim() 函数，应该在程序中将不足的长度补齐；
4. 【推荐】用多表连接代替 exists 子句；
5. 【推荐】在使用 like 的时候，尽量避免在 value 的左边使用 % 号；
6. 【推荐】不同的连接实现的业务逻辑是不同的，出现的外连接多是没必要的；
7. 【推荐】尽量使用批处理，减少访问数据库的次数；
8. 【推荐】SQL 里的表名一定要引用表别名；
9. 【推荐】尽量使用表连接代替子查询；
10. 【推荐】子查询外部条件内推；
11. 【推荐】不要关联不必要关联的表，高级版中有很多表都设计了冗余字段，应该尽量利用冗余字段避免不必要的表访问；
12. 【推荐】如果有多表连接时，应该有主从之分，并尽量从一个表取数，如 select a.col1,a.col3 from a join b on a.col3=b.col4 where b.col5='a'；
13. 【推荐】尽量避免使用 order by 和 group by 排序操作，因为大量的排序操作影响系统性能。如必须使用排序操作，尽量建立在有索引的列上；
14. 【推荐】用 where 子句替换 having 子句，避免使用 having 子句作为条件。having 只会在检索出所有记录之后才对结果集进行过滤，而这个处理需要排序、总计等操作。如果能通过 where 子句限制记录的数目，那就能减少这方面的开销。having 中的条件一般用于对一些几何函数的比较，如 count()等，除此之外，一般的条件应该写在 where 子句中；
