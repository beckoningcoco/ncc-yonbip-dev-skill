# 1. 数据权限简介 | YonBIP高级版

## 1\. 数据权限简介

## 1.1 数据权限概念

数据权限，用于控制业务系统当中数据的引用、查询以及维护等相关操作。 场景：业务单据查询、档案参照以及维护权限。

数据权限节点展示图：

![2-21](https://community.yonyou.com/datadict/bipbook/assets/2-21.imxHFvAc.png)

## 1.2 数据权限概念

YonBIP高级版的数据权限可以分为维护权限和使用权限两部分。顾名思义，维护数据权限是指对某种资源实体的数据有无增删改查的权限，典型的应用场景包括对某条单据进行修改操作或者删除操作时，校验当前用户是否有权限进行这项操作。而使用权限则是指在资源实体被引用到某个场景的时候，对当前用户的限制，换句话说就是控制用户对某资源实体的数据引用查看校验。

| 维护权限 | 使用权限 |
| --- | --- |
| 修改，删除等操作 | 查询，参照过滤等 |
| 授予类型的权限，如果某角色对某资源不设置维护数据权限，则认为该角色对这个资源无权 | 限制类型的权限，如果某角色对某资源不设置使用数据权限，则认为改角色对这个资源有权 |
| 可在不同的角色下定义不同的维护权限规则 | 可在不同的角色下定义不同的使用权限规则 |
| 分为两级：粗粒度级规则(全部有权或者合部无权)；粗粒度规则(指可以对元数据上的每个字段进行权限的控制) | 分为两级：粗粒度级规则即(全部有权或者合部无权)；粗粒度规则(指可以对元数据上的每个字段进行权限的控制) |

**特殊权限** ：特殊权限是在集团背景下针对某一类人设置的数据权限，其设置对象不是角色，是一类特殊的用户。特殊权限分为：主管权限、创建者权限和审核者权限 。

（1）创建者权限：创建者权限的应用范围为基础数据/单据的创建人，创建者权限的授权资源是资源实体，资源实体下细分操作。例如可以为销售订单的修改操作设置创建者权限。销售订单的修改操作启用了创建者权限后，则只有单据的创建者能够修改自建销售订单。

（2）主管权限：系统中提供设置主管的功能，允许为公司或者部门设置主管。主管权限的应用范围为系统内的全部“主管”，目前系统内主管权限的应用是基础数据/单据的审批。应用效果：主管只能审核管辖范围内人员创建的基础数据/单据。

（3）审核者权限：审核者权限的应用范围为基础数据/单据的审核人，目前系统中对于审核者权限的应用是反审核的操作。应用效果：只有审核人能够进行反审核操作。

## 2.3 维护权限和使用权限代码配置

**使用权限** ：YonBIP高级版当中单据模板上并没有相关的“是否启用数据权限”的控制选项。 YonBIP高级版参照当中可以在节点JS代码当中对相关字段的queryCondition当中增加isDataPowerEnable字段来做控制。 添加的isDataPowerEnable字段将会在YonBIP高级版的DefaultRefSqlBuilder.getDataPowerSQL方法当中的到处理： 如果其值为'Y'或者为空则使用数据权限，为N则不使用。默认的使用资源操作是default，如果需要传递使用资源操作则应该带上DataPowerOperationCode，值为资源操作的编码。 详情可见nccloud.web.refer.RefHelper.DefaultRefSqlBuilder类。

参照前端示例：

```javascript
javascriptlet meta = props.meta.getMeta();
meta[tableId].items.forEach((item)=>{
    if(item.attrcode == 'pk_region'){
        item.queryCondition=()=>{
            return{
                pk_country:value.value,
                isDataPowerEnable:'Y',//使用数据权限，默认启用
                DataPowerOperationCode:'default'//资源操作编码，默认default
            }
        }
    }

});
```

如果需要新增加一个参照，那么对于新增的参照，需要调用NCCAction当中的 setResourceCode方法。 其中setResourceCode方法传递的字符串即为sm\_permission\_res当中的resourcecode字段。 如下所示：

```java
javapublic class CategoryTreeRef extends DefaultTreeRefAction {
    public CategoryTreeRef() {
        super();
        setMdClassId("29785a9e-12dd-4302-80b4-86a3851726c7");
        setResourceCode("category");//数据权限资源编码
        ....
    }

}
```

**维护权限：** 维护权限的代码校验一般写在action类里面，继承nccloud.web.action.NCCAction，在execute方法里面加入校验

```java
javasuper.setResourceCode("jobsegexe");
super.setMdOperateCode("calculate");
super.checkDataPermission(request, jobs);
```
1. setMdOperateCode传入的字符串是md\_busiop表当中的name字段。
2. setRersourceCode传入的字符串是数据权限资源编码，表sm\_permission\_res当中的resourcecode
3. 最后一步就是调用checkDataPermission，如果鉴权不通过，该方法会抛出异常或者返回异常信息。

**其他情况处理：** 如果有需要自己去拼查询sql情况，可以调用 IDataPermissionPubService里面的getDataRefSQLWherePart方法获取数据权限相关sql片段。

```java
javaString getDataRefSQLWherePart(String userID, String pk_group, String resourceCode, String operationcode, String tableAlias) throws BusinessException;
```

相关参数的含义如下： (1) 第一个参数指需要鉴定权限的用户的cuserid。 (2) 参数是当前登录集团的主键。 (3) 参数是指资源实体的编码，亦即sm\_permission\_res的resourcecode字段。 (4) 参数是指操作的编码，亦即表格sm\_res\_operation的operationcode字段。 (5) 参数是指的表别名，返回的SQL片段当中会包含“别名.主键”的形式。

当我们需要添加数据权限时，到【数据权限】节点添加相应配置即可。在上面的参照框需要先选择角色，然后根据需要添加维护权限或者使用权限的规则。例如下面图片示例：

## 2.5 档案数据变动

通常情况下，存在一个后台任务能够扫描数据权限信息并且将用户有权限的数据分表记录，标准环境当中这个后台任务的名称即为“数据权限”。 数据使用权是采用数据库分表来记录符合条件的数据。

**后台任务** ：权限有效期及数据变更调整定任务。 需要支持使用权限的档案进行变化后需要通知权限系统档案数据变化并记录，当数据发生变化时，发送业务事件。

**事件源ID** ：默认规定事件源ID即为档案数据对应元数据实体的ID（即定义的权限实体资源所关联的元数据ID）

**事件类型** ： 新增后-IEventType.TYPE\_INSERT\_AFTER、修改后-IEventType.TYPE\_UPDATE\_AFTER 、删除后-IEventType.TYPE\_DELETE\_AFTER

**传输事件对象** ：nc.bs.businessevent.UsePermChangeEvent

平台注册的监听会收集上述事件并记录数据变化。 当发生数据变化时，调用 EventDispatcher.fireEvent(new UsePermChangeEvent(mdid, IEventType.xxx))。 经过上述步骤，即可实现当数据有了变动的时候即时更新数据权限的信息。

## 3\. 特殊数据权限开发

特殊数据权限是一种设置简便的数据权限，不针对具体的角色或者用户设置，而是作用于一类“特定”的操作员。

\*\*创建者权限：\*\*指定业务对象的创建者所能拥有的权限，勾选了这个标记，可以设置档案或者单据的创建者能够做哪些操作，具体的操作在创建者权限明细列表中定义。例如：单据的创建人才可以修改和删除单据，单据的创建人不能审核自己的单据，这两种就是创建者有权和创建者无权的场景。

\*\*主管权限：\*\*勾选了这个标记，表示该单据启用了主管权限，组织的主管对组织内人员制单的单据，有全部操作权限。一旦启用主管权限，是主管的用户就对可以自己下级用户创建的单据进行操作，例如，部门经理能查看、维护、审批其下级创建的单据或档案。通常主管权限需要跟创建者权限结合使用。

\*\*审核者权限：\*\*用于控制只有审核人才能做反审核，主要为支持不通过审批流完成审批的业务。

## 3.1 数据权限资源管理

特殊数据权限也需要在系统管理员登陆后在【数据权限资源管理】节点注册资源，但是在上面的表单上要启用特殊权限，如果有审批流审批等操作还要勾选“支持审核者权限设置”。

下面资源操作一般会划分的比较详细，例如修改、删除、审批、提交、收回等等，这样后面在配置权限时也可以进行灵活配置。如果是需要特殊数据权限用到的资源操作，则应该“用于特殊数据权限”设置为是，否则在后面配置【特殊数据权限】子表操作明细上会选不到。

规则类型页签与普通数据权限没有太多区别，参考上面第二小节的介绍。

```java
    /**

     * @param instanceObject 单据对象VO
     * @return 返回Map,key为单据vo，value为是否有操作权限。
     */

    public Map<Object, UFBoolean> isUserHasPermission(String cuserid, String resourceCode, String operationCode, String pk_group, Object[] instance) throws BusinessException;
}
```

## 3.2 特殊数据权限应用示例

**（1）创建者权限应用场景** 在【特殊数据权限】里面找到采购订单，勾选创建者权限，并设置修改操作、删除等操作创建者有权。然后使用ncc1用户到【采购订单维护】节点创建一个单据，然后用另外一个用户ncc2去执行删除或修改等操作，会发现系统提示“单据CD2022012100000011不满足操作权限规则，请调整”，说明创建者权限已经起作用。

**（2）主管权限应用**

在三个地方可以设置主管：在部门中可以设置部门主管， 也可以设置组织主管；在业务单元节点中可以设置组织主管；在特殊数据权限节点也可以设置组织主管。

例如在上面我们在采购订单创建了单据，此时只有ncc1有权限去修改、删除等操作权限，我们到【特殊数据权限】节点把采购订单中的创建者权限、主管权限勾选上，然后点右上角的“组织主管”按钮把ncc2加入到主管中，这时我们再去修改、删除采购订单CD2022012100000011。会发现不再会有“不满足操作权限规则”的报错提示了。
