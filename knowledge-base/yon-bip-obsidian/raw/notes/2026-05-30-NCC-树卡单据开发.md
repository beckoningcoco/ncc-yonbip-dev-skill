# 低代码 - 树卡单据开发 | YonBIP高级版

## 低代码 - 树卡单据开发

## 1 元数据

本篇文章基于“供应商档案”为场景出发，做树卡应用的 Demo。供应商是一个一主多子的档案型应用，这里个性化改造成一主多子的单据型树卡应用，改造后的应用带有拉单、审批流、导入导出、打印输出等功能，原供应商档案元数据主实体访问器类型为 NCVO，现在单据型应用其元数据主实体访问器类型应该为 AggVO。

效果图如下：

![](https://community.yonyou.com/datadict/bipbook/assets/1.Dm-Bb2bv.png)

### 1.1 设计元数据

#### 1.1.1 步骤如下

1. 以开发者用户身份登录系统
2. 打开元数据设计节点（动态建模平台/应用工厂/高阶定制/元数据设计）
3. 新增->新增BMF

4. 填写向导信息：树卡应用页面类型选择“树卡页面”，选择哪些元数据特性，是根据当前做的应用所支持的功能来决定的
	![](https://community.yonyou.com/datadict/bipbook/assets/3.BNdXEOk6.png)
5. 属性调整：向导页面点击确定后，元数据就已经设计好了，如果觉得默认生成的表名、实体名不规范，可以根据自己的业务做简单调整。
	![](https://community.yonyou.com/datadict/bipbook/assets/4.CMmeL0va.png)

#### 1.1.2 属性介绍

![](https://community.yonyou.com/datadict/bipbook/assets/5.Nn-6t_6-.png)

- 访问器类型：有NCVO和AggVO两种类型，其中NCVO类型不支持审批流特性、业务流特性、拉单特性；AggVO类型能支持所有特性。元数据设计器默认 AggVO。
- 扩展标签：TREE：树卡页面必须支持。其他的DOC（档案特性）、BUSILOG（业务日志）、URC（唯一性校验）、BDMODE（管控模式）等扩展标签，根据节点业务，自行选择填写即可，写多个页签时用英文逗号隔开；勾选档案特性时，元数据设计器默认填 DOC；

#### 1.1.3 元数据的结构介绍 - 接口

所涉及到的元数据业务接口一共有八个，元数据设计器根据选择的特性，自动实现了接口的实现，并且接口字段映射都设置好了。如果默认生成的映射不满足需求，可以自行调整。

| **接口名** | **显示名** | **作用** |
| --- | --- | --- |
| IBDObject | 档案接口 | 档案类型单据必须实现的接口，但不仅限于档案单据实现 |
| IAuditInfo | 审计信息接口 | 审计信息接口，防止单据出现并发问题 |
| IOrgInfo | 组织信息接口 | 组织信息接口，在做不仅限于管控范围业务时需要该接口 |
| IMakeTime | 制单时间接口 | 制单时间接口 |
| IPfBillLock | 业务 Pk 锁接口 | 单据的 Pk 锁接口 |
| IFlowBizItf | 流程信息获取、回写接口 | 审批流、业务流、拉单等功能需要该接口作为底层支撑 |
| IHeadBodyQueryItf | 主子单据查询接口 | 主子 VO 元数据实体查询时底层使用的接口 |
| IRowNo | 行号接口 | 拉单上游一般有行号属性，标记当前数据行与上游单据行 |

#### 1.1.4 注意点

树形结构要有 parent\_id 属性，类型为当前实体

### 2.2 基础配置

![](https://community.yonyou.com/datadict/bipbook/assets/10.Bk1DgA8H.png)

- 应用编码：一般情况下应为 8 位，若要不受产品授权控制则第三位是 H，如果特殊情况需要长应用编码，最长不得超过 50 位
- 组织类型：需要按照具体应用业务的组织类型选择
- 菜单编码：如无特殊情况与应用编码保持一致即可
- 其他属性：按照实际情况填写即可

![](https://community.yonyou.com/datadict/bipbook/assets/11.jDow0Htf.png)

代码风格选择：相较于旧版本增加了专为各领域代码实现的领域特点风格代码，客开风格为默认风格；想要选择领域代码风格，需要系统已安装相应领域产品，并且对领域代码结构有一定了解，根据自己实际情况选择，客开首选客开风格。

### 2.3 页面配置

![](https://community.yonyou.com/datadict/bipbook/assets/12.C2fjDHH0.png)

- 页面样式：根据元数据的类型选择：编辑表页面、主从页面、树表页面、树卡页面的一种。如：业务要求应用是一个树卡类型的应用，则可选择树卡页面，但元数据上至少需要实现 IBDObject 接口并配置接口属性映射，元数据需要实现的其他接口需要按照不同的插件功能去实现并配置接口属性映射。
- 插件选择：根据业务情况选择对应插件即可，注意，如果选择拉单插件，元数据必须实现审批流接口。

### 2.4 参数配置

![](https://community.yonyou.com/datadict/bipbook/assets/13.BRAhhOuF.png)

参数配置中一些需要注意的配置项说明：

- 生成的参照类型：每个应用都会默认生成一个参照，默认提供树形参照、表型参照、树表型参照
- 单据类型编码：审批流的单据类型编码和拉单插件的单据类型编码保持一致即可
- 审批页面配置：审批中心和单据追溯页面如果需要修改或者上传附件功能时选择
- 来源单据类型、子表名称：用于拉单上游配置，如果当前实体是一主多子的，并目主实体实现的审批流接口属性映射来源单据类型和来源单据 id 配置的是子实体的来源单据类型和来源单据 id，则要求这里的来源单据类型和子表名称都需要选择；如果单主或者一主多子、一主一子，主实体实现的审批流接口属性映射来源单据类型和来源单据 id 配置的是主实体本身的来源单据类型和来源单据 id，则要求这里的来源单据类型需要选择，子表名称无需选择

### 3.1 调试 - React运行机制

![](https://community.yonyou.com/datadict/bipbook/assets/23.OuoUpS2F.png)

高级版前端使用React框架开发页面，了解React的生命周期对于代码的理解会很有帮助。在React的生命周期中，总共分为三个阶段；一是挂载阶段，进入页面时，React将先通过 construct 完成 props 和 state 的初始化，然后依次执行 componentWillMount 函数、render 函数渲染页面、componentDidMount 函数；二是渲染阶段，页面初始化并渲染完成后将进入到运行中状态，当 props 或者 state 发生改变时，将触发相应的生命周期函数并重新执行 render 函数渲染页面，然后回到运行中状态；三是卸载阶段，关闭页面时触发，将执行 componentWillUnMount 函数后，完成页面卸载。

### 3.2 调试 - React运行机制

![](https://community.yonyou.com/datadict/bipbook/assets/24.BaGfQ-Rk.png)

当页面处在运行中状态时，用户的操作将触发动作派发，执行对应的业务功能，业务功能主动触发setState方法使 state 发生改变，从而执行渲染阶段的生命周期。

### 3.3 代码结构

![](https://community.yonyou.com/datadict/bipbook/assets/25.B5UXnX4I.png)

高级版前端框架是以 tinper 为基础，包装后得到 platform 作为高级版前端底座，低代码平台生成的前端代码也是以此为基础，代码包含前端应用 Index、前端参照组件、依赖组件引入配置、前端多语、前端工具；前端应用 Index 进行了严格的函数区域划分，有 React 生命周期函数区、单一职责功能方法区、动作派发区、业务功能方法区，方便各位开发查找。

```js
                    sum = FloatAdd(sum,parseFloat(row.values.cnum.value))
                }
            });
            //给表单上总数量属性赋值
            this.props.form.setFormItemsValue('lcdpsupplierForm',{
                'cnum':{
                    display:sum,
                    value:sum
                }
            })
        }
    }
```

### 3.15 调试 - 奇怪的 setMeta()

![](https://community.yonyou.com/datadict/bipbook/assets/46.CKfyXV2u.png)

解释：调用 initMeta() 处理完模板后，后续代码有 setMeta()处理，而其他地方没有，所以处理 meta 之后一定要记得 setMeta()

```js

             * 需要用到,全局配置中包含所有它需要用到的参数,
             */
            ...this.config,
            /**
             * 加载NCC资源之后的回调方法
             */
            callback : ({template,button,context,lang,inlt})=>{
                /**
                 * 初始化模板,修改模板相关配置
                 * 初始化按钮,修改按钮相关配置,
                 * 并将模板配置,按钮配置输入到平台,通过setMeta,setButtons输入.
                 * 让平台进行初始化.当平台初始化完成后,通过then继续后续的工作
                 */
                this.initMeta({meta:template,context});
                this.initButton(button);
                /**
                 * 合并处理 模板 和 按钮
                 * Promise.all就像后端开多线程，并等待所有线程回来之后再then（执行下一步）
                 * 而且可以多个then

                Promise.all([

                            resolve(true);
                        });
                    }),
                    new Promise(resolve=>{

                            resolve(true);
                        })
                    }),
                    new Promise(resolve=>{
                        this.setState({...this.state,context},()=>{
                            resolve(true);
                        })
                    })
                ]).then(()=>{
                    return this.pageReady();
                }).then(()=>{//初始化完成
                   /**
                    * 解构state(解构的意思请移步ES6文档查看【https://es6.ruanyifeng.com/】)

                    * showmode:显示模式  list|card|fetchlist|fetchcard

                    let {isPageReady, showmode,editmode, btn,head,dragComp,execlExport,printOutput,attachment} = this.state;
```
