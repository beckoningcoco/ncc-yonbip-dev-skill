# 专业开发

> YonBuilder 专业开发是针对专业开发人员的高级定制开发产品，支持本地代码扩展、独立部署和全生命周期管理。

## 产品定位

应用构建有三个版本：

| 版本 | 定位 | 开发模式 |
|------|------|----------|
| 基础版 | 租户快速实施 | 无代码声明式 + 在线函数式低代码 |
| 专业版 | 原厂/ISV/专属化客户 | 基础版全部能力 + 本地代码扩展 + 独立部署 |
| 升级版 | 新版统一平台 | 零代码 + 低代码 + 专业开发三合一，支持平滑升级 |

## 核心组件

- **应用引擎**：提供脚手架下载、本地调试、CI/CD 全生命周期支撑。管理引擎信息、协作人员、托管应用
- **技术平台**：面向开发者的 DevOps 平台，提供容器、微服务、流水线等云原生能力
- **YDS 工具**：一站式 IDE，覆盖项目创建、调试、部署、制品、补丁全流程
- **YPD 框架**：后端强类型实体 + 插件扩展开发框架

## 技术栈

| 层级 | 技术 |
|------|------|
| 前端 | Node.js + React |
| 后端 | Java + SpringBoot |
| 数据库 | MySQL / Oracle / 达梦 / Oscar / PostgreSQL |
| 中间件 | YonMatrix + Redis + Kafka + Nginx + ES + MinIO/SFTP |
| 云原生 | K8s + Docker + DevOps |


## 脚手架

Freemarker 模板引擎 + Git 远端模板仓库生成：
- 后端：YPD 框架脚手架 / MDD 框架脚手架
- 前端：MDF 框架脚手架 / YNF 框架脚手架

模板在服务内做一级缓存，远端 Git 获取最新版本。私有云预置固化版本。

## 引擎类型：YPD vs YDS

| 维度 | YPD（R6） | YDS（BIP5 / R6+） |
|------|----------|-------------------|
| 创建方式 | Web 平台（数字化工作台） | YDS IDE 命令（`Ctrl+Shift+P`） |
| 脚手架获取 | 自动推送 Git，克隆到本地 | YDS 本地生成 |
| JDK | JDK 8 | BIP5: JDK 21 / R6: JDK 8 |
| 健康检查 | `https://域名/引擎编码/extend/healthycheck` | `http://localhost:62871/_yms/ready` |
| 后端端口 | 8080 | 62871 |
| 调试方式 | 手动配置 | YDS 一键 RDBGA / HOST |

> R6 版本同时支持 YPD（Web 平台）和 YDS 两种模式。BIP5 仅使用 YDS 模式。

## 应用创建流程

### 前置准备

1. **领域注册**（如领域不存在）：数字化工作台 → 数字化建模 → 系统管理 → 应用管理 → 领域管理 → 新建领域（领域编码、名称、图标）
2. **菜单管理**（可选）：添加一级导航目录，用于应用功能发布
3. **应用引擎**：需提前创建（参考 [[环境搭建]]），若用 YDS 引擎需通过 YDS 浏览器访问工作台

### 新建应用

数字化工作台 → 云平台 → 开发平台 → 低代码开发 → 应用构建：

1. 使用 **"客开模板"** 新建
2. 选择对应领域 + 自建应用引擎
3. 填写应用编码和名称
4. **编码规范**：`c{1}{2}{3}` 格式，**禁止 `-`、`_` 等特殊字符**（示例：`cpurart`）

> ISV 生态开发的应用编码需提前申请。

### 常见问题

| 问题 | 解决 |
|------|------|
| 只有公共引擎可选 | 将账号添加到自建引擎的成员列表中 |
| 编码已重复 | 修改编码值后重新保存 |
| 应用构建打不开/白页 | 确认服务已安装 → 技术平台查看微服务状态 → 重启流水线 |
| 导航菜单没有"应用管理" | 功能权限不足，换账号或申请授权 |

## 开发流程

1. 创建应用引擎 → 创建应用
2. 对象建模 → 页面建模 → 流程&自动化配置 → 集成配置
3. Git 下载脚手架 → 本地前后端开发调试
4. 开发成果迁移（传输包 → 测试环境 → 生产环境）
5. 租户应用可发布为系统应用

## 开发体系

> 零代码、低代码、专业开发共用同一底层架构。

### YDS 端到端生命周期

YDS 覆盖项目创建 → 本地开发 → 调试 → 部署 → 制品的全流程。

**远程调试（Remote Debug）**：无需访问目标服务器，本地 YDS 直接对远程服务进行断点调试、日志分析与性能监控。支持多线程、不中断服务。

**本地调试**：将线上服务复刻到本地容器，在隔离仿真环境中排查问题。

**产品盘及补丁**：将代码、配置、脚本、依赖包、开发成果完整封装，标准化交付和一键部署。

### 生态脚手架

> `com.yonyou.iuap:yonbip-eco-parent:5.0.0.0-RELEASE`

| 特性 | 说明 |
|------|------|
| 统一生态底座 | 标准化命名、依赖、版本，统一配置 |
| SDK 深度赋能 | 内置扩展接口 + SDK 工具链，快速调用平台/领域能力 |
| 与产品解耦 | 并行于 BIP 发版周期独立更新，高版本兼容低版本 |
| 依赖拍平 | pom 层级扁平化，冲突率降低 |

### API 网关

完整 API 托管服务，覆盖 API 的创建、维护、发布、监控、安全管控全生命周期，支持多协议接入和统一管理。

### 集成总线

新一代企业服务总线：多协议连接不同 SaaS 服务和企业应用，云原生运行时引擎 + 可视化设计器 + 低代码脚本编排。

## 单据开发详解

> 以一主多子单据（商品采购需求单）为例。

### 对象建模

1. **新建枚举**：枚举管理 → 新增枚举档案。保存即生效，无需发布。商开环境编码以 `_P` 结尾。
2. **创建业务对象**：新增业务对象 → 勾选"同时生成主实体"
3. **主实体开发**：编辑主实体 → 勾选引用接口（审批、业务流、逻辑删除、自动编码）→ 添加业务属性及数据库表名 → 发布
4. **子实体开发**：主实体 → 新增 → 子实体 → 录入名称/编码/表名 → 添加业务属性 → 发布
5. **数据库表创建**：实体发布后，导出 SQL 脚本在引擎对应数据库执行

**关键参照**：采购组织 `org.func.PurchaseOrg`、库存组织 `org.func.InventoryOrg`、员工 `bd.staff.Staff`、物料 `pc.product.Product`

**快速新增子表字段**（逗号分隔格式）：
```
materialId,物料,单选引用,pc.product.Product;
materialCode,物料编码,文本;
num,数量,整数;
unit,计量单位,文本;
total,预估金额,数值;
```

### 页面建模

1. **新建页面**：选中业务对象 → 【新建页面】→ 选择"一主多子" → 填入名称和编码 → 勾选"生成列表页面"
2. **列表配置**：拖拽查询条件、日期/金额比较符改"区间"、固定列开关
3. **卡片界面配置**：
   - 卡片分区：右键卡片组 → 插入 → 卡片 → 拖拽字段到对应分区
   - 多页签：附件页签（表单+附件组件）、明细子表页签
   - 参照带出：选择物料后自动填充物料编码、计量单位到子表字段
   - 字段配置：小数精度默认 2 位、不可编辑开关、必填项、占整行

### 页面规则

| 类型 | 触发时机 | 示例 |
|------|----------|------|
| 新增赋默认值 | 页面初始化 | 付款方式=`转账`、采购日期=`系统日期` |
| 值改变赋值 | 值发生变化 | 明细预估金额求和 → 预估总金额 |

> ISV 开发禁止使用页面规则与公式。赋默认值场景下，配置迁移需先迁移默认值档案。

## 应用发布

开发完成后，将应用功能发布到数字化工作台供业务用户使用。

### 发布步骤

1. 进入应用 → **发布管理** → 左侧添加**分组**（菜单分组结构）
2. 选择分组 → 点击 **"+"** 添加功能节点 → 填写服务编码（可直接复制单据编码）
3. 勾选节点 → 【添加】→ 【发布应用】
4. 配置发布参数：
   - 发布到工作台 / 移动端
   - 选择一级菜单（展示在导航"企业自定义"下）
   - 着陆页默认按结构生成
5. 【确定】发布 → 协同实施人员进行菜单管理与权限分配

### 常见问题

| 问题 | 解决 |
|------|------|
| 服务编码重复 | 添加后缀 |
| 角色编辑页找不到功能节点 | 客开单据属业务类角色，平台功能属管理类角色 |
| 授权后菜单没显示 | 等缓存（10 分钟内），刷新或重新登录 |
| 页面建模修改后需再发布？ | 不涉及功能权限则不需要 |
| 菜单功能报 502 | 线上未部署，需通过开发成果迁移生成制品盘安装 |

## 支撑组件配置

### 编码规则

业务对象编号由编码元素按规则拼接（如 `TT-20150615-0001` = 常量 + 日期 + 序列号）。

**流水依据**：所有流水依据元素的组合，序列号在流水依据变化时重新计数。

**配置步骤**：
1. 数据模型勾选引用接口"自动编码"（自动生成编码字段）
2. 卡片页面选中编码字段 → 属性中【配置编码规则】
3. 新增编码元素：常量、属性值、系统时间、序列号
4. 保存后可在数字化建模 → 编码规则定义中查看

### 审批流

1. **数据模型**：勾选引用接口"审批" → 页面自动生成审批功能
2. **新建流程**：流程&自动化 → 新建工作流 → 设计
3. **配置节点**：参与人、抄送人、操作、时限、业务活动、智能审核、服务项
4. **保存并发布**

**应用效果**：提交 → 待办消息 → 消息中心审批 → 同意/退回。

### 打印配置

1. 页面建模 → 打印模板 → 添加模板（可创建空白模板）
2. 设计界面拖拽页眉、卡片组件 → 设置列数 → 选中数据字段 → 保存
3. 可配置多个模板，设置默认打印模板
4. 卡片页面自动生成打印按钮

## 后端插件开发（YPD）

> YPD 框架通过插件扩展机制在单据动作前后注入业务逻辑。脚手架：`bootstrap`（启动）+ `service`（插件开发）。

### 开发流程

1. **导出 JAVAPOJO**：对象建模 → 选择业务对象 → 导出 → 解压到 `service` 工程的 `entity` 包
2. **创建插件类**：在 `com.yonyou.ucf.mdf.plugin` 包下创建类，继承 `AbstractBillPlugin`
3. **添加注解**：`@BillPlugin(busiObj = "业务对象编码")`。一个业务对象只能有一个插件类
4. **重写钩子方法**：`beforeSave` / `afterSave` / `beforeAudit` / `afterAudit` 等 23 个动作码
5. **获取数据**：`billContext.getBillDO()` → 强转为具体 POJO
6. **赋值/校验**：直接调用 set 方法，校验失败抛出 `BizException`

### 核心插件方法

| 动作码 | 方法 | 时机 |
|--------|------|------|
| `ACTIONCODE_SAVE` | `beforeSave`、`afterSave`、`beforeDB`、`initBillConfig` | 保存 |
| `ACTIONCODE_AUDIT` | `beforeAudit`、`afterAudit` | 终审节点 |
| `ACTIONCODE_DELETE` | `beforeDelete`、`afterDelete` | 删除 |
| `ACTIONCODE_SUBMIT` | `beforeSubmit`、`afterSubmit` | 提交 |
| `ACTIONCODE_ADD` | `initBillDefaultValueWhenNew` | 新增初始化 |
| `ACTIONCODE_LIST` | `beforeBillListQuery`、`afterBillListQuery` | 列表查询 |
| `ACTIONCODE_BIZFLOWPUSH` | `beforePush`、`afterConvert`、`afterPush` | 推单/生单 |
| `ACTIONCODE_COPY` | `beforeCopy`、`afterCopy` | 复制 |

**0 租户插件**：`@BillPlugin` 不绑定 busiObj，其他插件未命中时作为兜底。

**YpdBillContext**：`billDO`（单据对象）、`billDOs`（数组）、`workflowParam`（工作流）、`formulaMap`（公式）。

### 开发环境

BIP5 适配 JDK 21，VS Code 配置 `java.configuration.runtimes`。

## 前端扩展开发（MDF）

> 通过前端脚手架中的扩展脚本，基于 ViewModel 事件和 API 实现页面级业务逻辑。

### 扩展脚本规范

- **命名**：`{应用编码}_{billNo}_VM.Extend.js`
- **位置**：`src/business/{应用编码}/`
- **结构**：`cb.define` → `doAction`（入口）+ `init`（业务逻辑）
- **公共脚本**：`business/common/common_VM.Extend.js`，通过数组参数引入

### 调试方式

YDS 内置 Resource Override 插件，将线上压缩的 `extend.min.js` 代理到本地源码，支持断点调试。

> **注意**：`debugger` 和 `console.log` 打包时自动去除，生产环境不生效。

### ViewModel 模态

| mode | 说明 |
|------|------|
| `browse` | 预览态，仅查看 |
| `edit` | 编辑态，修改已有数据 |
| `add` | 新增态 |

### cb.utils 常用工具

`isArray`、`extend`、`getNewId`、`getCookie`、`isEmpty`、`alert`、`queryString`（get/set/has/toStr）。

### 部署

代码上传 Git → 执行流水线（无需本地 build）。

## MDF vs YNF 前端框架对比

YNF（YonBuilder New Framework）+ YPD 是新一代开发框架，**暂不支持 MDF/MDD 应用升级**。

| 维度 | MDF (旧) | YNF (新) |
|------|----------|----------|
| 应用构建 | 低代码开发-应用构建 | **应用构建升级版** |
| 业务对象创建 | 手动创建主子实体 | 可同时勾选主子实体+数量 |
| 建模方式 | — | **实体模型 + 系统模型** |
| 卡片布局 | 卡片组 | **折叠面板 + 表单** |
| 查询区间 | 属性配置 | **FilterDateRangePicker** + compareLogic |
| 页面交互 | 页面规则 | **自定义赋值**（规则/公式/表达式） |
| 字段快捷栏 | 无 | **右侧字段快捷栏** |
| 原厂扩展 | 无 | **原厂应用扩展** |
| 功能发布 | 分组+功能节点 | 选择单据+**权限配置**（全员/自定义/角色） |
| 发布后修改 | 功能权限需重发 | **数据同源，无需重发** |
| 脚手架工具 | YDS/VSCode 内置 | **ynf-dx-cli** 命令行 |
| 本地端口 | 3003 | **9090** |
| 调试代理 | YDS 内置自动 | **手动安装 Resource Override** |
| 架构 | viewModel + Model | **rootStore + stores + actions + reactions** |
| 部署 | Git + 流水线 | 手动 Nginx 上传（暂不成熟） |

### YNF 核心概念

| 概念 | 说明 |
|------|------|
| **rootStore** | 页面根状态，管理所有数据/动作/反馈（对应 MDF 的 viewModel） |
| **store** | 数据模型：formStore（主表）、tableStore（列表/子表）、filterStore（筛选）、pageStore（页面）、cardStore（单卡） |
| **action** | 页面事件（增删改查/工作流/导航），调用：`rootStore.actions.xxx.doAction` |
| **reaction** | 副作用/响应：cardReaction、ListReaction、TreeCardReaction，监听 store 属性变化自动触发 |

### YNF 扩展开发

> 扩展脚本位于 `src/pages/{billNo}/index.extend.js`，导出四个函数。

**四个扩展函数**：

| 函数 | 用途 |
|------|------|
| `ExtendStore(rootStore)` | 扩展领域/行业业务 stores（Domain Store） |
| `ExtendAction(rootStore)` | 扩展明确的、静态的 action 动作 |
| `ExtendReaction(rootStore)` | 扩展 reaction，track 状态值并按逻辑更新 domain store |
| `ExtendRootStore(rootStore)` | 扩展全局 api/utils/pageStore，或直接定义业务逻辑 |

**配置文件注册**（`yds.config.js`）：
```javascript
tnsConfig.providerEntry: {
  './{单据编码}': path.resolve(__dirname, './src/pages/{单据编码}/index.extend')
}
```

**reaction afterHook 模式**（生命周期钩子）：
```javascript
export function ExtendReaction(rootStore) {
  return {
    init: {
      afterHook(rootStore, res) {
        if (rootStore.cardStore.isAdd) {
          const field = rootStore.formStore.get('fieldName')
          field.setDisabled(true)
          field.setValue('默认值')
        }
      },
    },
  }
}
```

**子表自定义方法**（ExtendRootStore + UI 协议绑定）：
```javascript
export function ExtendRootStore(rootStore) {
  const tableStore = rootStore.{子表Store名称}
  tableStore.onFieldChange = function (newVal, row) {
    // 校验逻辑...
    rootStore.utils.message.danger('错误提示')
    return false  // 阻止继续
  }
}
```

**页面跳转**：`rootStore.pageStore.push({ billNo: '页面编码', params: {...} })`

**自定义弹框/抽屉**：
```javascript
rootStore.pageStore.openModal({
  uiType: 'component',
  component: '自定义 DOM',
  bodyStyle: { height: '500px' },
  containerType: 'drawer',  // 不传为居中弹窗
  onOk: () => {}, onCancel: () => {}
})
```

**覆盖默认 action**：通过 `cardReaction` 配置，可覆盖新增/浏览/编辑/复制态的默认数据加载行为。

**YNF 关键 API**：

| API | 用途 |
|-----|------|
| `rootStore.cardStore.isAdd` | 判断新增态 |
| `rootStore.formStore.get('field')` | 获取主表字段 |
| `rootStore.{tableStore}.getValue()` | 获取子表全量数据 |
| `rootStore.utils.message.danger('msg')` | 错误消息提示 |
| `rootStore.pageStore.push({...})` | YNF 单据间跳转 |
| `rootStore.pageStore.openModal({...})` | 打开自定义弹框/抽屉 |

### YNF 环境搭建

```bash
npm install ynpm-tool -g        # 包管理工具
ynpm install ynf-dx-cli -g      # 脚手架工具
ynf init app ynf-${domainKey}   # 初始化（app/lib/sdk）
```

## 前端扩展开发（YNF）

> YNF 扩展开发三大场景：样式扩展、查询方案扩展、自定义组件开发。

### 样式扩展

**通过 UI 协议 + MobX 表达式动态控制样式**：

| 协议 | 方法签名 | 用途 |
|------|----------|------|
| `rowClassName` | `getRowClassName(record, index)` | 控制行样式，返回 className |
| `getCellClassName` | `getCellClassName(record, index, column)` | 控制单元格样式 |
| — | `column.dataIndex` | 判断当前列标识 |

```javascript
rootStore.tableStore.getRowClassName = function (record, index) {
  if (index === 1) return 'bg-orange'
}
rootStore.tableStore.getCellClassName = function (record, index, column) {
  if (index === 0 && column.dataIndex.includes('verifyState')) return 'blue'
}
```

> 行背景色被内部单元格白色覆盖时，需在 `style.css` 中清除单元格背景。内置样式查 `tinper-next.css`（Sources 面板 Ctrl+P 搜索）。

**tinper-next 预置样式**：背景 `bg-red/bg-pink/bg-purple/bg-blue/bg-cyan/bg-green/bg-yellow`（均 `!important`），文字 `blue/pink/yellow/purple/orange`。

### 查询方案扩展（filterStore）

```javascript
let { filterStore } = rootStore
let filterItem = filterStore.getFilterItemByName("code")
```

**赋初始值（监听 `isQueryReady`）**：
```javascript
defaultSearchCondition: reaction(() => rootStore?.filterStore.isQueryReady, () => {
  rootStore?.filterStore?.getFilterItemByName("inputType").setValue(2)
  rootStore?.filterStore?.copyCondition()
  rootStore?.filterStore?.setQueryTrigger()
})
```

> 设值后必须执行 `copyCondition()` + `setQueryTrigger()` 重新触发查询。

**设置查询条件必填**：通过 `afterHook` 修改 `filterItem.meta.required = true`。

### 自定义组件开发

**文件结构**：
```
src/
├── index.share.js          # 导出组件内容
├── index.designer.js       # 组件库设计器元数据（LibraryMeta）
└── components/{组件名}/
    ├── index.js            # React Component
    ├── index.less          # 组件样式
    └── index.designer.js   # 组件级设计器配置（CompMeta）
```

**打包注册**：`yds.config.js` → `providerEntry` 导入 `index.share.js` 和 `index.designer.js`。

**设计器关键类**：
- `CompMeta`：定义组件 `code`/`name`，通过 `addBoolProp/addGroup/addProps` 配置右侧属性面板
- `LibraryMeta`：定义组件库 `code`（`ynf-{domainKey}`）/`name`，通过 `addGroup().addComps(...)` 组织组件分组，场景开关 `isPc/isMobile/isYnf/isMdf/isNoc`

**资源中心注册**：新建 → 二方包编码 `ynf-{domainKey}` → 组件来源"设计器二方包" → 校验 → 确定 → 发布。

## 原厂扩展开发

> 对 YonBIP 标准产品进行扩展：实体扩展、页面扩展、代码扩展。

### 扩展方式

| 扩展类型 | 方式 | 说明 |
|----------|------|------|
| 实体扩展字段 | 特征扩展 | 在标准实体上增加布尔/文本等字段 |
| 扩展子实体 | 元数据平行表 | 主实体下扩展新子表 |
| 页面扩展 | 手工配置 UI 模板 | 新增子实体需配置页面模板 |
| 前端代码扩展 | `customInit`/`afterLoadData` 钩子 | 控制扩展字段的显隐/可用性 |
| 后端校验 | `cb.rest.DynamicProxy` 调用 | `beforeSave` 时调用后端接口校验 |
| Java 规则类 | 实现 `IRuleValidator` | 服务端规则函数校验 |

```javascript
  });
}
```
