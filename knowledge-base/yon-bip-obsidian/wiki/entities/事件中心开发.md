# 事件中心开发

> 集中化事件管理中心，发送方发布事件 → 订阅方注册订阅接口 → 事件触发时调用订阅接口，实现上下游业务解耦。

## 为什么需要事件中心

传统同步调用：`产品入库单 → 直接调用生产订单关闭接口`，问题：
- 产品入库单依赖了生产订单接口（紧耦合）
- 新增下游需求需要改上游代码（不可扩展）

事件中心方式：`产品入库单 → 发布"入库完成"事件 → 事件中心广播 → 各订阅方独立处理`（松耦合）

## 核心架构

| 组件 | 说明 |
|------|------|
| **事件源** | 注册事件发送方（如产品入库单服务） |
| **事件类型** | 定义事件名称、属性（如 "入库完成"） |
| **事件订阅** | 注册接收方监听接口 |
| **事件推送** | 事件触发时，中心调用订阅方接口 |

## 三种推送模式

| 模式 | 说明 |
|------|------|
| **P2P（点对点）** | 一对一精确投递 |
| **Broadcast（广播）** | 一对多，所有订阅方都收到 |
| **Partition（分区）** | 按分区键分组投递 |

## 关键特性

- **有序事件**：保证事件按发生顺序投递
- **消息拉取**：订阅方主动 pull 而非被动 push
- **消息精简**：合并短时间内重复事件，减少冗余调用
- **消息忽略**：指定条件下跳过事件
- **重试策略**：订阅失败自动重试
- **消费状态查询**：查看事件是否已被消费

## 发送方接入（3 步）

1. 事件中心页面 → 注册事件源、事件类型
2. 业务代码中发送事件，三种方式：
   - **RPC**：`EventCenterClient.publish(eventName, data)`
   - **REST**：`POST /event-center/api/publish`
   - **EOS**：本地事件发送（同 JVM 内）
3. 注意：需先注册才能发送

## 接收方接入（2 步）

1. 事件中心页面 → 注册事件监听（关联事件类型）
2. 提供订阅接口，两种方式：
   - **RPC**：实现 `IEventSubscriber` 接口
   - **REST**：暴露 HTTP 接口，接收 POST 回调

## 流程产品事件类型（参考）

| 事件 | 说明 |
|------|------|
| 流程启动 | 审批流程实例创建 |
| 任务创建 | 审批节点到达 |
| 任务完成 | 审批节点通过/驳回 |
| 流程结束 | 流程正常结束/终止 |
