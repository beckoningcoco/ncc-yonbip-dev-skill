# 参照模糊搜索过滤字段配置

## 问题现象

参照模糊搜索返回了从正常逻辑上**不应该看到**（或预期应有但被过滤掉）的数据。

以央视（中央广播电视总台）预算调整单为例：在预算对象参照中模糊搜索 "40"，参照弹窗返回了大量与 "40" 不相关的树节点数据。

参照编码：`c-kk-cctv-budget.RefTable_c597123467`，模糊搜索请求参数 `likeValue: "40"`, `dataType: "tree"`。

## 根因分析

通过后端 SQL 追踪（YMC / bpr 面板）查看模糊搜索实际执行的 SQL，发现 WHERE 条件中**同时包含了三个字段的 LIKE 匹配**：

```sql
SELECT ... FROM cctv_bgt_guid TO
WHERE (
    TO."code" LIKE '%40%' ESCAPE '\'
    OR TO."name" LIKE '%40%' ESCAPE '\'
    OR TO.path LIKE '%40%' ESCAPE '\'    -- ← 罪魁祸首
)
AND TO.dr = 0
AND TO.ytenant_id = 'udq6xdvm'
ORDER BY "pubts" DESC
OFFSET 0 LIMIT 50
```

`path` 字段存储的是树形参照的**层级路径**（如 `总部/华东区/上海`），当搜索 "40" 时，路径中包含 "40" 字符串的节点全部被命中，导致返回了大量逻辑上无关的数据。

### 排查方法

1. 打开 YMC（YonBIP Management Center）→ **bpr**（Business Process Record）面板
2. 复现参照模糊搜索操作
3. 查看 bpr 中捕获的 SQL 调用，搜索 `LIKE` 关键字定位模糊搜索 SQL
4. 检查 WHERE 条件中是否包含了不该有的字段

## 解决方案

### 操作步骤

1. 找到客开参照在 `pub_ref` 表（`iuap_metadata_service` schema）中的配置记录
2. 查看 **模糊搜索过滤字段** 配置列（如 `creffiltersql` 或类似字段），当前值为 `code,name,path`
3. **删除 `path`**，修改为 `code,name`
4. **重启客开后端服务**
5. **重启 `iuap-metadata-ui`** 服务

### 配置位置

- **数据库**：`pub_ref` 表（`iuap_metadata_service` schema），在参照的扩展配置字段中存储了模糊搜索的过滤字段列表
- **设计器**：客开参照的元数据定义 → 模糊搜索配置 → 过滤字段

## 关键知识点

1. **模糊搜索 ≠ 精确搜索**：精确搜索只匹配用户输入的完整值，模糊搜索会做 LIKE 匹配，`path` 字段参与 LIKE 匹配会引入大量噪音
2. **path 字段的作用**：在树形参照中用于层级路径展示（如 "总部/华东区/上海"），但它不应参与模糊搜索的过滤逻辑
3. **为什么默认带 path**：平台预制逻辑认为 path 也是可搜索的文本内容，但业务场景常不需要
4. **影响范围**：只影响当前参照的模糊搜索行为，不影响精确搜索和其他参照
