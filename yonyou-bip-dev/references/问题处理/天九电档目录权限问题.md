## 关于文件目录权限的知识

###  **Linux 文件与目录权限知识体系**

### 1. 看懂权限字符串：1+3+3+3 结构

在 Linux 中使用 `ls -l` 查看到的权限（如 `drwxr-xr-x`），可以拆解为四个部分：

- **第 1 位（文件类型）**：`d` 代表目录 (directory)，`-` 代表普通文件，`l` 代表软链接。
- **第 2-4 位（所有者 User/Owner）**：文件或目录创建者的权限。
- **第 5-7 位（所属组 Group）**：和文件所有者在同一用户组的用户的权限。
- **第 8-10 位（其他人 Others）**：系统中其他所有用户的权限。

###  2. rwx 权限的核心含义（重点！）

Linux 中用 `r`、`w`、`x` 来分别代表读、写、执行权限，它们在文件和目录上的意义完全不同。**这也是你之前踩坑的根本原因：**

表格



| 权限     | 字符 | 数字值 | 对【文件】的作用         | 对【目录】的作用（极其重要）                        |
| :------- | :--: | :----: | :----------------------- | :-------------------------------------------------- |
| **读**   |  r   |   4    | 读取文件内容（如 `cat`） | 列出目录内的文件名（如 `ls`）                       |
| **写**   |  w   |   2    | 修改文件内容             | **在目录内创建、删除、重命名文件**                  |
| **执行** |  x   |   1    | 运行程序或脚本           | **允许进入该目录（如 `cd`），并访问其内部文件属性** |

**💡 核心避坑点**：对于目录而言，**`x` 权限是“通行证”**。如果一个目录没有 `x` 权限（比如变成了 `rw-`），哪怕你有读写权限，也无法 `cd` 进去，更无法读取里面子文件的详细信息（会报 `Permission denied` 或显示 `?????`）。

###  3. 数字权限表示法（八进制）

我们常说的 755、775 其实就是把 `rwx` 转换成了数字相加：

- `rwx` = 4 + 2 + 1 = **7**
- `r-x` = 4 + 0 + 1 = **5**
- `rw-` = 4 + 2 + 0 = **6**
- `r--` = 4 + 0 + 0 = **4**

**常见组合速查：**

- **755 (`drwxr-xr-x`)**：最标准的目录权限。所有者全权负责，其他人只能看和进入，不能改。
- **775 (`drwxrwxr-x`)**：团队协作常用。所有者和同组的人都能改，其他人只能看。
- **644 (`-rw-r--r--`)**：最标准的普通文件权限。自己能改，别人只能读。

### 4. 常用权限管理命令

- 修改权限 (`chmod`)

  ：

  - `chmod 755 目录名`：将目录改为标准权限。
  - `chmod -R 775 目录名`：递归修改，把目录下现有的所有子文件和文件夹都改成 775。

- 修改归属 (`chown`)

  ：

  - `chown user1:user1 目录名`：把目录的所有者和所属组都改成 `user1`。

- 设置默认继承权限 (`setfacl`) —— 你的终极武器

  ：

  - `setfacl -R -d -m u::rwx,g::rx,o::rx /你的目录`：强制让该目录下未来新建的任何内容，都自动继承 755 权限，无视程序的 umask 干扰。

###  5. 为什么 Java 代码会在 Linux 上报错？

你在 Windows 本地测试正常，是因为 Windows 的文件系统（NTFS）对“目录执行权限”的概念非常模糊，忽略 `setExecutable(false)` 也不会影响程序读写文件。

但在 Linux 中，权限控制是底层且严格的。一旦代码里剥夺了目录的 `x` 权限，Java 进程（即使是文件所有者）也会被操作系统直接挡在门外，从而抛出 `java.nio.file.AccessDeniedException`。
