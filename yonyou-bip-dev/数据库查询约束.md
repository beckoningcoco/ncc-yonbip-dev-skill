# 数据库查询安全约束

## 连接前：必须确认项目和环境

连接数据库前，必须让用户**逐项确认**：

1. **项目**: 列出可用项目，确认当前操作的是哪个项目
2. **环境**: 明确是 test（测试）还是 prod（生产）
3. **用户**: 确认使用的数据库用户

**禁止**直接使用默认配置连接数据库，尤其是当配置中包含生产环境地址时。

## 生产环境：双重确认

如果用户选择的生产环境，必须执行以下流程：

```
第一次确认: "你正在连接 [项目名] 的 生产环境 数据库，确认? (输入 'yes')"
第二次确认: 如果执行的是写操作，要求输入项目名称拼音 + 输入 'YES-PROD'
```

**生产环境默认只允许执行 SELECT 语句**，禁止一切写入操作，除非经过明确的二次确认。

## 写操作（UPDATE/DELETE/INSERT/DROP/ALTER/TRUNCATE）

无论测试环境还是生产环境，执行写操作前必须：

1. **展示 SQL**: 完整显示将要执行的语句
2. **影响评估**: 如可能，先执行等价的 SELECT COUNT(*) 展示影响行数
3. **确认**: 用户必须明确确认执行
4. **生产环境**: 在以上基础上，要求输入 `YES-PROD` 再次确认

## BIP 旗舰版 SQL 编写规则

> **写 SQL 前必须先确认表名和字段名，绝不能凭记忆编造！**

1. **先查知识库**：去 `D:\yon-bip-obsidian\yon-bip-obsidian\wiki\entities\` 下找对应实体的元数据，确认 `schema`（数据库名）、`物理表`（表名）、`字段编码`（列名）
2. **schema 前缀必须写**：BIP 不同领域有不同 schema（如 `fieia`、`fiearap`、`c_scm_gterp_service_db`），SQL 中必须加 `schema.table` 前缀，不能省略
3. **字段名用 snake_case**：数据库列名是 `column_name` 格式，不是 Java 驼峰格式
4. **不确定时用 SQL 查**：`SELECT column_name FROM information_schema.columns WHERE table_schema='schema名' AND table_name='表名'`

**反例**：`SELECT * FROM arap_ledger_h` → 缺 schema，会报 `relation does not exist`
**正例**：`SELECT * FROM fiearap.arap_ledger_h WHERE dr = 0`

## NCC SQL 编写规则

> **写 SQL 前必须查 NCC 数据字典！**

1. **先查数据字典**：去 `C:\Users\99558\.claude\skills\yon-ncc-dev\ncc数据字典\` 下找对应模块的表定义
   - 数据字典按模块编号组织，如 `2006_arap_应收应付`、`2012_fa_固定资产`
   - 每个 `.md` 文件包含完整的字段编码、字段类型、引用关系
2. **注意 schema 前缀**：NCC 在 PostgreSQL 上使用时，表在特定 schema 下（如 `bipuser`），SQL 需加 `bipuser.` 前缀
3. **字段名用数据库列名**：数据字典中的"字段编码"是数据库列名，直接用于 SQL

**反例**：`SELECT * FROM fa_card` → 在 PostgreSQL 上缺 schema
**正例**：`SELECT pk_card, code, name FROM bipuser.fa_card WHERE dr = 0`

## SQL 自验证规则（AI 自动执行）

> 当 AI 生成了包含 SQL 的代码（如查询方法、DAO 方法）后，**必须自动验证 SQL 的正确性**，无需用户提示。

### 验证流程

1. **提取 SQL**：从生成的代码中提取 SQL 语句（类字段 `SQL_*`、方法中拼接的 SQL 字符串等）
2. **安全检查**：
   - SELECT 语句：保持原样，**自动追加 LIMIT 1 或 ROWNUM <= 1** 防止全表扫描
   - INSERT/UPDATE/DELETE：改写为等价的 SELECT COUNT(*) 或 SELECT 列名 FROM ... WHERE 1=0，仅验证表名和列名是否存在
   - 从 `db_config.json` 匹配对应的测试环境执行，**不碰生产库**
3. **自动执行**：通过 `db-query` 技能的 `db_query.py` 脚本执行验证
4. **结果处理**：
   - ✅ 通过 → 在回答中注明"SQL 已验证"
   - ❌ 失败 → 分析错误原因，修正后重新生成代码，**不交给用户半成品**

### 改写规则

| 原始 SQL 类型 | 改写方式 | 验证目的 |
|--------------|---------|---------|
| `SELECT ... FROM t WHERE ...` | 加 `LIMIT 1` / `ROWNUM <= 1` | 表名+列名+语法 |
| `INSERT INTO t (a,b) VALUES (1,2)` | `SELECT a,b FROM t WHERE 1=0` | 表名+列名 |
| `UPDATE t SET a=1 WHERE b=2` | `SELECT a,b FROM t WHERE 1=0` | 表名+列名 |
| `DELETE FROM t WHERE id='x'` | `SELECT id FROM t WHERE 1=0` | 表名+列名 |

### QuerySchema 验证（旗舰版特有）

> 旗舰版常使用 `IBillQueryRepository.queryBySchema()` + `QuerySchema` 代替裸 SQL，验证时需先翻译为 SQL。

**翻译步骤**：

1. **提取 entity URI**：从代码中提取 `queryBySchema("bd.supplier.SupplierVO", schema)` 的 URI
2. **查知识库获取元数据**：`D:\yon-bip-obsidian\yon-bip-obsidian\wiki\entities\` 下找对应 `.md`
   - 获取 `schema`（数据库名，如 `fiearap`）
   - 获取 `物理表`（表名，如 `arap_ledger_h`）
   - 获取字段的 `fieldName`（数据库列名）
   - 如果知识库没有对应实体，**不验证**（提醒用户补充元数据到知识库）
3. **翻译为 SQL**（按实际 database 类型适配方言）：
   ```java
   // 原始代码
   QuerySchema.create()
       .addSelect("id", "code", "name")
       .appendQueryCondition(QueryCondition.name("code").eq("001"))
   
   // 翻译为查询语句（用于验证列名+表名）
   // → SELECT id, code, name FROM fiearap.arap_ledger_h WHERE 1=0
   ```
   - `addSelect("id","code","name")` → 直接展开为 SELECT 列
   - `QueryCondition` 只引用第一个条件中的列（避免复杂条件出错），其余条件忽略
   - 始终加 `WHERE 1=0` 或 `LIMIT 1` 防止查询大量数据
   - schema 前缀根据知识库中的 domain/schema 字段确定
4. **自动执行**：
   ```bash
   python C:\Users\99558\.claude\skills\db-query\scripts\db_query.py \
     -p "泸州老窖" -e test --yes -s "SELECT id,code,name FROM fiearap.arap_ledger_h WHERE 1=0"
   ```

**翻译示例对照表**：

| QuerySchema 特性 | 翻译为 SQL | 说明 |
|-----------------|-----------|------|
| `.addSelect("id","code","name")` | `SELECT id,code,name` | 直接展开 |
| `QueryCondition.name("code").eq("001")` | 忽略值，只提取列名 | 不关注查询结果，只验证列名存在 |
| `.appendQueryCondition(a,b)` | 取第一个条件中的列 | 简化，避免复杂条件 |
| `.page(0, 10)` | 忽略 | 不翻译 |

| 数据库类型 | LIMIT 写法 |
|-----------|-----------|
| Oracle / 达梦 | `WHERE ROWNUM <= 1` |
| MySQL / PostgreSQL | `LIMIT 1` |

**特殊处理**：
- 如果 `queryBySchema` 中指定了子表（如 `st.purinrecord.PurInRecords`），需要同时验证主子表的列
- 包含 `QueryCondition.name("ytenant")` 的可以省略（平台公共字段）
- 知识库缺少对应实体 URI 时，**跳过验证并提醒用户**，不要猜测表名直接执行
